# Jetonomy 1.5 — Finish the search adapter (evidence-based)

**Status:** PLANNING — supersedes v1.5-search-adapters.md (the prior version was full of invented classes and speculative architecture; this one is grounded in file:line citations).
**Target release:** 1.5.0
**Created:** 2026-04-30
**Scope:** Free plugin only. No Pro extensions in this plan. ES / Algolia adapters are deferred until the free contract is real and exercised.

### Interface contract

```php
// includes/adapters/interface-search-adapter.php:11
interface Search_Adapter {
    public function is_active(): bool;
    public function index( string $object_type, int $object_id, array $data ): void;
    public function search( string $query, string $type, ?int $space_id, int $limit, int $offset ): array;
    public function delete( string $object_type, int $object_id ): void;
}
```

### Default adapter

`Fulltext_Search` implements all 4 contract methods plus a public static helper `build_boolean_query()` (line 130). `index()` and `delete()` are intentional no-ops (line 22–24, 47–49 — comments explain that MySQL FULLTEXT is "automatic"). `search()` dispatches by `$type` to private `search_posts` / `search_replies` / `search_spaces` methods.

### Registry semantics

`Adapter_Registry::get_search()` (line 67) iterates registered adapters and returns the **first** one whose `is_active()` returns true. There is no `set_active()` method. There is no priority. `Fulltext_Search::is_active()` always returns `true` (line 17–19). If two adapters ever register and both report active, the iteration order silently decides.

## What's broken (3 distinct bugs, evidenced in the codebase)

### Bug 1 — Abilities `execute_search` calls methods that don't exist on the interface

```php
// includes/class-abilities.php:1600
$adapter = Adapters\Adapter_Registry::get_search();
$results['posts']  = $adapter->search_posts( $query, $limit );    // line 1609
$results['spaces'] = $adapter->search_spaces( $query, $limit );   // line 1612
```

`Search_Adapter` defines no `search_posts` or `search_spaces` methods. `Fulltext_Search` has them but they are `private` (line 51, 101).

This isn't speculation — `phpstan-baseline.neon` explicitly suppresses both errors:

```yaml
- message: "#^Call to an undefined method Jetonomy\\\\Adapters\\\\Search_Adapter\\:\\:search_posts\\(\\)\\.$#"
  count: 1
  path: includes/class-abilities.php

- message: "#^Call to an undefined method Jetonomy\\\\Adapters\\\\Search_Adapter\\:\\:search_spaces\\(\\)\\.$#"
  count: 1
  path: includes/class-abilities.php
```

The ability is registered at `class-abilities.php:1093` as `jetonomy/search`, exposed by the WP Abilities API, and `execute_callback` is `execute_search` (line 1141). **A REST call against this ability fatals at runtime.** Nobody has reported it because nobody calls it — the WP Abilities surface isn't widely consumed yet — but it is broken.

### Bug 2 — Search_Controller bypasses the adapter entirely

`Search_Controller::search()` (line 79–159) calls **private** methods `$this->search_posts()` (line 97), `$this->search_spaces()` (line 98), `$this->search_tags()` (line 99), `$this->search_replies()` (line 135). It builds its own SQL directly (line 191 has the `MATCH AGAINST` clause). It never calls `Adapter_Registry::get_search()`.

This means: if a customer registers an alternative adapter (the entire point of the registry pattern), the front-end search bar on `/community/search/` will silently keep using FULLTEXT regardless. The adapter pattern is a lie at the controller layer.

### Bug 3 — Interface can't express what the controller already does

The controller's `search_posts()` (line 174) takes:

```
$q, $space_id, $date_from, $date_to, $author_id, $tag_slug, $sort
```

Plus it does **viewer-aware private-post visibility filtering** (line 195–204) using `Permission_Engine::is_space_privileged( $viewer_id, $space_id )`.

The interface signature is:

```
$query, $type, ?$space_id, $limit, $offset
```

If we route the controller through the adapter today, we'd lose: tag filtering, date filtering, author filtering, sort options, viewer-aware visibility. The adapter contract is **too narrow** to be a drop-in for the controller. Until this is widened, the bypass at Bug 2 is technically forced — there's no way to route through and keep features.

## What's NOT broken (correcting my prior plan)

- **There is no missing indexing pipeline.** `index()` and `delete()` are documented no-ops for FULLTEXT and the comments explicitly say "this method exists for external adapters" (line 23). FULLTEXT operates on live tables and needs no separate index. Lifecycle indexing is something a future ES/Algolia adapter would need; the absence is not a bug for the current default.
- **There is no missing `Document_Serializer`.** Nothing today produces or consumes documents.
- **There is no missing CLI.** `wp jetonomy search reindex` would only matter for an external-index adapter.
- **There is no missing admin UI.** The single registered adapter is FULLTEXT; there's nothing to pick.

The previous plan invented all of these. They're not "what we have not finished" — they're future work that only matters if Pro extensions land.

## Plan — finish what's broken, in this order

### Step 1 — Fix the Abilities fatal (Bug 1)

Make `execute_search` use only methods that exist on the interface.

```php
// includes/class-abilities.php:1608–1614 (the broken block today)
if ( in_array( $filter, [ 'all', 'posts' ], true ) ) {
    $results['posts'] = $adapter->search( $query, 'post', null, $limit, 0 );
}
if ( in_array( $filter, [ 'all', 'spaces' ], true ) ) {
    $results['spaces'] = $adapter->search( $query, 'space', null, $limit, 0 );
}
```

Then **delete the two phpstan-baseline.neon entries** for the now-undefined-method errors. Baseline shrinks.

LOE: ~30 minutes including manual test against the WP Abilities REST endpoint.

**Acceptance:** `vendor/bin/phpstan analyse` clean without those two suppressions; `wp jetonomy ability invoke jetonomy/search --input='{"query":"test"}'` returns a non-error result.

### Step 2 — Decide the interface widening question

Two options, pick one before writing code:

**Option A (minimal):** Keep the interface narrow (4 methods). Search_Controller continues to do its own SQL and the adapter is only used by Abilities. Document this in `interface-search-adapter.php` so future readers know the controller intentionally doesn't route through the registry. Pro ES/Algolia adapters would only serve the Abilities surface, not the front-end search bar.

**Option B (widen to controller-shape):** Add filter parameters to `search()` (or add a new method like `query()`) so the controller can route through. Update `Fulltext_Search::search_posts()` to accept all the filters the controller already builds inline. Eliminate the duplicated SQL between the two classes.

**Recommendation:** B. The whole point of the registry is that any adapter can serve all consumers. A is a permanent admission that the abstraction failed. B fixes the architecture and unblocks Pro extensions to actually replace the front-end search experience.

If B: the new `search()` signature stays back-compat by making new args nullable with defaults. Existing callers (Abilities, soon Search_Controller) pass them. External adapters that don't support a filter just ignore it.

LOE for B: ~1 day. Files: `interface-search-adapter.php`, `class-fulltext-search.php`, `class-search-controller.php`, `class-abilities.php`, plus tests.

**Acceptance:** `Search_Controller::search()` calls `Adapter_Registry::get_search()->search( ..., $filters )` for the post / reply / space / tag flows. The duplicated `MATCH AGAINST` SQL between controller and adapter no longer exists. Search behaves identically (browser-verified vs current main).

### Step 3 — Resolve the COUNT-query pagination TODO

```php
// includes/api/class-search-controller.php:151
// TODO: implement proper search pagination with COUNT query for accurate total.
```

This is also in `class-abilities.php:1265, 1324` (same TODO twice).

For Step 3 to make sense, Step 2 has to land first (the SQL builder consolidates). Then add a `count()` companion to whatever search method we standardise on, OR have `search()` return a tuple `{hits, total}`.

LOE: ~half day after Step 2.

**Acceptance:** The `meta.total` field on `GET /jetonomy/v1/search` reports the actual total result count, not `count($items)`. Pagination links resolve correctly when the result set exceeds 20.

### Step 4 — Adapter selection mechanism (only if Step 2 is Option B)

If Step 2 widens the interface so multiple adapters could meaningfully co-exist, the registry needs a way to pick which adapter is active. Currently `get_search()` returns the first `is_active()` adapter — brittle.

Add a setting `jetonomy_search_adapter_id` (default `'fulltext'`). `get_search()` becomes:

```php
public static function get_search(): ?Search_Adapter {
    $id = get_option( 'jetonomy_search_adapter_id', 'fulltext' );
    if ( isset( self::$search[ $id ] ) && self::$search[ $id ]->is_active() ) {
        return self::$search[ $id ];
    }
    // Fallback: first active.
    foreach ( self::$search as $adapter ) {
        if ( $adapter->is_active() ) {
            return $adapter;
        }
    }
    return null;
}
```

Plus a setter: `Adapter_Registry::set_active_search( string $id )` updates the option.

LOE: ~2 hours. **Skip if Step 2 ends as Option A.**

**Acceptance:** `get_option( 'jetonomy_search_adapter_id' )` controls which adapter handles searches; default is `'fulltext'`; if a future adapter registers it can be selected without touching code.

Pro ES/Algolia adapters become a clean follow-on because they have a real contract to implement.

## What this plan deliberately does NOT include

- Pro Elasticsearch adapter — out of scope until contract is real.
- Pro Algolia adapter — same.
- Indexing pipeline (`Search_Indexer` etc) — only matters for external-index backends; not needed for FULLTEXT.
- Admin UI for picking adapter — only matters when there's more than one.
- CLI bulk reindex — same.
- Typeahead UI — separate feature, separate plan.
- Multi-language stemming, faceting, highlights, suggestions — all features of specific external backends; they belong in those adapters' implementations, not the contract.

If a Pro ES adapter ever needs facets/highlights, that's the moment to widen the interface again — not now, when there's no consumer for those features.
