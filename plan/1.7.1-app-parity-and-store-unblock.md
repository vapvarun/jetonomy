# 1.7.1 — App Parity + App Store Unblock

**Goal:** get the iOS app approved AND bring it fully current with plugin 1.7.0.
Both, in one release. A parity-incomplete app is itself a rejection risk (Apple 1.2
asks for a recording proving the safety mechanisms work; 4.2 catches dead surfaces).

**Repos (3):**
- `jetonomy` (free) — branch `1.7.1-dev`
- `jetonomy-pro` — branch `1.7.1-dev`
- `jetonomy-app` — branch `main` (Expo / React Native / expo-router)

**Rejection of record:** Submission `f9b26f1f`, 2026-07-08, v1.0 (5), reviewed on
iPad Air 11-inch (M3). Two guidelines: **5.1.1(ii)** purpose strings, **1.2** UGC.

---

## Why the app was rejected

| # | Guideline | Apple's ask | State today |
|---|---|---|---|
| 1 | 5.1.1(ii) | Photo-library purpose string must say *how* data is used, with an example | `app.json` has **no** `NSPhotoLibraryUsageDescription`; `expo-image-picker` is in `package.json` but **not** in the `plugins` array → iOS ships Expo's generic default string |
| 2 | 1.2 | EULA before register/login, zero-tolerance terms | **Missing entirely** — no EULA anywhere in the app |
| 3 | 1.2 | Mechanism to flag objectionable content | **EXISTS** — `FlagButton.tsx` on posts + replies → `POST /flags` → moderation queue |
| 4 | 1.2 | Mechanism to block abusive users; blocked content leaves the feed **instantly**; developer notified | **MISSING — and the plugin cannot do it.** Only DM-conversation block exists (`POST /conversations/{id}/block`). No user-level block anywhere in Jetonomy, web or mobile. |

The 2026-07-03 submission pack (`~/Desktop/jetonomy-app-submission/app-privacy-and-age-rating.txt`)
asserted "Block/mute or remove members" existed. It did not. Apple checked.

## Parity gap (app was built against plugin 1.6.0)

| Feature | Plugin | App |
|---|---|---|
| Anonymous posting (1.7.0, Pro) | `is_anonymous` on post + reply create; per-space `allow_anonymous`; server-side gate; `POST /anonymous/reveal` | **Zero code.** grep `anonym` → no matches |
| File attachments (1.7.0, Pro) | `jetonomy-pro/v1/attachments` (POST/DELETE/download/batch); `attachments[]` injected into post + reply payloads | **None consumed.** Only inline composer image + avatar, both via `POST /media`. No document picker. |
| Search pagination | server returns one ranked page | no load-more (`api/search.ts:4`) — violates big-site checklist |
| Digest prefs | `/users/me/digest-preferences` | row gated on the **wrong** flag: `features.custom_fields` (`app/settings.tsx:139`) |
| AI suggest-reply | `POST /ai/suggest-reply` | never called |

## Plugin-side blockers (the app cannot gate cleanly without these)

- `/app/config` `features{}` omits `anonymous` and `attachments` (`class-app-config-controller.php:147`)
- Attachment limits (`max_files`, `max_size_bytes`, `allowed_types`) are **not on REST** — web gets them via an inline `window.jetonomyAttachments` blob
- Reply + attachments has no single-request path (free doesn't pass `$request` to `jetonomy_after_create_reply`)

## Waves

### Wave 1 — Plugin contract (blocking; app depends on all of it)

**1A · User blocking — FREE** (new feature; safety primitive, must not be paywalled)
- Migration: `jt_blocked_users` (`blocker_id`, `blocked_id`, `created_at`; UNIQUE(blocker,blocked); KEY(blocker))
- Model + REST: `GET/POST /users/me/blocks`, `DELETE /users/me/blocks/{user_id}`
- **Viewer-scoped filtering** — blocked authors' content disappears for the blocker in:
  feed, space post lists, post detail, replies, search, notifications, mentions,
  DM recipient suggestions, and new-DM creation
- Big-site: load the blocker's block-list **once per request** (no N+1); index-backed
  `WHERE author_id NOT IN (...)`; degrade to no-op when the list is empty
- Blocking is one-way and silent (blocked user is not told), per platform norm
- Moderator notification: reuse the existing flag → moderation-queue path

**1B · Account deletion — FREE** (Guideline 5.1.1(v); pre-empt the next rejection)
- `DELETE /users/me` — requires re-auth/confirmation; decide content policy
  (recommend: anonymize authored content, hard-delete profile + PII + tokens)
- Must clean `jt_*` rows (blocks, votes, bookmarks, subs, push devices, conversations)

**1C · `/app/config` additions — FREE + PRO**
- Free: `features.blocking`, `terms_url`, `privacy_url`
- Pro (via `jetonomy_app_config` filter): `features.anonymous`, `features.attachments`
- Attachment limits block: `max_files`, `max_size_bytes`, `allowed_types`

**1D · Reply attachments single-request path — FREE + PRO**
- Free: pass `$request` to `jetonomy_after_create_reply`
- Pro: read `attachment_ids` off the reply create request (mirrors the post path)

### Wave 2 — App: store blockers

**2A** `app.json` — declare `expo-image-picker` in `plugins` with a specific
`photosPermission` (and camera string if used). Must state *what* the photo is for
and give an example. Also audit every other purpose string.

**2B** EULA gate — presented **before** register/login; zero-tolerance language for
objectionable content and abusive users; acceptance persisted; links to terms + privacy.

**2C** User blocking UI — Block on user profile + post/reply overflow menu; Blocked
Users list in Settings (unblock); **optimistic cache invalidation** so content vanishes
the instant Block is tapped.

**2D** Delete Account screen in Settings — confirmation + consequences copy.

### Wave 3 — App: parity

**3A** File attachments — `expo-document-picker`; upload; attachment list on posts +
replies; image thumbs; PDF view; force-download for other types; delete; respect the
limits from `/app/config`; both post AND reply paths.

**3B** Anonymous posting — compose toggle gated on the space's `allow_anonymous`
(read from `GET /spaces/{id}` → `settings`) and `features.anonymous`; render anonymous
authors (id 0 / "Anonymous" / no avatar / no profile link).

**3C** Search pagination — infinite scroll (needs a server-side check: does `/search`
honour limit/offset? If not, that's a Wave 1 addition).

**3D** Digest flag fix — gate on the correct feature flag, not `features.custom_fields`.

**3E** AI suggest-reply surface in the reply composer (Pro, feature-gated).

### Wave 4 — Verify + resubmit

- Per-item device verification (verify-per-item rule; no batching)
- Big-site checklist on every list touched (feed, search, blocked-users, attachments)
- **Screen recording on a physical device** showing: EULA at login → flagging content →
  blocking a user (content disappearing). Apple explicitly requires this in the Notes field.
- Correct `~/Desktop/jetonomy-app-submission/` pack: the block claim, plugin zips (1.6.0 → 1.7.1)
- Bump app to 1.0.1 (build 6); resubmit

---

## Non-negotiables

- Blocking filters **every** read surface, not just the feed. A blocked user's reply
  showing up in a thread = Apple 1.2 failure on the recording.
- No raw `$wpdb` outside models.
- Every list touched runs the big-site checklist (pagination, indexes, N+1, COUNT(*)).
- App: every UI change verified at 390px AND on iPad (Apple reviewed on an iPad Air 11").
- Attachments/anonymous must be **feature-gated** off `/app/config` — a free-tier or
  extension-disabled site must not show dead UI.
