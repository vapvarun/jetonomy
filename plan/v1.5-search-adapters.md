# Jetonomy 1.5 — Pluggable search adapters (Elasticsearch + Algolia)

**Status:** PLANNING — design complete, no code changes until this plan is signed off.
**Target release:** 1.5.0
**Created:** 2026-04-30

## Why this matters

Jetonomy's current search is FULLTEXT-only against MySQL (`MATCH ... AGAINST`). It works fine up to ~5–10k posts. Past that:

- No typo tolerance ("loginn" returns nothing).
- No synonyms or stemming ("login" ≠ "logging in" ≠ "log in").
- No multi-language stemming (CJK, Arabic, Russian degrade to literal substring search).
- No "search-as-you-type" with ranked suggestions.
- Single-table FULLTEXT scans rebuild on every write (slow at scale).

This is the single most-cited "Discourse is better" complaint we'd hit when pushing past medium-sized communities. Closing it doesn't require us to ship our own search engine — it requires us to make the existing search backend pluggable so customers running large communities can drop in Elasticsearch or Algolia (or Meilisearch later) without forking the plugin.

The good news: most of the scaffolding is already in place from earlier work. This plan finishes what's started.

Bootstrap:


Existing `Search_Adapter` interface:
```php
interface Search_Adapter {
    public function is_active(): bool;
    public function index( string $object_type, int $object_id, array $data ): void;
    public function search( string $query, string $type, ?int $space_id, int $limit, int $offset ): array;
    public function delete( string $object_type, int $object_id ): void;
}
```

This is good — the contract is right and the default backend works.

### What's missing

1. **`Search_Controller` doesn't use the adapter.** `includes/api/class-search-controller.php` runs raw `MATCH ... AGAINST` SQL directly (lines 191, 308). The default adapter exists but the controller bypasses it.
2. **No indexing pipeline.** `index()` and `delete()` aren't hooked to post/reply lifecycle. Pure FULLTEXT survives this because it queries the live tables; ES/Algolia would need a separate index, populated on every write.
3. **No Elasticsearch adapter.**
4. **No Algolia adapter.**
5. **No admin UI to pick + configure backend.**
6. **No bulk reindex CLI.**
7. **Interface is too narrow for ES/Algolia.** Missing facet/filter support, missing query-modifiers (boost, fuzziness), missing pagination cursor for large result sets.

## Design

### Phase 1 — Tighten the interface (additive; backwards-compatible)

The existing `Search_Adapter` interface stays valid as a baseline. Add a richer companion contract that ES/Algolia adapters declare so the controller can take advantage of their capabilities without requiring them from every backend.

```php
// includes/adapters/interface-search-adapter.php — UNCHANGED, kept for back-compat
interface Search_Adapter {
    public function is_active(): bool;
    public function index( string $object_type, int $object_id, array $data ): void;
    public function search( string $query, string $type, ?int $space_id, int $limit, int $offset ): array;
    public function delete( string $object_type, int $object_id ): void;
}

// includes/adapters/interface-advanced-search-adapter.php — NEW
interface Advanced_Search_Adapter extends Search_Adapter {
    /**
     * Capabilities the adapter supports. Used by the controller to decide
     * which features to expose. Default adapter declares minimal caps.
     *
     * @return array{
     *   typo_tolerance: bool,
     *   synonyms: bool,
     *   stemming_languages: string[],   // e.g. ['en', 'de', 'ja']
     *   facets: bool,                    // counts per space/tag/author
     *   highlight: bool,                 // <mark> snippets
     *   suggest: bool,                   // typeahead
     *   filters: string[],               // ['space_id', 'author_id', 'tag_slug', 'date_from', 'date_to']
     *   sort: string[],                  // ['relevance', 'newest', 'oldest', 'reply_count']
     *   max_query_length: int,
     *   rate_limit_per_minute: ?int,
     *   provider: string,                // 'fulltext' | 'elasticsearch' | 'algolia' | 'meilisearch'
     *   region: ?string,                 // 'us-east-1', 'eu-west-1', etc — null for self-hosted
     * }
     */
    public function capabilities(): array;

    /**
     * Rich search with filters, facets, highlights, suggestions.
     *
     * @param string $query
     * @param array  $args { type, space_id, filters, sort, page, per_page, highlight, suggest, facets }
     * @return array {
     *   hits: array<int, array{id, type, score, highlights, snippet, ...}>,
     *   total: int,
     *   facets: array<string, array<string, int>>,    // facet_name => { value => count }
     *   suggestions: string[],                          // typeahead suggestions
     *   query_time_ms: int,
     *   adapter_id: string,
     * }
     */
    public function query( string $query, array $args ): array;

    /**
     * Bulk index — for backfill / reindex CLI. Accepts an iterable so adapters
     * can stream and the caller doesn't need to load 100k posts into memory.
     */
    public function bulk_index( iterable $documents ): array;

    /**
     * Reindex from scratch — drops the index and rebuilds. Used by
     * `wp jetonomy search reindex --confirm`.
     */
    public function reindex_all(): array;

    /**
     * Health check — returns 'ok' / 'degraded' / 'down' with details.
     * Used by the admin Health page.
     *
     * @return array{ status: string, latency_ms: ?int, last_indexed_at: ?string, errors: string[] }
     */
    public function health(): array;
}
```

`Fulltext_Search` declares minimal capabilities and implements the basic shape. `Elasticsearch_Search` and `Algolia_Search` declare full capabilities.

### Phase 2 — Refactor `Search_Controller` to consume the adapter

Single point of change: `includes/api/class-search-controller.php`.

```php
public function search( WP_REST_Request $request ): WP_REST_Response|WP_Error {
    $adapter = Adapters\Adapter_Registry::get_search();
    if ( ! $adapter || ! $adapter->is_active() ) {
        return new \WP_Error( 'jetonomy_search_unavailable', __( 'Search is currently unavailable.', 'jetonomy' ), [ 'status' => 503 ] );
    }

    $args = $this->normalize_args( $request );  // type, space_id, filters, sort, page, per_page

    if ( $adapter instanceof Adapters\Advanced_Search_Adapter ) {
        $result = $adapter->query( $args['q'], $args );
    } else {
        // Back-compat path for adapters implementing only the basic interface.
        $hits = $adapter->search( $args['q'], $args['type'], $args['space_id'], $args['per_page'], ($args['page'] - 1) * $args['per_page'] );
        $result = [ 'hits' => $hits, 'total' => count( $hits ), 'adapter_id' => 'basic' ];
    }

    return new WP_REST_Response( $result, 200 );
}
```

The current 396-line controller shrinks to ~150 lines. The MATCH AGAINST SQL moves entirely into `Fulltext_Search::query()`.

**Migration safety:** the response shape stays the same as today's flat-list-of-results so existing JS clients don't break. New facets / suggestions / highlights are additive fields the JS can ignore until the typeahead UI lands.

### Phase 3 — Indexing pipeline (lifecycle hooks)

When a post or reply is created, updated, or deleted, push the change to the active search adapter. This is a no-op for FULLTEXT (it queries live tables) but mandatory for ES / Algolia.

Wire to the hooks free already fires (post-1.4.1):

```php
// includes/search/class-search-indexer.php — NEW
class Search_Indexer {
    public function boot(): void {
        add_action( 'jetonomy_after_create_post', [ $this, 'on_post_save' ], 20, 2 );
        add_action( 'jetonomy_after_update_post', [ $this, 'on_post_save' ], 20, 2 );
        add_action( 'jetonomy_after_delete_post', [ $this, 'on_post_delete' ], 20, 1 );
        add_action( 'jetonomy_after_create_reply', [ $this, 'on_reply_save' ], 20, 2 );
        add_action( 'jetonomy_after_update_reply', [ $this, 'on_reply_save' ], 20, 2 );
        add_action( 'jetonomy_after_delete_reply', [ $this, 'on_reply_delete' ], 20, 1 );
    }

    public function on_post_save( $post, array $context = [] ): void {
        $adapter = Adapter_Registry::get_search();
        if ( ! $adapter ) return;
        $adapter->index( 'post', (int) $post->id, $this->serialize_post( $post ) );
    }
    // ...
}
```

`serialize_post()` produces the document shape every adapter ingests:

```json
{
  "id": 1234,
  "title": "...",
  "content_plain": "...",
  "content_html": "...",
  "space_id": 7,
  "space_slug": "support",
  "space_title": "Support",
  "author_id": 42,
  "author_login": "alice",
  "author_display_name": "Alice",
  "tags": ["billing", "auth"],
  "created_at": "2026-04-30T10:15:00Z",
  "updated_at": "2026-04-30T11:45:00Z",
  "vote_score": 5,
  "reply_count": 3,
  "is_resolved": true,
  "url": "https://example.com/community/s/support/t/why-..."
}
```

**Async indexing (Pro extensions only):** for large communities, the indexer dispatches via `Queue::async` (Action Scheduler) so the lifecycle hook returns instantly. Default FULLTEXT skips the queue (no-op anyway).

### Phase 4 — Elasticsearch adapter

Live in **Pro** as `jetonomy-pro/includes/extensions/elasticsearch-search/`. Lives in Pro because:
- Customer base is small (only large communities need it).
- Configuration is non-trivial (cluster URL, index prefix, auth).
- Pro has the AI extension pattern + license gating already.

```
jetonomy-pro/includes/extensions/elasticsearch-search/
├── class-extension.php                 ← boot(), settings, admin UI
├── class-elasticsearch-search.php      ← implements Advanced_Search_Adapter
├── class-elasticsearch-client.php      ← thin HTTP wrapper around ES REST API
├── class-index-mapping.php             ← analyzer + field mapping per language
└── views/
    └── settings.php                    ← cluster URL, index prefix, auth, language
```

Key implementation notes:

- **Client:** use `wp_remote_post()` with `Authorization: Basic <user:pw>` or `ApiKey <key>` header. Don't pull in elasticsearch-php (Composer dep adds 5MB; plain HTTP is enough for our query shape).
- **Index naming:** `{prefix}_{site_id}_posts` and `{prefix}_{site_id}_replies` — `prefix` defaults to `jetonomy`, `site_id` is `get_current_blog_id()` for multisite isolation.
- **Mapping:** declared in `class-index-mapping.php`. `title` and `content_plain` use the language-specific analyzer (`standard`, `english`, `japanese`, `arabic`, etc.) selected from settings. `tags`, `space_id`, `author_id` indexed as keyword for exact-match facets.
- **Bulk indexing:** batches of 500 documents via the `_bulk` endpoint. `bulk_index()` returns `[ 'indexed' => int, 'failed' => int, 'errors' => array ]`.
- **Query DSL:** multi-match across title (boost x3) + content_plain (boost x1) with `fuzziness: AUTO`. Filter clause uses `term`/`terms` for space_id, author_id, tags, range for dates.
- **Highlights:** `<mark>` tags via the highlight clause; render in JS without escaping the marks.
- **Suggestions:** `completion` suggester on `title.suggest` field. Typeahead UI lives in 1.5.1.
- **Health:** GET `/_cluster/health` mapped to `green` → `ok`, `yellow` → `degraded`, `red` → `down`.

**Settings UI:**
```
Search Backend (admin → Jetonomy → Settings → Search)
  ○ MySQL FULLTEXT (default, recommended for <10k posts)
  ● Elasticsearch
    URL:                    [https://es.example.com:9200    ]
    Index prefix:           [jetonomy                         ]
    Auth method:            [Basic ▼]
    Username:               [...                              ]
    Password:               [...] (encrypted at rest via wp_salt)
    Language analyzer:      [English ▼]
    [Test connection] [Reindex all]
  ○ Algolia (Pro)
```

**Migration:** when an admin switches from FULLTEXT to ES, the UI runs `reindex_all()` synchronously with progress reporting. CLI fallback: `wp jetonomy search reindex --provider=elasticsearch`.

### Phase 5 — Algolia adapter

Live in **Pro** as `jetonomy-pro/includes/extensions/algolia-search/`. Same architecture as ES.

```
jetonomy-pro/includes/extensions/algolia-search/
├── class-extension.php
├── class-algolia-search.php            ← implements Advanced_Search_Adapter
├── class-algolia-client.php
├── class-index-settings.php            ← Algolia-specific tunables
└── views/
    └── settings.php
```

Algolia-specific differences from ES:

- **Client:** `https://{appId}-dsn.algolia.net/1/indexes/{indexName}` for read, `https://{appId}.algolia.net/1/indexes/{indexName}` for writes. Both via `wp_remote_request()`.
- **No mapping step** — Algolia auto-detects field types. Only `searchableAttributes`, `customRanking`, `attributesForFaceting` are declared at index-create time.
- **Settings** push as one PUT to `/1/indexes/{name}/settings`. Stored in code so on-version-bump we can re-push.
- **Bulk indexing:** `/1/indexes/{name}/batch` accepts up to 1000 ops per request.
- **Query:** single `query` parameter + `filters` with Algolia DSL syntax (`space_id:7 AND tags:billing`).
- **Highlights:** Algolia returns `_highlightResult` per field; render directly.
- **Suggestions:** Algolia `query suggestions` is a separate index — out of 1.5 scope, ship in 1.6.
- **Pricing awareness:** show estimated record count on settings page next to "Reindex" button so admins don't accidentally exceed their plan tier.

**Why ship both ES and Algolia:**
- ES: customers who self-host (data residency, EU privacy) or already have an ES cluster.
- Algolia: customers who want managed + best-in-class typo tolerance, willing to pay per-record.

## Migration story for existing customers

**On upgrade to 1.5.0:** nothing changes. FULLTEXT remains the default. Search keeps working identically. Lifecycle indexing hooks fire but call into a no-op default adapter `index()`.

**Customer wants to switch to ES (or Algolia):**
1. Activate Pro `elasticsearch-search` (or `algolia-search`) extension.
2. Configure cluster URL / API key.
3. "Test connection" must pass.
4. Click "Reindex all" — synchronous progress bar, ~5 min for 50k posts on a typical ES cluster.
5. Default adapter swaps via `Adapter_Registry::set_active( 'elasticsearch' )`.
6. Search queries route to ES from the next request.

**Rollback:** un-tick the Pro extension. Adapter falls back to FULLTEXT. ES/Algolia indexes remain (cleanup is opt-in via `wp jetonomy search reset --provider=elasticsearch`).

## What this plan does NOT do

- **Does not implement Meilisearch.** The shape is identical to Algolia's; can be added in 1.6 if a customer asks.
- **Does not implement OpenSearch.** ES adapter works against OpenSearch with the same client (API-compatible). Documented, not separately implemented.
- **Does not federate search across sites.** Multisite-network-wide search is a separate feature.
- **Does not add a typeahead UI.** Only the data path. Typeahead lands in 1.6.
- **Does not implement query analytics.** "Most searched terms" dashboard is 1.7+ work.

## Open design questions (decide before implementation starts)

1. **Encryption at rest for ES password / Algolia API key.** WordPress has no first-class secret storage. Options: (a) `wp_salt()` AES-encrypt and store in `jetonomy_settings`; (b) require `wp-config.php` constants (`JT_ES_PASSWORD`, `JT_ALGOLIA_KEY`) — never write to DB; (c) lean on a secrets-manager plugin if installed. **Recommendation:** option (b) — config-file constants are the standard WordPress pattern for sensitive credentials and avoid DB-snapshot leaks.

2. **Indexing throttle for high-write communities.** A community ingesting 100 posts/sec would saturate the Action Scheduler queue. **Recommendation:** rate-limit at the indexer (max 50 ops/sec batched into _bulk requests every 2s). Configurable.

3. **What happens when ES is down mid-write?** Post still saves to MySQL; indexer call fails. Options: (a) fail-soft and log; (b) retry queue with exponential backoff; (c) fall back to FULLTEXT for searches until ES recovers. **Recommendation:** (a) + (b) — fail-soft so user-facing writes succeed, queue retries up to 3 times, surfaces in admin Health panel after that.

4. **Should the default `Fulltext_Search` adapter implement `Advanced_Search_Adapter` too** (with limited capabilities) so the controller has a single code path? Or keep two paths? **Recommendation:** yes, refactor `Fulltext_Search` to implement both — controller stays simple, capabilities reported truthfully (no typo tolerance, but does support facets via separate count queries).

5. **Per-space access checks at query time.** ES doesn't know about Jetonomy's permission engine. Options: (a) post-filter results in PHP (simple, slow at scale); (b) push `allowed_space_ids` into ES query as a filter (fast, requires recomputing on space-membership changes). **Recommendation:** (b) for ES, with a `jetonomy_search_visible_space_ids` filter so Pro extensions like advanced moderation can override.
