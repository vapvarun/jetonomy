# 1.4.1 Master Plan

Single source of truth for the 1.4.1 release. Every issue surfaced by the manifest audit (`audit/manifest.json` + `audit/REST_AUDIT.md` once Phase A1 lands) is addressed here. Each work package has scoped safety checks in [`1.4.1-safety-checks.md`](1.4.1-safety-checks.md).

**Branch:** `1.4.1` (already pushed on both `jetonomy` and `jetonomy-pro`)
**Version target:** 1.4.1 (Free + Pro lockstep — `bin/build-release.sh` step 3 enforces)
**Calendar:** ~3 weeks if Track A and Track B run in parallel
**Release gate:** every package's safety check signed off in `plan/1.4.1-baselines/CHECKLIST.md` + green `/jetonomy-smoke` in both FREE and FREE+PRO modes

## Critical-path & sequencing

```
A1 → A2 → A3 ── (security track, gated)
              
A4, A5 ─── REST additions (parallel, independent)
A6, A7 ─── admin pages (parallel, independent)
A8       ─── email templates (parallel, independent)
A9, A10  ─── frontend + cron (parallel, independent)

B1 ─── White Label (parallel with everything)
B2 ─── Analytics dual-path (parallel, but needs full cycle for cutover decision)
B3 ─── Email Digest (parallel)
```

Only hard dependency: A1 → A2 → A3. Everything else can start day 1.

### Suggested 3-week schedule

**Week 1**
- Day 1–2: A1 (Explore agent reads 18 callbacks → `audit/REST_AUDIT.md`), A2 (manifest schema bump + refresh)
- Day 3–5: A3 (security fixes, one route per commit, smoke gate after each); B1 (White Label wiring) in parallel

**Week 2**
- A4 + A5 (1.5 days total, parallel)
- A6 + A7 (2 days each, parallel)
- B2 starts: dual-path Analytics; needs 7 days of data parity, so kick off no later than start of week 2

**Week 3**
- A8 (3 days) + A9 (2 days) + A10 (1 day) — sequential or parallel
- B3 (1 day)
- Final 2 days: full smoke + manual QA + release-build dry-run + manifest refresh
- Final commit: `docs: 1.4.1 audit refresh` regenerates `audit/manifest.json` to reflect everything that shipped

### B2 cutover rule

B2's success criterion: `|from_query − from_events| / from_query < 1%` for 7 consecutive days. If that fails by week 3:
- **Ship 1.4.1 with old path still active** — keep dual-path code, mark it experimental
- **Defer cutover to 1.5.0**
- Document the discrepancy in `plan/1.4.1-baselines/B2/discrepancies.log`

This rule means B2 never blocks the release.

## Out-of-band: known issues we are intentionally NOT fixing in 1.4.1

These came up in the audit but the team has deliberately scoped them out (record reasoning so they don't get re-discussed):

| Issue | Reason for deferral |
|---|---|
| 18 capabilities declared but never visibly checked | A2 will reveal whether they're actually checked inside callbacks. Decision (enforce vs remove) deferred until that data is in. |
| 33 orphan action hooks + 35 orphan filter hooks (Free side) | Most are valid extension points awaiting third-party use. Pro consumes 17 — rest are intentional. Filter cleanup is not a 1.4.1 deliverable. |
| 10 of 17 Pro tables have no corresponding REST endpoint | All 10 are internal join/cache/log tables that should not be REST-addressable. False positive from manifest analysis. |
| Setup page detached from main menu (`parent: null`) | Confirmed intentional — wizard is a one-time experience, hides itself. UX-design call needed before changing. |
