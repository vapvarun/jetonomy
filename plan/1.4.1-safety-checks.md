# 1.4.1 Per-Task Safety Checks

Every work package in the 1.4.1 cycle has its own targeted safety check below. The global `/jetonomy-smoke` runbook still runs after each commit; **these checks layer on top** to verify the specific behavior the task touches.

## How to use this file

Before starting a package, the implementing agent (wp-builder, wp-fixer, etc.) reads its section and:

1. **Captures the PRE-state** (snapshot, screenshot, DB row, HTTP response) — committed to `plan/1.4.1-baselines/<package>/` in the **first commit of the package**.
2. **Implements the change** in subsequent commits.
3. **Captures the POST-state** and runs the verification commands listed below.
4. **Diffs PRE vs POST** — divergence outside the documented expected delta = blocked.

If a check fails, **revert that commit** and re-plan. Do not push past a failing scoped check just because the global smoke happens to pass.

**PRE:**

**POST:**
- [ ] `jq . audit/manifest.json > /dev/null` — still valid JSON
- [ ] All counts unchanged (no items lost during reshape):
  ```bash
  jq '.rest.endpoints | length' audit/manifest.json | diff - /tmp/a2.before.rest
  ```

### A3 — REST security fixes (per-route, one commit each)

**Risk:** HIGH. Each fix may break a legitimate user flow.

**Per-route protocol:**

#### Before commit:
```bash
# Capture baseline responses for THIS specific route, e.g. PATCH /posts/{id}
curl -s -o /dev/null -w "owner=%{http_code} " \
  -X PATCH http://forums.local/wp-json/jetonomy/v1/posts/123 \
  -H "X-WP-Nonce: $(wp eval 'echo wp_create_nonce("wp_rest");' --user=author1)" \
  --cookie "$(wp user generate-cookie author1)"
# expect 200

curl -s -o /dev/null -w "non_owner=%{http_code} " \
  -X PATCH http://forums.local/wp-json/jetonomy/v1/posts/123 \
  --cookie "$(wp user generate-cookie author2)"
# expect 403 after fix; before fix may show 200

curl -s -o /dev/null -w "anon=%{http_code} " \
  -X PATCH http://forums.local/wp-json/jetonomy/v1/posts/123
# expect 401 always
```

Pin all three response codes to `plan/1.4.1-baselines/A3/<route>.txt`.

#### After commit:
- [ ] Owner still gets `200` (legitimate flow unbroken)
- [ ] Non-owner gets `403` (the bug is closed)
- [ ] Anonymous gets `401` (no regression on auth)
- [ ] **Browser smoke**: log in as `author1`, edit own post in UI → save succeeds. Log in as `author2`, try to access edit URL of author1's post → blocked.
- [ ] **Existing customer journey**: post composer (new post) still works for editor/admin

#### Bisectability:
One route per commit. Commit message: `fix(rest): tighten <METHOD> /<route> ownership check (closes IDOR)`. If smoke fails, `git revert HEAD` recovers a working state.

### A8 — Email Templates editor

**Risk:** medium — actually writes to `jetonomy_email_templates` option, can corrupt notifications.

**PRE:**
```bash
wp option get jetonomy_email_templates --format=json > /tmp/a8.before.json
```

**POST per change:**
- [ ] Save a template → option key updated:
  ```bash
  wp option get jetonomy_email_templates --format=json > /tmp/a8.after.json
  diff /tmp/a8.before.json /tmp/a8.after.json    # only the edited template differs
  ```
- [ ] **Send test email** for the edited template — content reflects the new copy. Use the existing `jetonomy_test_email` AJAX action.
- [ ] **Reset to default** button restores the documented default value (same as before customization)
- [ ] **Trigger a real notification** (post a reply, expecting "reply_to_post" email) — recipient gets the customized version
- [ ] **No XSS:** put `<script>alert(1)</script>` in template body, save, send → email body has the literal string, not script execution
- [ ] **No regression on un-edited templates** — `jetonomy_email_templates['user_welcome']` unchanged after editing only `reply_to_post`

## Anti-patterns to NOT do

- ❌ Run `/jetonomy-smoke` once at the end of the cycle and call it done. Smoke catches regressions, not all regressions.
- ❌ Skip the PRE baseline for "obvious" changes. Email Templates editor saved against an unwritten baseline = silent corruption.
- ❌ Bundle multiple packages into one commit. The whole point of bisectability is one revert undoes one logical change.
- ❌ Use a real customer's account for the test fixtures. Use `wp user create test-author1 ...` and clean up after.
- ❌ Skip the disable/cleanup step in B1. White Label "enabled but turned off" must produce pixel-identical baseline; failing this hides cleanup bugs.
