## 🚨 Release-blocker discovered during browser verification

**Symptom:** Clicking reply vote buttons (up or down) produces no DOM mutation, no fetch to REST, no console error. Pre-WS3-B code (1.4.2) handled this directly with fetch + DOM manipulation. WS3-B refactored to `*voteReplyUp/Down() { yield jetonomyOptimistic.gen({ apply, fetch, onSuccess, revert }) }`.

**Likely cause:** The WP Interactivity API's `store({ actions: { ... } })` expects each action to be either an `async function` or a generator that yields *promises*. `jetonomyOptimistic.gen()` returns an object that IS an iterable per its WS3-A implementation, but the yield contract with WP IA may not match — IA might be calling `.next()` once and seeing a non-promise value, then bailing without invoking the handler.

**Next steps required:**
1. Inspect actual IA event dispatch flow (read `@wordpress/interactivity` source for how `data-wp-on--click="actions.X"` invokes `actions.X` — does it await? does it iterate generators?)
2. Either (a) restructure `jetonomyOptimistic.gen()` to yield promises only, OR (b) make WS3-B action handlers `async` functions that internally `await` the helper instead of `yield`-ing
3. Re-verify ALL 11 sites migrated in WS3-B (voteUp/Down on post, voteReplyUp/Down, followPost, followSpace, toggleBookmark, pinPost, acceptReply, setIdeaStatus, toggleReaction) — if the helper is broken, ALL of them are broken
4. Add a Playwright regression test that actually clicks each vote button and asserts DOM + DB change
