## Feature findings

### pro:reply-by-email — BROKEN

The core flow does not work end-to-end: free fires the email-headers filter as (headers, string $to, string $subject) while Pro's callback is typed (array $headers, int $user_id, $notification), so with the feature enabled every notification email send throws an uncaught TypeError (verified fatal in coercive mode) — and even if it didn't fatal, Pro never receives the notification object it needs, so a Reply-To token is never injected and no member can ever reply by email. The inbound half (webhook + IMAP + free's consumer) is wired and would work if a token existed, but it carries serious gaps: token expiry not covered by the HMAC (forgeable/permanent tokens), webhook signature validation that matches neither Mailgun nor SendGrid and is skipped when absent, zero loop prevention against autoresponders, no permission/state checks on the created reply, and failures are silently swallowed with no bounce to the member.

- Entry points — frontend: missing in effect - the member surface IS the Reply-To address in notification emails, and it is never injected because the free/pro filter contract is broken (fatal TypeError or early-return); no member opt-in/opt-out surface either | admin: present - settings tab with enable toggle, IMAP/webhook config, webhook URL + secret display (jetonomy-pro/includes/extensions/reply-by-email/class-extension.php:551-708), saved via nonce+cap-checked admin_init handler (:713-753); free fires the tab hooks (jetonomy/includes/admin/views/settings.php:25,34). No observability though - no last-poll status or error log when IMAP fails | REST: present - POST /jetonomy/v1/reply-by-email/inbound registered unconditionally (class-extension.php:451-459) with 503/400 guards when disabled; plus a WP-CLI journey (wp jetonomy-pro reply-by-email). Route serves ESPs, which is the correct design for this feature

- **[CRITICAL]** Expectation: Enabling Reply-by-Email adds a tokenized Reply-To header to notification emails so members can reply from their inbox, without breaking anything
  - Current: Contract mismatch: free fires apply_filters('jetonomy_notification_email_headers', $headers, $to, $subject) with $to = recipient email string (jetonomy/includes/adapters/class-wp-mail-adapter.php:33), but Pro hooks inject_reply_to_header(array $headers, int $user_id, $notification) with accepted_args 3 (jetonomy-pro/.../reply-by-email/class-extension.php:97, 272). Passing a non-numeric email string to the int $user_id param throws TypeError (no strict_types; verified with php -r) -> uncaught fatal on every notification email send while enabled, including the inline <=25-subscriber path inside a member's reply/post request (jetonomy/includes/notifications/class-notifier.php:240-251). Even with matching types, Pro never receives user_id/notification, so generate_token() is never reached and no Reply-To is ever emitted - the entire outbound half of the feature is dead.
  - Files: wp-content/plugins/jetonomy/includes/adapters/class-wp-mail-adapter.php, wp-content/plugins/jetonomy-pro/includes/extensions/reply-by-email/class-extension.php

- **[CRITICAL]** Expectation: Reply tokens are cryptographically bound to their expiry and are not reusable forever
  - Current: generate_token() signs only user:post:type - 'h' => wp_hash($user_id.':'.$post_id.':'.$type) (class-extension.php:180) - and decode_token() verifies expiry from the attacker-controlled plain-base64 'e' field before checking a hash that does not cover it (class-extension.php:210-217). Anyone holding any token (forwarded email, CC, leaked mailbox) can re-encode the JSON with a new expiry indefinitely; tokens are also deterministic per (user,post) with no nonce and no single-use/revocation, so the 7-day TOKEN_TTL is decorative.
  - Files: wp-content/plugins/jetonomy-pro/includes/extensions/reply-by-email/class-extension.php

- **[MAJOR]** Expectation: Inbound emails do not trigger notification loops (autoresponders, out-of-office, bounces)
  - Current: Neither process_imap_message (class-extension.php:397-437) nor rest_inbound (:468-529) checks Auto-Submitted, Precedence: bulk, X-Auto-Response-Suppress, or whether From is the site's own sender. A vacation autoresponder replying to the tokenized Reply-To posts a forum reply as the vacationing user, which fires a new notification, which triggers the autoresponder again - only the 10/hour/user rate limiter (:247-257) caps the loop at 10 junk replies per hour, forever. Expectation explicitly includes loop prevention; none exists.
  - Files: wp-content/plugins/jetonomy-pro/includes/extensions/reply-by-email/class-extension.php

- **[MAJOR]** Expectation: An emailed reply passes the same permission and state checks as a web reply (post exists, not locked/closed, user not banned, still has space access)
  - Current: Free's consumer Notifier::on_reply_from_email calls Reply::create() directly with only positive-int and non-empty-content guards (jetonomy/includes/notifications/class-notifier.php:325-341); Reply::create is a pure model with no permission checks (jetonomy/includes/models/class-reply.php:29-50) and nothing hooks jetonomy_before_create_reply. The REST controller's Permission_Engine gate is bypassed: a token holder can reply to locked posts, private spaces they were removed from, or deleted posts (row inserted before the parent-post lookup at class-reply.php:62, leaving orphan replies).
  - Files: wp-content/plugins/jetonomy/includes/notifications/class-notifier.php, wp-content/plugins/jetonomy/includes/models/class-reply.php

- **[MAJOR]** Expectation: Webhook endpoint authenticates inbound requests from the configured ESP
  - Current: rest_inbound only validates a signature IF a header is present ('if ( $signature_header )', class-extension.php:484), so a request with no header skips auth entirely; and the scheme - HMAC-SHA256 of the raw body compared to X-Mailgun-Signature or X-Twilio-Email-Event-Webhook-Signature (:481-487) - matches neither provider (Mailgun signs timestamp+token as POST body params; SendGrid Inbound Parse sends no signature, and the Twilio header is ECDSA from the Event Webhook, not inbound parse). Real providers therefore either bypass the check or get 403'd on every delivery, while the admin UI displays the secret as functional (:685-692). Effective auth is the token alone, which per the expiry gap is forgeable.
  - Files: wp-content/plugins/jetonomy-pro/includes/extensions/reply-by-email/class-extension.php

- **[MAJOR]** Expectation: When an emailed reply fails (expired token, rate limit, empty after stripping), the member finds out - e.g. a bounce email, like Discourse sends
  - Current: IMAP path ignores the return value of create_reply_from_email and marks the message Seen regardless (class-extension.php:432-436); expired/invalid tokens are silently dropped (:418-421). Webhook path returns the error to the ESP, never to the sender. No bounce/notification email exists anywhere; the failure ring buffer lives only in the CLI journey, which itself documents 'the extension itself ships no persistent failure logging' (jetonomy-pro/includes/cli/journeys/class-reply-by-email-journey.php:63-66). Members' replies vanish without a trace.
  - Files: wp-content/plugins/jetonomy-pro/includes/extensions/reply-by-email/class-extension.php, wp-content/plugins/jetonomy-pro/includes/cli/journeys/class-reply-by-email-journey.php

- **[MINOR]** Expectation: Inbound sender is verified against the token's user (defense-in-depth used by Discourse)
  - Current: From is never compared to the token user's email: rest_inbound never reads a From param (class-extension.php:494-523) and process_imap_message ignores $header->from (:407-416). The token in the To address is the sole identity proof.
  - Files: wp-content/plugins/jetonomy-pro/includes/extensions/reply-by-email/class-extension.php

- **[MINOR]** Expectation: Email parsing strips quotes, signatures, AND attribution lines; multipart bodies decode correctly
  - Current: Stripping covers only '>'-prefixed lines and a trailing '-- ' signature (class-extension.php:306-324) - the 'On <date>, <name> wrote:' attribution line ships into every posted reply. IMAP decoding reads the encoding of the TOP-LEVEL structure while fetching body part '1' (imap_fetchstructure(...)->encoding at :425 vs imap_fetchbody(..., '1') at :400), so for multipart messages the part's own quoted-printable/base64 encoding is missed and raw encoded text can be posted.
  - Files: wp-content/plugins/jetonomy-pro/includes/extensions/reply-by-email/class-extension.php

- **[MINOR]** Expectation: Site owner can see whether IMAP polling is working (last poll time, connection errors)
  - Current: process_imap_inbox returns silently on missing imap extension or failed imap_open with @-suppression (class-extension.php:360-372); no error is logged, no admin notice, no last-poll timestamp on the settings tab (:566-708). A typo'd IMAP password fails invisibly forever.
  - Files: wp-content/plugins/jetonomy-pro/includes/extensions/reply-by-email/class-extension.php

### free:spaces — MAJOR-GAPS

The spaces lifecycle is broadly well-built — create/browse/join/request/approve flows exist on all three surfaces, with good SQL-side visibility predicates (content_visibility_sql / listing_visibility_sql) applied to search, feed, sitemaps, oEmbed, and the topic listing. However, the feature's core promise ("privacy never leaks") is violated on three confirmed paths: guests can enumerate hidden/private spaces via the REST ?visibility= filter, the front-end /members/ page of private/hidden spaces has no gate, and the /roadmap/ page leaks idea titles+excerpts of private/hidden ideas spaces. Additionally two expected flow steps are missing on the frontend entirely: a member cannot leave a space, and a front-end space admin who sets join_policy=invite has no way to generate an invite link.

- Entry points — frontend: partial — browse/create/join/request/edit/members/my-spaces all present; missing: Leave Space (no UI anywhere calls DELETE /spaces/{id}/members/{user_id}) and invite-link generation for space admins; privacy gates missing on members + roadmap views | admin: present — Jetonomy → Spaces (paginated list, filters), space-edit tabs for members/access-rules/join-requests/invite-links via Spaces_Handler AJAX, all nonce+cap gated (jetonomy_manage_spaces) | REST: present — full CRUD + membership + join-requests + invites under jetonomy/v1/spaces; but ?visibility= filter bypasses the viewer gate and GET /spaces/{id}/members is unpaginated with per-row N+1

- **[CRITICAL]** Expectation: A hidden space must never be discoverable by non-members; a private space's card must not be exposed to guests (guest listing = public only).
  - Current: GET /jetonomy/v1/spaces?visibility=hidden (or private) returns ALL such spaces to guests and non-members: Spaces_Controller::list_items passes the request's visibility param straight into Space::list_visible (includes/api/class-spaces-controller.php:280,320-332), and list_visible SKIPS the viewer gate whenever an explicit visibility filter is set — 'if ( ! $visibility && ! $is_admin )' at includes/models/class-space.php:439-447, with the filter clause added unguarded at :433-436. The route's permission_callback is only the community-mode check (class-spaces-controller.php:51). Guests get titles, descriptions, slugs, settings JSON, member/post counts of every hidden space.
  - Files: wp-content/plugins/jetonomy/includes/api/class-spaces-controller.php, wp-content/plugins/jetonomy/includes/models/class-space.php

- **[CRITICAL]** Expectation: The member roster of a private or hidden space is members-only (the REST equivalent already enforces exactly this).
  - Current: templates/views/space-members.php has NO visibility/membership gate — it renders the full paginated roster (names, profile links, join dates, reputation, admin/mod badges) for ANY visitor including logged-out guests; 'space-members' is also absent from the auth-required route list (includes/class-template-loader.php:40). Contrast: REST get_members blocks non-members on private/hidden at includes/api/class-spaces-controller.php:685-689. A guest hitting /community/s/{private-or-hidden-slug}/members/ reads the whole roster.
  - Files: wp-content/plugins/jetonomy/templates/views/space-members.php, wp-content/plugins/jetonomy/includes/class-template-loader.php

- **[CRITICAL]** Expectation: The roadmap of a private/hidden ideas space is gated like the space itself, and private (is_private=1) posts never surface to ordinary viewers.
  - Current: templates/views/space-roadmap.php:34-46 runs a raw $wpdb query — SELECT * FROM posts WHERE space_id AND status='publish' — with no space-visibility gate and no is_private filter, then renders titles + 22-word content excerpts + vote/reply counts (lines 139-152). Guests can read idea content of private/hidden ideas spaces via /s/{slug}/roadmap/, and is_private posts with an idea_status appear on public roadmaps. (Also violates the 'no raw SQL outside models' house rule.)
  - Files: wp-content/plugins/jetonomy/templates/views/space-roadmap.php

- **[MAJOR]** Expectation: A member can leave a space they joined (standard in Discourse/Circle/Facebook Groups).
  - Current: REST DELETE /spaces/{id}/members/{user_id} exists and supports self-removal (includes/api/class-spaces-controller.php:783-819), but no template or JS ever calls it — grep across templates/ and assets/js finds zero 'Leave' affordance (space.php shows only Join/Follow, my-spaces.php cards have no leave action, space-members.php only offers role-change/ban to admins). A joined member is stuck unless a space admin removes them from wp-admin.
  - Files: wp-content/plugins/jetonomy/includes/api/class-spaces-controller.php, wp-content/plugins/jetonomy/templates/views/space.php, wp-content/plugins/jetonomy/templates/views/my-spaces.php

- **[MAJOR]** Expectation: A space admin managing their space from the frontend can invite people to an invite-only space.
  - Current: Front-end space-edit lets a space admin set join_policy=invite (templates/views/space-edit.php:167), and REST POST /spaces/{id}/invite authorizes space admins (class-spaces-controller.php:1036-1072), but the ONLY invite-generation UI is the wp-admin panel gated by jetonomy_manage_spaces (includes/admin/ajax/class-spaces-handler.php:594-650) which frontend-only space admins don't hold. A non-staff space admin can flip their space to invite-only and then has no way to invite anyone — the space is stranded. Three-entry-point rule: frontend surface missing for invite links.
  - Files: wp-content/plugins/jetonomy/templates/views/space-edit.php, wp-content/plugins/jetonomy/includes/admin/ajax/class-spaces-handler.php, wp-content/plugins/jetonomy/includes/api/class-spaces-controller.php

- **[MAJOR]** Expectation: Big-site rule: every REST list is paginated with COUNT and no per-row queries.
  - Current: GET /spaces/{id}/members calls SpaceMember::list_by_space($id) with NO limit (includes/api/class-spaces-controller.php:691), then prepare_member runs get_userdata + UserProfile::find_by_user PER ROW (:1152-1156). A 5,000-member space returns everything in one response with ~10k queries. The front-end template paginates at 25/page (space-members.php:29-39) but the REST surface — the one the mobile app will use — does not.
  - Files: wp-content/plugins/jetonomy/includes/api/class-spaces-controller.php, wp-content/plugins/jetonomy/includes/models/class-space-member.php

- **[MINOR]** Expectation: Private+open spaces are discoverable everywhere so members can find and join them (the exact bug the 1.5.x 'Private+open spaces invisible' fix addressed).
  - Current: The fix landed only on template listings: home/category use listing_visibility_sql (public+private discoverable, includes/models/class-space.php:189-212 via :226-228), but REST GET /spaces default listing still uses the old 'public OR member' rule inside list_visible (class-space.php:445). A logged-in non-member browsing via REST (mobile app, headless) never sees private+open spaces they could join — the two directory surfaces disagree.
  - Files: wp-content/plugins/jetonomy/includes/models/class-space.php, wp-content/plugins/jetonomy/includes/api/class-spaces-controller.php

- **[MINOR]** Expectation: Admin bulk operations stay bounded on large sites.
  - Current: ajax_sync_access_rule loads get_users(fields=>ID) for EVERY user on the site, then loops adapters + a per-user SpaceMember::is_member query (includes/admin/ajax/class-spaces-handler.php:509-525). On a 10k-user site this is an unbounded request that will time out. Admin-only surface, but violates the big-site checklist.
  - Files: wp-content/plugins/jetonomy/includes/admin/ajax/class-spaces-handler.php

- **[MINOR]** Expectation: Approve-join-request handlers validate that the request belongs to the posted space (the REST handler does: request must match space and be pending).
  - Current: Admin ajax_approve_join_request never checks $request->space_id === $space_id — it approves request N then adds the requester to whatever space_id was posted (includes/admin/ajax/class-spaces-handler.php:552-558). REST's resolve_join_request does verify (class-spaces-controller.php:933). Cap-gated to site staff so exploitation risk is low, but mismatched params silently admit a user to the wrong space.
  - Files: wp-content/plugins/jetonomy/includes/admin/ajax/class-spaces-handler.php

- **[MINOR]** Expectation: Membership mutations don't happen on unauthenticated-nonce GET requests; invite acceptance asks for confirmation.
  - Current: GET /jetonomy/v1/invite/{token} joins the logged-in caller immediately (class-spaces-controller.php:193-201 → InviteLink::accept), and the invite landing template auto-joins on page load with zero confirmation then meta-refreshes into the space (templates/views/invite.php:28,66-68). A GET that mutates membership is force-joinable via <img src> / prefetch for any logged-in user who can be lured to the URL.
  - Files: wp-content/plugins/jetonomy/includes/api/class-spaces-controller.php, wp-content/plugins/jetonomy/templates/views/invite.php

### free:voting-reputation — MAJOR-GAPS

The primary member flow is genuinely solid: REST voting (POST/DELETE /posts|replies/{id}/vote) with transaction-wrapped three-state Vote::cast, DB-level double-vote prevention (UNIQUE user_object key), self-downvote block, rate limiting, optimistic frontend with revert/error toasts, SSR voted-state, configurable points map, and admin-tunable trust thresholds. But the feature breaks the house three-entry-point contract at behavior level: the Abilities/MCP vote path skips reputation, self-vote guards, and rate limits entirely; trust auto-promotion cron re-processes the same first 500 profiles forever so users beyond 500 never promote; and the admin threshold matrix saves reputation/replies_received fields the evaluator never reads.

- Entry points — frontend: present — vote buttons with optimistic UI + revert on post/feed/reply cards (assets/js/view.js:1190-1420), SSR voted-state + scores in all templates, leaderboard + profile show reputation | admin: partial — Settings→Permissions has trust thresholds, rate limits, reputation points cards (includes/admin/views/settings.php:334-504); Users list shows reputation + trust-level change (views/users.php:89-100). No recount button outside import flow; no post_created/reply_created rows in the points card | REST: present — vote/unvote routes with REST_Auth (class-votes-controller.php:29-67), GET /leaderboards, reputation in user payloads (class-base-controller.php:366), admin-gated POST /admin/recount (class-admin-controller.php:31)

- **[MAJOR]** Expectation: Voting behaves identically through every entry point: same guards, same reputation accrual, whether the vote arrives via REST, the Abilities/MCP surface, or CLI
  - Current: Abilities execute_vote (includes/class-abilities.php:1333-1363) calls Vote::cast directly with no Reputation::award/revoke, no self-downvote block, and no Rate_Limiter check — all three live only in the REST controller (class-votes-controller.php:130-184). CLI Content_Journey::vote (includes/cli/journeys/class-content-journey.php:270-300) and the wpForo importer (class-wpforo-importer.php:356) likewise skip reputation. The model fires jetonomy_vote_cast/jetonomy_vote_retracted (class-vote.php:105,142,174-175) precisely so listeners can react, yet free's own reputation isn't hooked to them — it's controller-bound. Result: votes via the MCP tools (jetonomy_vote_post/jetonomy_vote_reply) update scores but never move reputation, permit self-downvote, and bypass rate limits; vote_score and reputation permanently drift apart across surfaces.
  - Files: wp-content/plugins/jetonomy/includes/class-abilities.php, wp-content/plugins/jetonomy/includes/api/class-votes-controller.php, wp-content/plugins/jetonomy/includes/models/class-vote.php, wp-content/plugins/jetonomy/includes/cli/journeys/class-content-journey.php

- **[MAJOR]** Expectation: Trust-level auto-promotion eventually evaluates every member on a large community (2000+ users is the house baseline)
  - Current: Cron::evaluate_trust_levels (includes/class-cron.php:124-129) selects profiles with `WHERE trust_level < 4 LIMIT 500` — no ORDER BY, no OFFSET, no watermark/cursor. Auto-earnable levels max at 3 (< 4), so evaluated rows never leave the result window; every 12h run re-scans the same first ~500 profiles and users 501+ are never auto-promoted. The docblock ('Processes at most 500 per run to avoid max_execution_time') implies batched coverage that does not exist.
  - Files: wp-content/plugins/jetonomy/includes/class-cron.php

- **[MAJOR]** Expectation: Every threshold field the admin can save on Settings → Permissions actually gates promotion ('settings save AND take effect')
  - Current: The UI renders and the sanitizer persists a full 4-column matrix (posts, days_active, reputation, replies_received) for levels 1-3 (settings.php:358-361, class-admin.php:431-436), but Trust_Evaluator::evaluate_level ignores `reputation` at level 1 (class-trust-evaluator.php:46-50) and ignores `replies_received` at levels 2 and 3 (class-trust-evaluator.php:58-62, 70-74). An owner raising 'Replies Received' for Level 2 or 'Reputation' for Level 1 sees it saved but it never applies — the exact 'saved but not applied' contract-bug class the house rules call out.
  - Files: wp-content/plugins/jetonomy/includes/trust/class-trust-evaluator.php, wp-content/plugins/jetonomy/includes/admin/views/settings.php, wp-content/plugins/jetonomy/includes/admin/class-admin.php

- **[MINOR]** Expectation: Members cannot farm reputation from their own votes (Stack Overflow/Discourse block self-vote rep; Reddit excludes self-votes from karma)
  - Current: Self-upvote is allowed by default (filterable, class-votes-controller.php:159-162) AND maybe_adjust_reputation awards the author — i.e. the voter themselves — post_upvoted +10 / reply_upvoted +5 (class-votes-controller.php:291-320 has no self-vote exclusion). A new user can create a post (+2) and self-upvote (+10): 12 rep per post toward the TL2 reputation=50 gate. Allowing the vote is a documented design call; awarding reputation for it is a rep-economy hole.
  - Files: wp-content/plugins/jetonomy/includes/api/class-votes-controller.php

- **[MINOR]** Expectation: GDPR erasure leaves displayed counts consistent
  - Current: Privacy eraser deletes all the user's jt_votes rows (includes/class-privacy.php:228) without applying score deltas or triggering Recount, so every post/reply they voted on keeps a stale denormalized vote_score everywhere it displays until someone manually runs recount. Similarly Post::delete/Reply::delete (class-post.php:206-219, class-reply.php:174-208) never purge jt_votes rows for the deleted object — orphans accumulate forever.

### pro:polls — MAJOR-GAPS

Polls has all three entry points (composer builder + inline creator + vote UI on the frontend; paginated/searchable admin list with close/delete; full REST CRUD plus 3 Abilities and a WP-CLI command), and the single-choice happy path is solid end-to-end. ### pro:reactions — MAJOR-GAPS
