# Jetonomy 1.3.6 — Release Spec

**Status:** queued. No code to ship yet.
**Scope boundary:** free plugin unless noted. `jetonomy-pro` is unaffected.

## Themes in this release

1. **SEO hardening for UGC** — close the indexing gaps for categories, tags, sort/pagination, and emit social + article meta (Section A).
2. **Role & permission clarity** — resolve Basecamp card `9725751235` (Shashank): make the four-layer user/role model obvious to admins and stop the terminology collision between WP "admin" and space "admin" (Section B).
3. **Admin REST API hardening for content seeding** — close the gaps that forced workaround scripts during the 2026-04-17 community rebuild. WordPress core owns user and post CRUD; Jetonomy owns everything space-specific (Section C).

## Section A — SEO Hardening for UGC

### Why

Forum content is user-generated, sits in custom `wp_jt_*` tables, and is invisible to third-party SEO plugins unless we feed them. Today's free build covers Spaces + Topics (via WP core `wp_sitemaps`), emits `DiscussionForumPosting` / `QAPage` / `BreadcrumbList` schema, and applies customisable title patterns. The gaps are categories, tags, sort/page canonicalisation, crawl efficiency, social meta, and the admin's ability to see which SEO surface is active.

## Design principle — one pipeline, three exits

We keep a single source of truth for forum URLs. That source publishes through whichever exit the admin's environment exposes:

1. **Always:** WP core `wp_sitemaps` registry (what we already use). This is the WordPress-native API every well-behaved SEO plugin respects.
2. **Auto-detect and bridge:** When Yoast / Rank Math / AIOSEO is active, also register via that plugin's documented filter so our URLs merge into its sitemap index. Admin gets one unified sitemap, not two.
3. **Admin visibility:** Settings → SEO shows a "Sitemap Status" card with the live detected bridge, counts per entity, and direct links.

When the admin disables Yoast, WP core's sitemap takes back over with zero admin action.

## Changes — free plugin

### 1. New sitemap providers

- `Categories_Sitemap_Provider` — reads `wp_jt_categories`, 2k per page, `lastmod` = category `updated_at` or latest child space activity.
- `Tags_Sitemap_Provider` — reads `wp_jt_tags`, 2k per page, `lastmod` = latest tagged-post `updated_at`.

Register both in `Sitemap::register()` alongside existing providers.

### 2. SEO plugin bridges

New namespace `Jetonomy\SEO\Bridges\*`. Each bridge ~40 lines, same data source:

- `Yoast_Bridge` — hooks `wpseo_sitemap_index` + `wpseo_sitemap` to emit our four provider URL lists.
- `Rank_Math_Bridge` — hooks `rank_math/sitemap/providers` with a custom provider class.
- `AIOSEO_Bridge` — hooks `aioseo_sitemap_additional_urls`.

Detection at `init` priority 20 via class-exists / defined-constant checks. Only one bridge loads at a time.

### 3. Canonical + pagination hygiene

Template-layer touches in `templates/views/single-post.php` and `templates/views/space.php`:

- Sort variants (`?sort=latest|oldest|best`) → `<link rel="canonical">` points at the no-query URL (default sort).
- Paginated topic listings (`?page=N`) → each page self-canonicalises. Do NOT point page N back at page 1 (per Google's 2023 guidance).

### 4. Crawl efficiency

- `Last-Modified` header on single topic + space views, driven by `last_reply_at` (topic) / `last_activity_at` (space).
- Handle `If-Modified-Since` with 304 short-circuit before template rendering.

### 5. Social + article meta

Extend `Schema_Markup::output_schema()` to also emit on single topics:

- `og:type="article"`, `og:title`, `og:description`, `og:url`, `og:image` (first embedded image or space OG image fallback)
- `article:published_time`, `article:modified_time`, `article:author`
- `twitter:card="summary_large_image"`, `twitter:title`, `twitter:description`, `twitter:image`

### 6. UGC link hardening

Post + reply body renderer (`jetonomy_format_content`) — add `rel="ugc nofollow"` to every external `<a>` in user-generated content. Internal links untouched. Protects site authority and matches Google's explicit UGC guidance.

### 7. Admin — Sitemap Status card

New card in Settings → SEO tab:

- Detected bridge name (Yoast / Rank Math / AIOSEO / WP core)
- Link to the active sitemap index
- Live counts per entity (spaces, topics, categories, tags) read from the provider classes
- Toggle to disable all bridges (fallback to WP core only) for admins who want manual control

## Out of scope — deliberately

- hreflang — no i18n story in 1.3.6
- Profile `Person` schema — profiles stay `noindex` by default, skip until the admin opts in
- Reply-level canonicals — replies inherit the topic URL via fragment; no separate page to canonicalise
- SEO Pro extension changes — per-space overrides already work; this release is about community-level coverage

## Verification plan

Throwaway Local site, for each SEO plugin {none, Yoast, Rank Math, AIOSEO}:

1. Activate the plugin.
2. Visit `/sitemap_index.xml` (or `/wp-sitemap.xml` for none) — confirm our four URL types are present.

## Section B — Role & Permission Clarity

Resolves: Basecamp card [`9725751235`](https://3.basecamp.com/5798509/buckets/46596502/card_tables/cards/9725751235) — "Problem is user role clarity" (Shashank, March 27).

### The problem

The mechanics work. The mental model doesn't.

Four concepts coexist, three use overlapping words:

| Layer | Stored where | Scope | Example |
|-------|--------------|-------|---------|
| **WordPress role** | `wp_usermeta` (core) | Site-wide | `administrator`, `editor`, `subscriber` |
| **Jetonomy capability** | Custom caps mapped to WP roles | Plugin-wide | `jetonomy_manage_spaces`, `jetonomy_moderate_content` |
| **Space role** | `wp_jt_space_members.role` | Per-space | `viewer`, `member`, `moderator`, `admin` |
| **Trust level** | `wp_jt_user_profiles.trust_level` | Community reputation | 0–5 (new member → elder) |

"admin" means the WP site admin in one place and the space owner in another. No admin screen explains how a WP user becomes a Jetonomy user (answer: it's automatic — profile is lazily created on first community visit).

### Design principle — clarity via naming + visibility

We don't invent new mechanics. We rename where the overload causes confusion, surface the state where admins look for it, and document the flow inside the product.

### Changes

#### 1. Rename space role `admin` → `owner`

- Schema: `jt_space_members.role ENUM('viewer','member','moderator','admin')` → `ENUM('viewer','member','moderator','owner')`.
- Migration `1_2_6`: `ALTER TABLE` + in-place `UPDATE role='owner' WHERE role='admin'`.
- Code-wide rename of the string `'admin'` in space-role context only — every call site reviewed; callers that conflate this with the WP admin check get fixed.
- UI strings: "Make admin" → "Make owner", "Space admin" → "Space owner", member badges in member lists updated.

Why: "Space owner" is unambiguous. Leaves `moderator` untouched (no collision). Does not rename `administrator` (WP role) or `jetonomy_manage_spaces` (capability).

#### 2. Role-to-capability mapping admin page

New admin tab Settings → Permissions:

- Matrix: rows = WP roles (admin/editor/author/contributor/subscriber/custom), columns = each `jetonomy_*` capability.
- Checkboxes let admins grant/revoke caps per WP role without touching code.
- Writes via `WP_Roles::add_cap` / `remove_cap` (standard WP API) so changes persist and survive plugin deactivation.
- "Reset to defaults" button restores the shipped mapping.

#### 3. Extend Users admin list

`wp-admin/admin.php?page=jetonomy-users` gets three new columns:

- **WP Role** (from `$user->roles`)
- **Trust Level** (from `wp_jt_user_profiles.trust_level`, already fetched)
- **Spaces** (count of rows in `jt_space_members` where `user_id = ?`, cached 60s)

One glance shows the full role picture for any user.

#### 4. User & Role Model explainer card

New card on Settings → General, rendered once at the top (dismissable):

- Short prose: "Jetonomy users are WordPress users. When a WP user first visits the community, their Jetonomy profile is created automatically. You don't need a separate signup flow."
- Four-line legend showing the layer table above.
- Link to the new docs page.
- `jetonomy_dismissed_role_explainer` option hides it once clicked.

#### 5. Docs page

`docs/website/admin-settings/06-users-and-roles.md` — the authoritative explanation of all four layers, registration flow, and the mapping UI. Added to the docs sitemap and linked from the in-admin explainer.

#### 6. "How to make someone a space owner" in-product help

On the space Members tab, a help tooltip next to the role dropdown explains what each role can do and links to the docs page.

### Out of scope — deliberately

- No change to capability names (breaking for anyone who extended us via filters).
- No new WP roles shipped; admins can create their own and the matrix respects them.
- No change to trust-level system.
- No change to BuddyPress group sync (still maps BP mod → space moderator, BP admin → space owner after the rename).

### Verification plan

1. Fresh install → activate → Settings → General → explainer card visible, dismissable.
2. Upgrade from 1.3.5 → migration `1_2_6` runs → `SELECT COUNT(*) FROM wp_jt_space_members WHERE role='admin'` returns 0 (all migrated to `owner`).
3. Make a test user Editor → Settings → Permissions → grant `jetonomy_moderate_content` → test user can now access `/community/mod/`.
4. Admin → Users → three new columns visible, sorted + filterable.
5. BuddyPress round-trip: promote BP group member to admin → linked space's member row shows `owner`, not `admin`.
6. REST: `POST /jetonomy/v1/spaces/:id/members` with `{user_id, role: 'owner'}` succeeds. Legacy callers sending `role: 'admin'` get a clear deprecation response (`role=admin is deprecated; use role=owner`).
7. Docs page renders at `/community/docs/users-and-roles/` on the marketing site.

## Section C — Admin REST API Hardening for Content Seeding

### Why

Rebuilding community.wbcomdesigns.com on 2026-04-17 (after the plugin-reset wiped `wp_jt_*` tables) needed 137 topics + 147 replies authored by 12 distinct personas. The Jetonomy REST API forced six workarounds. Every agency migrating from bbPress / wpForo / Discourse will hit the same walls.

### Design principle — defer to WP core where it already works

**Do not duplicate WordPress core functionality.** WP core owns:

- User CRUD — `POST /wp/v2/users` creates accounts, `PATCH /wp/v2/users/{id}` edits them, role management built in. Jetonomy does not need its own user-creation or edit-other-user REST surface.
- Post CRUD on standard post types — Jetonomy's posts table is custom, so we build our own, but mirror WP core's conventions (`author` override, `status` filter, `_method=DELETE?force=true`) so REST clients have one mental model.
- Permissions — `manage_options`, `edit_others_posts`, etc. Reuse.

Jetonomy owns everything space-specific — memberships, space roles, trust levels, moderation queue — because WP core has no native concept of them.

### Changes

#### 1. Space membership — missing entirely from REST

Today's only path to add `(space_id, user_id, role)` is admin-AJAX (`wp_ajax_jetonomy_add_space_member`) which requires a nonce. Unusable from remote tooling or migrations.

Add:

- `POST /admin/spaces/{id}/members` body `{user_id, role}` — requires `manage_options`. Creates the `jt_space_members` row, fires auto-join hooks, respects the new `owner` role after Section B.
- `DELETE /admin/spaces/{id}/members/{user_id}` — removes.
- `PATCH /admin/spaces/{id}/members/{user_id}` body `{role}` — changes role.
- `GET /admin/spaces/{id}/members?role=...` — list with optional role filter.

User accounts themselves continue to be created via WP core `POST /wp/v2/users`. We only manage the `jt_space_members` join table.

#### 2. `author` on POST `/spaces/{id}/posts` and POST `/posts/{id}/replies`

Mirror `POST /wp/v2/posts` convention: admin caller (with `edit_others_posts` or `manage_options`) can pass `author` (WP user ID). Server validates the user exists and assigns. Non-admin callers get their own ID stamped silently (current behaviour).

Today: admin sends `{author_id: 5083288}` → server returns `author_id: 5083287`. That's a silent drop, not even a warning.

#### 3. `status` query param on list endpoints must actually filter

`GET /spaces/{id}/posts?status=spam` currently returns the same set as `?status=publish`. Fix the controller to honour the param. Admins see any requested status; members see `publish` + their own drafts; anonymous sees `publish` only.

Same fix on `GET /spaces/{id}/posts/{id}/replies`.

#### 4. Cascade-delete children when parent is hard-deleted

`DELETE /posts/{id}?force=true` today leaves the post's replies behind as orphans (`jt_replies` rows with a `post_id` that no longer resolves). Recount still counts them against the author.

Fix: on hard-delete, wrap `DELETE FROM jt_posts` + `DELETE FROM jt_replies WHERE post_id = ?` in one transaction. Soft-delete (trash) stays as-is — replies remain trashed with their parent.

#### 5. `skip_akismet` flag for admin-driven content seeding

Posts authored by TL0 personas via bulk REST trigger Akismet, even when the POST is clearly an admin-driven seed. Scripts have to run a follow-up unflag pass.

Add a `skip_akismet` boolean on POST `/spaces/{id}/posts` and POST `/posts/{id}/replies`. Only honoured when the caller has `manage_options`. Aligns with the 1.3.4 admin/moderator Akismet-skip logic already in the UI path.

#### 6. Consistent `?force=true` across all resource endpoints

`DELETE /posts/{id}?force=true` hard-deletes, `DELETE /replies/{id}?force=true` hard-deletes, but `DELETE /spaces/{id}?force=true` behavior is undocumented. Audit every DELETE route and pick one convention:

- Default (no flag): soft delete (trash / status flip).
- `?force=true`: hard delete with cascade.

Document in `docs/developer/01-rest-api.md`.

### Out of scope — deliberately

- **No Jetonomy user-creation endpoint.** WP core `/wp/v2/users` covers it.
- **No bulk import CLI beyond what `wp jetonomy` already offers.** CLI is a separate surface; REST is the web-facing one.
- **No WP-AJAX deprecation.** Admin UI continues to use AJAX; REST is additive for headless clients.

### Verification plan

1. PHPUnit integration test for each new admin endpoint (auth, happy path, unauthorized).
2. Migration regression: run the 2026-04-17 community rebuild script against a fresh install using only the new endpoints — should complete without workaround scripts.
3. REST API docs updated: `docs/developer/01-rest-api.md` gains a "Space memberships" section and a "Compared to WP core" callout listing which endpoints to use for what.
4. Acceptance: re-run the full seeder in under 5 min with no spam-unflag pass and no orphan cleanup pass.
