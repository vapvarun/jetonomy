# Jetonomy 1.3.7 — Release Spec

**Status:** queued. No code to ship yet.
**Scope boundary:** free plugin; Pro unaffected unless noted.

## Theme in this release

**Space type specializations.** Today all four space types (`forum`, `qa`, `ideas`, `social`) render through the same `templates/views/space.php` with only cosmetic label differences. Only QA has a genuine behavioral branch (accepted-answer). Ideas and Social are advertised as distinct community types but deliver no distinct experience. This release gives each type a real identity.

---

## Why

Today nobody — not a customer, not a support rep, not a sales engineer giving a demo — can tell a forum space from an ideas space or a Q&A space just by looking at it. They all render the same topic list. The only cue is a small badge in the admin, which end users never see. A space type is supposed to be a *visible* product decision, not a label attribute.

**Design bar for 1.3.7: the type of a space must be obvious at a glance — under 2 seconds, without reading any labels.** If two people open two different spaces side-by-side, they should immediately say "oh, that one is Q&A, that one is a roadmap" from layout alone.

Concretely:

- **Forum spaces** — topic list with title, excerpt, author, reply count, last-reply-at. The default; this is what people think "WordPress forum" means.
- **Q&A spaces** — topic list with question / answer icon, a resolved/open badge on every card, resolved tab filter at the top. The visual cue: "this space is for things that have right answers."
- **Ideas spaces** — vote-count on the LEFT of every card (big, clickable), status pill (`open`/`planned`/`in-progress`/`shipped`), kanban view toggle in the header. The visual cue: "this space is a roadmap."
- **Social spaces** — media-first cards (image/video occupies 60%+ of the card), compact text, inline composer at the top of the feed, replies fold under each card. The visual cue: "this space is a feed, not a forum."

There is also a naming inconsistency to fix: admin UI uses `feed`, REST API accepts `social`. Both surfaces must agree on one term — `social` wins.

---

## Design principle

Template-per-type + shared partials. Each type gets its own view in `templates/views/space/{type}.php` routed from a single dispatcher. Shared UI (breadcrumb, author chip, vote controls, reply composer) stays as partials reused across types. No per-type duplication of the header, footer, or sidebar.

## Changes

### 1. Naming fix — hard-rename `feed` → `social`

Customer count is small; backward compatibility is not worth the carry-cost. Hard-rename:

- `jt_spaces.type` ENUM: `'forum','qa','ideas','social'`. `'feed'` is removed.
- Migration `1_2_7` runs one-shot `UPDATE jt_spaces SET type='social' WHERE type='feed';` then `ALTER TABLE` to drop `'feed'` from the ENUM.
- Admin dropdown: only `'social'` appears. REST rejects `'feed'` with a clear 400 error (no alias, no deprecation header).
- All internal string references (`$space_type_map`, CSS classes, PHP constants) use `'social'` everywhere. Grep for `'feed'` in the codebase after the change and ensure zero hits outside historical changelog entries.

### 2. Space type is immutable after creation

Once a space is created with `type='ideas'`, it stays `ideas` for life. This is enforced in three places:

- **Schema:** no change needed; the column is just never updated after insert for existing spaces.
- **Admin UI:** `templates/admin/views/space-edit.php` renders the Type field as read-only (`<p>` with a value + helper text "Space type cannot be changed after creation. To change, create a new space."). Only the create form exposes the dropdown.
- **REST:** `PATCH /spaces/{id}` silently drops any `type` field in the request body. `POST /spaces` honours it. A validation helper in `Spaces_Controller` enforces this so every future update path shares the same rule.
- **CLI:** `wp jetonomy space update <id> --type=...` returns an error pointing at "create a new space."

**Why lock it:**
- Per-type schema additions (`status_pipeline` column for Ideas, inline-composer state for Social) assume the type doesn't flip. A forum → ideas conversion would leave every existing post with no vote score + no status; an ideas → forum would strand the status column; social → forum would strand inline-reply data.
- Per-type URL structures differ subtly (Social has `/s/slug/?since=<cursor>`, Ideas has `/s/slug/?status=planned`). Flipping a type breaks inbound links.
- Conceptually: a "community type" is a founding decision, not a reversible attribute. This mirrors how Discourse treats sub-categories.

Explicit docs note in `readme.txt` + `CLAUDE.md`: type is immutable.

### 3. Template dispatch

- New `templates/views/space/dispatch.php` — the single entry, reads `$space->type` and requires the correct sub-template.
- New `templates/views/space/forum.php` — current content of `space.php` moved here unchanged.
- New `templates/views/space/qa.php` — forum template + resolved/unresolved filter tab (next item).
- New `templates/views/space/ideas.php` — voting-sorted list + status column (next item).
- New `templates/views/space/social.php` — media-first card layout + inline composer (next item).

`templates/views/space.php` becomes a thin shim that requires `space/dispatch.php` (for theme overrides that hardcode `jetonomy/templates/views/space.php`).

### 4. QA specialization

Addition on top of the current QA accept-answer behavior:

- Filter tabs at the top of the topic list: **All** / **Open** / **Resolved** — driven by the existing `is_resolved` column.
- Resolved topics get a muted card state (border color + "Resolved" tag) so scanning the list shows what still needs answers.
- Default sort on **Open** tab: newest first (bubble unanswered questions up); default sort on **Resolved**: most-voted answer.

Small change. Reuses existing data (`is_resolved`, `accepted_reply_id`).

### 5. Ideas specialization

First-class product-roadmap behavior:

- New column `jt_posts.status_pipeline ENUM('open','planned','in-progress','shipped','declined')` DEFAULT 'open'. Only used when the parent space's `type='ideas'`.
- Template renders a kanban-style status pipeline view OR a voting-sorted list (toggle in space header).
- Default sort on ideas listing: vote score DESC (so the community sees what they wanted voted up).
- **Merge duplicates** action in the post's more-options menu for space moderators/owners: select a "canonical" idea, merge votes from this one into it, trash the duplicate. Adds `jt_post_merges` audit table.
- Status-change action in the more-options menu, wired to the new `status_pipeline` column. Status change fires a `jetonomy_idea_status_changed` action so the digest extension can notify voters.
- Kanban column-change is drag-and-drop via existing Interactivity API store.

Migration `1_2_7` adds the column + the new `jt_post_merges` table.

### 6. Social specialization

Feed-style community for photo/status/link sharing:

- Card partial `templates/partials/space-social-card.php` — media-first layout: image/video first, compact text, author + time + reactions below. Unlike the forum topic card (title first, body preview second).
- Inline composer at the top of the listing — no "New Post" button that routes to a separate page. `<textarea>` + upload + post-as-you-type draft autosave. Submitting stays on the page and prepends the new card.
- Infinite scroll cursor pagination (server already supports it). Tighter vertical spacing than forum/qa.
- Replies render inline under each card (compact, stacked), like Facebook-style comments. Not a separate thread page.

Reuses existing `Post` + `Reply` models. No schema change here beyond reusing `type='status'`.

### 7. Forum specialization

**No functional change.** Template moves to `templates/views/space/forum.php`, content identical. Serves as the "default" / reference template for the other three.

## Gap analysis — scenarios the current spec doesn't cover yet

Done before implementation so we don't ping-pong mid-build. Each row is now explicitly in or out of scope.

### In scope — added to the build

| Scenario | What needs to happen |
|----------|---------------------|
| **Topic moved between spaces** | Today `POST /posts/{id}/move --target-space=<id>` exists. When a topic with `status_pipeline` leaves an Ideas space into a non-Ideas space, clear the column (set NULL). When a topic enters an Ideas space, default to `'open'`. Handled in `Post::move()`. |
| **Guest / anonymous rendering** | Each type needs a guest view. Forum = current behavior. QA = can see questions + votes, can't accept. Ideas = see votes + status, can't vote (login-to-vote nudge inline). Social = see feed + replies, can't post (login-to-post nudge on the composer). Spelled out in `templates/views/space/{type}.php`. |
| **Mobile design (390px) per type** | Each type gets a 390px screenshot in the verification plan. Ideas kanban stacks to a single column with status tabs at the top (can't do 5 columns side-by-side on mobile). Social stays media-first, cards scale down. QA filter tabs collapse to a dropdown. |
| **Accessibility equivalents for drag/scroll** | Kanban DnD must have a keyboard equivalent: per-card status dropdown always rendered, DnD is the enhancement. Infinite scroll must expose a visible "Load more" button for keyboard users + reduced-motion preferences. ARIA labels on all new interactive controls. |
| **Pro extension audit for `'feed'`** | Before the migration, grep both plugins for `'feed'` references as a space-type value; update any Pro code that hardcodes it (email-digest, webhooks, analytics are the candidates). Part of the release checklist, not a separate task. |
| **Permissions matrix for new actions** | Status change on Ideas: space `moderator` + `owner` + `jetonomy_manage_spaces`. Merge duplicates: same three. Accept answer on QA (existing): topic author + the three above. Documented in `docs/permissions.md`. |
| **SEO per type** | `status_pipeline='declined'` ideas get `<meta name="robots" content="noindex, follow">`. Schema.org per type: Forum = `DiscussionForumPosting` (already), QA = `QAPage` (already), Ideas = `DiscussionForumPosting` + `InteractionCounter` for vote count, Social = `SocialMediaPosting`. Already flagged in the 1.3.6 SEO spec — verify 1.3.7 Ideas/Social land those emitters. |
| **RSS feed shape** | Each type gets a type-appropriate feed: Social shows `<media:content>` for images, Ideas shows `<category>` for status, QA flags `<jetonomy:accepted>1</jetonomy:accepted>` on resolved. Default feed = chronological. |
| **Status filter query param contract** | REST: `GET /spaces/{id}/posts?status_pipeline=planned` filters ideas list. Invalid values return 400 with the valid set. Listed alongside the 1.3.6 Section C `?status=` fix (both are filter gaps). |
| **Merge concurrency** | Two moderators merging A→B and B→A simultaneously: use a row-level lock on `jt_posts` during merge, fail the second merge with 409. Test covered in the concurrency suite. |

### Out of scope — deliberately (punted to later, with reason)

| Scenario | Why punted |
|----------|-----------|
| **Cross-type conversion** (forum → ideas) | Types locked after creation per this spec. Conversion is a "create new space, bulk-move topics" flow that doesn't need plugin support; moderators script it via CLI. |
| **Per-type custom fields** | Belongs in the Custom Fields Pro extension, not the free plugin core. |
| **Per-type email digest shapes** | The digest extension already receives post metadata; it can render per-type differently. Tracked separately. |
| **Ideas analytics dashboard** (conversion funnel from open → shipped) | Analytics extension in Pro. |
| **Native dark mode for new card partials** | The token system already cascades dark mode via `--jt-*` vars. Verify no hard-coded values creep in; no separate dark-mode design work. |
| **RTL support** | `jetonomy.css` already uses logical properties for the existing layouts. New partials must follow same rule; no separate RTL task. |
| **Print stylesheet** | Out of scope for any space type. |
| **Bulk status change + CSV export for Ideas** | Ship basic status change first. Bulk actions deferred to 1.3.8+ after real usage data. |
| **A native app for Social spaces** | Referenced in Feature Requests space as a future idea; not in any near-term release. |

### Dependencies we're declaring upfront

- **1.3.7 requires 1.3.6 to ship first.** DB version chain 1.2.5 → 1.2.6 → 1.2.7; migrations are not skip-safe.
- **Pro compatibility**: `jetonomy-pro` v1.3.7 must ship the same day; lockstep rule already enforced by `bin/build-release.sh`.
- **WP core minimum**: same as 1.3.6 (WP 6.7+, PHP 8.1+). No bump needed.

## Out of scope — deliberately

- **Cross-type conversion** (turn a forum space into an ideas space mid-life). Edge case; punt.
- **Per-type custom fields** (e.g., Ideas "estimated impact" metadata). Punt to the Custom Fields Pro extension.

## Stability & non-regression guarantees

**1.3.7 ships only if every one of these pre-commits holds.** Treat each as a go / no-go gate; the release is blocked until all pass.

### A. Existing data survives byte-identical where possible

- Migration `1_2_7` runs `feed` → `social` as an `UPDATE jt_spaces SET type='social' WHERE type='feed'`; no rows deleted, no URLs changed, no slugs changed.
- New column `jt_posts.status_pipeline` is nullable + defaults to `'open'`; existing rows get the default silently. No existing query that selects `jt_posts.*` breaks.
- New `jt_post_merges` table is additive.
- Upgrade path tested on a **production snapshot** (community.wbcomdesigns.com + a second customer site with >5k posts) before tag is cut. `wp db export` → test migration → verify `SELECT COUNT(*) FROM jt_*` matches pre-migration counts.

### B. Theme overrides keep working (for overrides people actually use)

- The four new templates (`space/forum.php` etc.) are individually override-able via `yourtheme/jetonomy/views/space/{type}.php` following the same theme-loader convention.
- `templates/views/space.php` is retired as part of this release (no alias, no shim). Theme authors who copied it get an upgrade note in the release changelog with a one-liner: "Copy `views/space/forum.php` instead." Customer count is small enough that a direct notice + Basecamp ping covers the population.

### C. Forum spaces render pixel-identical to 1.3.6

- `space/forum.php` is a verbatim copy of 1.3.6's `space.php` — no refactors, no "while we're in there" cleanups. Git diff of the moved file is zero-delta.
- Playwright screenshot comparison: 1.3.6 forum-space page vs 1.3.7 forum-space page → pixel-identical at 1440px and 390px.
- Any forum-space regression blocks the release.

### D. REST API — clean break, no backward compat

- Customer count is small; backward compat is explicit non-goal. `type='feed'` is removed from ENUM and REST accepts list. POST/PATCH with `type: 'feed'` returns `400` with a clear message pointing at `social`.
- `PATCH /spaces/{id}` rejects changes to the `type` field once the space exists (enforces Change #2 in this spec).
- All REST clients (our own admin JS, any agency integration) must update. Release notes list every breaking field.
- Contract tests updated to assert the new shape; old tests removed, not adapted.

### E. JavaScript doesn't break non-Interactivity themes

- Kanban drag-and-drop (Ideas) and inline composer (Social) use the existing `view.js` Interactivity store; no new library dependencies.
- Feature-detect the Interactivity API before attaching DnD handlers. Fall back to status dropdown when the API is absent (old WP 6.4 themes) — drag is an enhancement, not a requirement.

### F. Test coverage targets

- **PHPUnit:** new code reaches ≥ 80% line coverage; all four type dispatchers get unit tests; merge-duplicates covered by a concurrency test (two mods merging at once).
- **Integration:** `composer test:combo` green (full free + pro suite). No new flaky tests.
- **`wp jetonomy qa-actions`:** 210/210 smoke checks stay green after the migration. New checks added for status_pipeline + merge.
- **Playwright:** one browser flow per type (create → post → act-on-type-specific-feature → verify). Part of the existing usability suite; no separate run.

### G. WPCS + PHPStan + Plugin Check

All green on the release branch — enforced by the pre-commit hook we shipped in 1.3.5. PHPStan baseline must not grow.

### H. Staged rollout

- **Internal staging first** — run 1.3.7 on community.wbcomdesigns.com for one week after the RC tag before the public release.
- **Opt-in beta channel** — tag an `v1.3.7-beta.1` release; store gets a beta download link for customers who want to opt in early; bugs route to a dedicated Basecamp card.
- Only after the beta week is quiet (zero error-path Basecamp cards) does the `v1.3.7` tag get promoted.

### I. Rollback plan

- Migration `1_2_7` is reversible: the ENUM widening is non-destructive, the new column is nullable, the new table is orthogonal. A `1_2_7_down.php` script captures the exact `ALTER TABLE` statements to revert.
- If a site admin needs to revert post-update: `wp jetonomy migrator rollback --to=1.2.6` restores the schema; templates fall back to the old monolithic `space.php` (kept in the repo for one release cycle, removed in 1.4.x).
- Before tag: rollback scripted + tested on the same production snapshot used for the forward migration.

### J. Performance budget

- Page load for a forum space (12 topics) must stay within ±5% of 1.3.6 baseline. Same target for QA and Social at equivalent content density.
- Ideas space with 100 ideas + kanban view: first paint < 1.5 s on a fresh mid-range machine with a cold object cache. Queries checked via Query Monitor — no N+1 from the status pipeline column.
- Benchmarks captured before tag.

## Verification plan

For each space type, create a fresh space, post 3–5 topics, and validate:

### Forum
- [ ] Renders identically to 1.3.6 — screenshot comparison should be pixel-identical except filename.

### QA
- [ ] Filter tabs: All / Open / Resolved visible at top.
- [ ] Accept a reply → topic moves to Resolved tab.
- [ ] Resolved topic renders with muted state.

### Ideas
- [ ] Default sort: vote score descending.
- [ ] Status pipeline column visible on idea cards.
- [ ] Moderator drags an idea from Open → Planned → In-Progress → Shipped; status persists.
- [ ] Merge duplicates: idea A has 3 votes, idea B has 5. Merge A into B → B shows 8 votes, A is trashed, audit row in `jt_post_merges`.
- [ ] `jetonomy_idea_status_changed` action fires with `(post_id, old_status, new_status)`.

### Social
- [ ] Inline composer at top of feed, no redirect.
- [ ] Image upload inline, preview before post.
- [ ] Posting prepends the new card without full page reload.
- [ ] Replies inline (not a separate thread page).
- [ ] Infinite scroll fetches next page on near-bottom.

### Naming fix
- [ ] Admin dropdown shows "Social" (not "Feed").
- [ ] Existing `type='feed'` rows migrated to `type='social'`.
- [ ] `wp option get jetonomy_db_version` → `1.2.7`.

### First-glance differentiation (the design bar)

**The 2-second test.** Open four spaces side-by-side — one of each type — in four browser tabs. Cycle through with Cmd+1..4. A non-technical person must correctly identify each type within 2 seconds per tab, without reading any text.

Scoring criteria (run with 3 internal non-dev reviewers before the tag):

- [ ] Forum: identified as "topic list / threaded discussion" on first impression.
- [ ] QA: identified as "question and answer site, like Stack Overflow" from the icon + resolved badges + filter tabs alone.
- [ ] Ideas: identified as "product roadmap / feature voting" from the left-hand vote count + status pills alone.
- [ ] Social: identified as "feed / timeline / Instagram-ish" from the media-first cards + inline composer alone.

If any reviewer misidentifies a type, the visual design for that type is not differentiated enough — iterate before tag.

### Type is immutable (locked)

- [ ] Admin space-edit page renders type as read-only text (not a dropdown) with helper text "cannot be changed after creation."
- [ ] `PATCH /spaces/{id}` with `{type: 'X'}` returns `400` (or silently drops the field and logs a notice — pick one, document it).
- [ ] `wp jetonomy space update 3 --type=ideas` returns an error.
- [ ] Only `POST /spaces` accepts `type`.
