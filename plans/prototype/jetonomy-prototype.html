How to properly structure a large Django project with multiple apps?

SK
Sarah Kim 3 Trusted · 1,247 reputation · 3 hours ago
django architecture best-practices

I'm building a SaaS platform and my Django project is getting unwieldy. Currently have 15+ apps and the dependency graph is becoming a nightmare. Looking for real-world advice on how to structure this properly.

Current structure looks like:

myproject/
├── apps/
│   ├── accounts/
│   ├── billing/
│   ├── dashboard/
│   ├── notifications/
│   ├── api/
│   └── ... 10 more apps
├── core/
│   ├── mixins.py
│   └── utils.py
└── config/
    └── settings/

The main problems I'm facing:

1. Circular imports between apps (billing needs accounts, accounts needs billing for subscription checks)
2. Fat models — my User model is 800+ lines
3. Where to put shared business logic that touches multiple apps?

Has anyone dealt with this at scale? Would love to see how others organize large Django projects. Thinking about moving to a services/ layer pattern but not sure if that's the Django way.

I've dealt with exactly this at my previous company (50+ Django apps). The pattern that saved us was the Domain Services Layer:

Instead of putting business logic in models or views, create a services/ module in each app that handles cross-app operations. The key rule: models only know about their own app, services can import from other apps.

For your circular import problem — the service layer breaks the cycle. Your billing/services.py imports from accounts/models.py, and accounts/services.py imports from billing/models.py. No circular dependency because services are leaf nodes.

Another approach worth considering: break your monolith into bounded contexts, even within Django. Group related apps into packages:

myproject/
├── domains/
│   ├── identity/     # accounts, auth, profiles
│   ├── commerce/     # billing, subscriptions, invoices
│   ├── engagement/   # notifications, emails, activity
│   └── platform/     # dashboard, api, admin
└── shared/           # truly shared utilities

Each domain has its own models/, services/, and api/. Cross-domain communication goes through explicit interfaces (events or service calls), never direct model imports.

How to properly debounce input in React 19 with the new compiler?

TW
Tom Wilson 1 4 hours ago
react hooks performance

With React 19's compiler automatically memoizing everything, do we still need useMemo / useCallback for debouncing? My old pattern was:

const debouncedSearch = useCallback(
  debounce((term) => fetchResults(term), 300),
  []
);

But with the React Compiler, this feels redundant. What's the idiomatic way to debounce in React 19?

5 Answers

89
SK
Sarah Kim 3 3 hours ago ✓ Accepted Answer

Great question! With React 19's compiler, useCallback is indeed unnecessary for memoization. But debouncing is about timing control, not memoization. You still need it.

The idiomatic React 19 pattern uses useDeferredValue for most cases:

function SearchInput() {
  const [query, setQuery] = useState('');
  const deferredQuery = useDeferredValue(query);

  // This re-renders with the deferred value
  // React automatically batches and defers updates
  return (
    <>
      <input value={query} onChange={e => setQuery(e.target.value)} />
      <Suspense fallback={<Spinner />}>
        <SearchResults query={deferredQuery} />
      </Suspense>
    </>
  );
}

For actual API debouncing (network requests), use a simple ref-based approach:

function useDebounce(fn, delay) {
  const timeoutRef = useRef(null);
  return (...args) => {
    clearTimeout(timeoutRef.current);
    timeoutRef.current = setTimeout(() => fn(...args), delay);
  };
}

The compiler handles the memoization. You handle the timing. Clean separation.
