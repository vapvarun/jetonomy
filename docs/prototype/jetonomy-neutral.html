How to properly structure a large Django project with multiple apps?

SK
Sarah Kim 3 3 hours ago djangoarchitecture

I'm building a SaaS platform and my Django project is getting unwieldy. Currently have 15+ apps and the dependency graph is becoming a nightmare.

myproject/
├── apps/
│   ├── accounts/
│   ├── billing/
│   ├── dashboard/
│   └── ... 10 more
├── core/
└── config/

Main problems: circular imports between apps, fat models (800+ line User model), and unclear where to put shared business logic.

1,247 views

MR
Mike Ross 2 2h ago

The pattern that saved us was the Domain Services Layer. Instead of business logic in models, create services/ in each app. Key rule: models only know their own app, services can import across.

SK
Sarah Kim OP 1h ago

Great! Do you put services in a separate top-level directory or inside each app?

AP
Anika Patel 3 1h ago

Consider bounded contexts:

domains/
├── identity/    # accounts, auth
├── commerce/    # billing, subscriptions
├── engagement/  # notifications, emails
└── platform/    # dashboard, api

Each domain owns its models, services, and API. Cross-domain communication through explicit interfaces only.

How to properly debounce input in React 19 with the new compiler?

TW
Tom Wilson 1 4h ago react hooks

With React 19's compiler auto-memoizing, do we still need useMemo/useCallback for debouncing?

const debouncedSearch = useCallback(
  debounce((term) => fetchResults(term), 300),
  []
);

What's the idiomatic way now?

5 Answers

89
SK
Sarah Kim 3 3h ago✓ Accepted

Debouncing is about timing, not memoization. Use useDeferredValue for most cases:

function SearchInput() {
  const [query, setQuery] = useState('');
  const deferred = useDeferredValue(query);
  return <>
    <input onChange={e => setQuery(e.target.value)} />
    <SearchResults query={deferred} />
  </>;
}

For API debouncing, use a ref-based approach. The compiler handles memoization. You handle timing.
