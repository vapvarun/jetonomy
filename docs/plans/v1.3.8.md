# v1.3.8 Plan: FluentCommunity Integration

**Status**: planning
**Branch**: `1.3.8` (both `jetonomy` and `jetonomy-pro`)
**Shipped in**: `jetonomy` (free). No `jetonomy-pro` changes required for v1.

## Goal

Make Jetonomy and FluentCommunity (FC) coexist on a single site without the customer noticing friction. Purely navigational and identity-level integration. No state sync, no destructive behavior on either side.

## What ships

All implemented in one file: `jetonomy/includes/integrations/class-fluent-community.php` (same pattern as the existing `class-buddypress.php`).

| # | Feature | Hook (verified present in FC 2.3.0 / Jetonomy 1.3.7) |
|---|---|---|
| 1 | FC avatar used everywhere (including Jetonomy profiles) | WP core `get_avatar_url` filter reads `wp_fcom_xprofile.avatar` by `user_id` |
| 2 | Admin pairs one FC space with one Jetonomy space | New Jetonomy Settings > Integrations tab via `jetonomy_admin_settings_tabs` + `jetonomy_admin_settings_tab_content` |
| 3 | Configurable tab label (default "Discussions") | Option `jetonomy_fc_tab_label` |
| 4 | FC space header shows tab linking to paired Jetonomy space | `fluent_community/space_header_links` filter (FC `app/Models/BaseSpace.php:589`) |
| 5 | Jetonomy space header shows tab linking to paired FC space | `jetonomy_header_nav_items` action |
| 6 | FC profile page shows "Discussions" section: topics started + topics followed + View all link | `fluent_community/activity/after_contents_user` action |
| 7 | Auto-enable when FC detected | `class_exists('FluentCommunity\\App\\App')` |

## Storage

One WordPress option: `jetonomy_fc_space_pairs`

Format:
```
{
  "<fc_space_id>": <jetonomy_space_id>,
  ...
}
```

Not written into either plugin's `settings` JSON column. Rationale:
- Two copies drift; one option is the single source of truth.
- Audit-friendly, backup-friendly, trivially clearable on uninstall.
- Zero DB migrations. One option row is the entire data footprint.

Second option: `jetonomy_fc_tab_label` (string, default `Discussions`).

## Identity keying rule

**Always join by `user_id` (primary key in both `wp_fcom_xprofile` and `wp_jt_user_profiles`). Never by username.**

Reason: the running site already has a collision: WP user `admin` (ID 1) corresponds to FC `xprofile.username = admin2`. Username is not a reliable key across the two systems.

## Stability guarantees

- **No core changes** to FC or Jetonomy. Everything in one integration file.
- **No writes to FC tables.** Read-only from FC. The integration can be removed and FC is untouched.
- **Stale pair handling.** At render time, if either side's space ID does not resolve to an existing space, the tab silently disappears. No admin cleanup required. No fatals.
- **Graceful degradation.** If any of the five FC hooks are renamed or removed in a future FC release, only the corresponding feature stops rendering. Plugin stays loaded.
- **One option row is the entire data footprint.** Uninstall deletes one row.

## Explicitly out of scope for v1

Each has a concrete reason, not just "later":

| Item | Reason |
|---|---|
| Membership sync (auto-join paired space on join) | Destructive on leave; retroactive paired-space problem when admin pairs two spaces that already have different members. Belongs in a Pro extension with explicit per-pair toggles and a backfill tool. |
| Privacy mirroring (paired spaces share visibility) | Can accidentally expose or hide content during admin edits. Same "explicit Pro toggle" path. |
| Profile field sync (bio, display name, two-way write) | Jetonomy has no `jetonomy_profile_updated` action today. Adding one is a core change, out of v1 scope. |
| Notification inbox merge | Jetonomy has no extension point for external notification sources. Would need a new adapter interface. |
| Points to reputation mapping | Customer deprioritized (see conversation 2026-04-22). |
| Unified search across FC + Jetonomy | No single FC search hook; each endpoint (`members_api_response`, `comments_api_response`, etc.) filters separately. Invasive to cover all surfaces. |
| Shared moderation queue | Different content models; cross-model flag normalization is a product question, not a hook question. |
| "Promote FC thread to Q and A" moderator action | No FC action for "convert this feed post to external." Custom button + API needed. |

## Queries used (reference)

Topics started (for Discussions section on FC profile):
```sql
SELECT id, title, slug, category_id
FROM wp_jt_posts
WHERE author_id = %d
  AND status = 'active'
ORDER BY created_at DESC
LIMIT 5
```

Topics followed:
```sql
SELECT p.id, p.title, p.slug, p.category_id
FROM wp_jt_subscriptions s
INNER JOIN wp_jt_posts p ON s.object_id = p.id
WHERE s.user_id = %d
  AND s.object_type = 'post'
ORDER BY s.created_at DESC
LIMIT 5
```

Both wrapped in `UserProfile` / `Post` / `Subscription` model methods, no raw SQL in the integration file. Follows the Jetonomy coding pattern (all data access via models).

## Admin UX

Jetonomy > Settings > **Integrations** tab (new):

- Section heading: FluentCommunity
- Status line: "FluentCommunity 2.3.0 detected. Integration active."
- Field: Tab label (text input, default "Discussions")
- Table: Space pairings
  - Columns: FC Space | Jetonomy Space (dropdown) | Actions (Remove)
  - "Add pair" button opens a row with two dropdowns
- Save button

Admin screen is self-contained in the integration class; no new controller needed.

## Uninstall

The integration adds nothing to `uninstall.php` directly. The two options (`jetonomy_fc_space_pairs` and `jetonomy_fc_tab_label`) follow the existing Jetonomy uninstall sweep (deletes all options starting with `jetonomy_`).

## REST architecture note (verified 2026-04-22)

FluentCommunity 2.3.0 exposes 154 routes under the `fluent-community/v2` namespace. The SPA consumes these; there is no server-rendered profile or space page to hook. **The PHP filters we rely on all execute during REST response preparation, and their output flows through to the Vue SPA render automatically.**

Verified against live endpoints:
- `GET /fluent-community/v2/spaces/{slug}/by-slug` returns `header_links: [...]` populated by `fluent_community/space_header_links` filter. Our tab append lands here.
- `GET /fluent-community/v2/profile/{username}` returns `profile_navs: [...]`. Mutating via `fluent_community/profile_view_data` filter adds tabs.
- `GET /fluent-community/v2/activities?...` calls `ActivityController@get` which runs `fluent_community/activity/after_contents_user` (filter takes string, appends to user activity view).

Implication: no Jetonomy-side REST additions or JS injection needed for v1. The integration remains one PHP file.

## Risk log

- **External-URL tab rendering.** All existing FC tabs carry a `route` object for SPA internal navigation. Our tabs use `url` only (link to Jetonomy). Whether FC's Vue tab component renders an external-URL entry without a `route` key is unverified. Browser test during build. Fallback hooks if FC ignores it: `fluent_community/portal_header` (server-rendered, always visible) or register a route via `fluent_community/app_route_paths` (SPA-aware).
- **Username collision in existing site.** Handled by keying on `user_id` throughout. Documented above.
- **88 Jetonomy spaces vs 3 FC spaces on the current dev site.** Most Jetonomy spaces will have no paired FC space. Expected and handled: only paired spaces render the tab.
