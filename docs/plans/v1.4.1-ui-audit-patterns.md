## C1 — Icon source (Lucide-only frontend, Dashicons-allowed admin)

### Frontend rule

Every icon on a frontend surface (templates/views, partials, blocks, shortcodes, emails) comes from `assets/icons/*.svg` via the helpers in `jetonomy.php`:

```php
jetonomy_echo_icon( 'icon-name', $size_in_px );
jetonomy_icon( 'icon-name', $size_in_px );           // returns string
jetonomy_render_space_icon( $space->icon, $size );   // emoji-defended space-icon path
```

**Forbidden in frontend surfaces:**
- Inline `<svg>` markup written into the template
- Unicode emoji in plugin-rendered strings (user-generated content fields exempt)
- Literal symbols used as icons: `←`, `→`, `↑`, `↓`, `▲`, `▼`, `✓`, `✗`, `★`, `⋮`, `″`, `‟`
- Dashicons on frontend surfaces

### Admin rule

Admin views (`includes/admin/views/*.php`) **may keep using Dashicons**. Reason: admin already loads Dashicons for free, the convention is consistent with WP itself, and frontend-REST-only / backend-AJAX-OK already accepts admin-side WP-native conventions.

**Exception in admin:** `includes/admin/views/setup-wizard.php` contains an inline `<svg>`. Step 1 inventory flagged this as the only handdrawn-svg violation. Sweep target: if the SVG is brand chrome (logo, hero illustration), keep it inline as a content asset, not an icon. If it's an icon-as-icon, swap to `jetonomy_echo_icon`.

### Email rule

Email templates (`templates/emails/*.php`) ship with **no inline SVG icons**. Email clients vary wildly on SVG support (Gmail strips them, Outlook ignores them, Apple Mail renders fine). Use:
- Brand image as `<img src="…" />` with absolute URL (already done in `base.php`)
- Text-only chrome for everything else
- No icon-as-text, no emoji

### Allowed icon palette

The 44 Lucide SVGs already in `assets/icons/` cover every UI need identified in the inventory. New icons added to a sweep commit must come from `lucide.dev`, drop into `assets/icons/`, and use `currentColor` for stroke (no inline color overrides).

**Pre-approved set used in this sweep:**
- Navigation: `home`, `search`, `bell`, `user`, `users`, `lock`, `eye`, `chevron-up`, `chevron-down`, `chevron-left` (NEW), `chevron-right` (NEW), `arrow-right`
- Engagement: `heart`, `smile-plus`, `bookmark`, `star`, `share`, `message-circle`, `quote`
- Content: `hash`, `pin`, `flag`, `edit`, `trash`, `merge`, `split`, `move`
- Hierarchy / metadata: `shield`, `award`, `more-horizontal`, `check-circle`
- States: `empty-posts`, `empty-replies`, `empty-search`, `empty-members`, `empty-notifications`, `help-circle`, `lightbulb`
- Misc: `book-open`, `image`, `link`, `folder`, `rocket`, `megaphone`, `hand`, `settings`

**Two NEW icons** must be added to `assets/icons/`:
- `chevron-left.svg` (for `tag.php` Previous pagination)
- `chevron-right.svg` (for `tag.php` Next pagination, plus future use)

These come from lucide.dev unmodified.

## C2 — Action density (visible, spacious, 4-or-fewer per surface)

The user's principle: "all options must be clean spacious easy to use by anyone… we are making things for all age groups, if things will be too crowded hidden in layer nobody uses them." The sweep removes redundant controls, it does **not** push them behind a `⋮` menu when they're commonly needed.

### Reply card (highest priority — appears N× on every topic page)

Today: 7 visible controls — `^` `v` Reply Quote Flag `⋮` React.

**Free target shape (unified vote cluster — both up AND down, equal visual weight):**

```
[ ▲ N ▼ ]                                    [💬 Reply]   [🚩 Report]   [⋮]
 vote cluster (up + score + down)             primary       de-emphasized   admin/owner only
```

**Pro target shape (when Reactions extension is active — Pro swaps the rendered partial transparently via existing filter):**

```
[😊 N · +]                                   [💬 Reply]   [🚩 Report]   [⋮]
 reactions cluster (positive + negative
  options, counts + add picker)               primary       de-emphasized   admin/owner only
```

End-user goal: users feel **free** to react in either direction. Both upvote and downvote, both positive and negative reactions — no friction asymmetry, no nudging toward one side, no confirmation modals on negative actions.

Decision #1 (signed off): free keeps Vote, Pro swaps in Reactions when active. Both modes render the *same* visible-control count (4 affordances) and the same spacing scale; only the leftmost cluster differs.

| Control | Visible (free)? | Visible (pro+reactions)? | Lucide icon | Notes |
|---|---|---|---|---|
| Vote cluster — up arrow | Yes | No (replaced) | `chevron-up` | Tap toggles upvote. Touch target ≥44×44px. |
| Vote cluster — score | Yes | No (replaced) | (text) | Display only, between the two arrows. |
| Vote cluster — down arrow | Yes | No (replaced) | `chevron-down` | **Equal visual weight to the up arrow.** Tap toggles downvote. Same tap target size. No "are you sure?" modal. |
| Reactions cluster (positive + negative + add) | No | Yes | `smile-plus` | Pro-only. Free renders Vote cluster in this slot. Reactions picker MUST include negative options (👎, 😕, 😠 — not just positive ones). |
| Reply | Yes | Yes | `message-circle` | Primary intent. Quote moves into the composer toolbar (where Quote already lives as a contextual action when text is selected). |
| Report | Yes | Yes | `flag` | Visible for moderation discoverability. De-emphasized: ghost-button styling, not chip. **Distinct from downvote** — Report is for rule violations, downvote is for "not useful." |
| `⋮` More | Yes | Yes | `more-horizontal` | Carries Edit (own reply) / Delete (own or moderator) / Pin (admin) / Permalink. Genuine admin/owner concerns only. |

Removed entirely (both modes):
- Standalone Quote button → moves into composer toolbar
- Standalone React pill (Pro-only today, separate from inline reactions) → folded into the Reactions cluster

### Vote cluster spec (CSS / markup)

```html
<div class="jt-vote-cluster" data-jt-vote="post:123">
    <button class="jt-vote-up" type="button" aria-label="Upvote" data-wp-on--click="actions.voteUp">
        <?php jetonomy_echo_icon( 'chevron-up', 18 ); ?>
    </button>
    <span class="jt-vote-score" data-wp-text="state.voteScore">42</span>
    <button class="jt-vote-down" type="button" aria-label="Downvote" data-wp-on--click="actions.voteDown">
        <?php jetonomy_echo_icon( 'chevron-down', 18 ); ?>
    </button>
</div>
```

```css
.jt-vote-cluster {
    display: inline-flex;
    align-items: center;
    gap: var(--jt-space-sm);  /* 8px */
}
.jt-vote-up,
.jt-vote-down {
    inline-size: 44px;
    block-size: 44px;
    display: inline-flex;
    align-items: center;
    justify-content: center;
    background: transparent;
    border: 1px solid transparent;
    border-radius: var(--jt-radius-sm);
    color: var(--jt-text-secondary);
    cursor: pointer;
    transition: background-color 0.1s, color 0.1s;
}
.jt-vote-up:hover,
.jt-vote-up[aria-pressed="true"] {
    color: var(--jt-success);
    background: var(--jt-success-light);
}
.jt-vote-down:hover,
.jt-vote-down[aria-pressed="true"] {
    color: var(--jt-warn);
    background: var(--jt-warn-light);
}
.jt-vote-score {
    font-variant-numeric: tabular-nums;
    font-weight: 600;
    color: var(--jt-text);
    min-inline-size: 1.5em;
    text-align: center;
}
@media (hover: hover) {
    .jt-vote-up, .jt-vote-down { inline-size: 36px; block-size: 36px; }
}
```

Both arrows have **identical CSS weight** — same size, same opacity, same hover treatment intensity. Only the active-state color differs (success vs warn) so the user knows which they pressed.

Touch targets: every visible control ≥ 44 × 44px. Action-row gap: 16px desktop, 12px mobile.

### Post card (forum listing, search results, profile activity)

Today: card with avatar, title, excerpt, no inline action bar.
**Target:** unchanged. Post cards are scan surfaces — clicking the card is the primary action. Adding inline controls invites mistaps. Keep it pure.

### Member row (space-members, leaderboard)

| Control | Visible? | Lucide icon |
|---|---|---|
| Message | Yes | `message-circle` |
| Follow | Yes | `user` (ghost-button) |
| `⋮` | Yes | Block / Mute / Report (admin / mod only items) |

### Notification row

| Control | Visible? |
|---|---|
| Mark read | Yes (single-tap to mark) |
| Open | Yes (whole row tappable) |
| `⋮` Mute / Mark all read | Yes (per-row only when relevant) |

### Composer

Toolbar: Bold, Italic, Code, Link, Quote, Image. Already correct. **Remove "Ctrl+Enter to submit" hint on touch devices** — replace with platform-aware copy:

```php
$is_touch_hint = wp_is_mobile()
    ? __( 'Markdown supported.', 'jetonomy' )
    : __( 'Markdown · Ctrl+Enter to submit.', 'jetonomy' );
```

### Pagination

Today: literal `←` `→` text in `tag.php` (and any other paginated surface).
**Target:** Lucide `chevron-left` / `chevron-right` icon + text label.

```php
<a class="jt-pager-prev" href="…">
    <?php jetonomy_echo_icon( 'chevron-left', 14 ); ?>
    <?php esc_html_e( 'Previous', 'jetonomy' ); ?>
</a>
<a class="jt-pager-next" href="…">
    <?php esc_html_e( 'Next', 'jetonomy' ); ?>
    <?php jetonomy_echo_icon( 'chevron-right', 14 ); ?>
</a>
```

CSS uses logical properties so RTL flips automatically.

## C3 — Byline structure

One canonical shape for every "user · role · time" surface (post card, reply card, member row, notification row, leaderboard cell).

### Canonical pattern

```
[Avatar] Username · ROLE_BADGE · OP_BADGE · 3 weeks ago
```

- **Avatar:** 32px round on mobile, 36px desktop. Single image, no border ring.
- **Username:** plain text, `jt-byline-name` class. Links to user profile.
- **ROLE_BADGE:** displayed only when `role` is one of `ADMIN` / `MOD`. Short text inside a chip with role-color background:
  - ADMIN → `--jt-accent` background
  - MOD → `--jt-warn-light` background
- **OP_BADGE:** displayed only when the byline is on a reply AND the author is the topic creator. Short text "OP" inside a neutral chip.
- **Timestamp:** relative human time. Tooltip via `title=` attribute carries the absolute timestamp.

### Removed from byline

| Element | Reason |
|---|---|
| Trust-level number (`2`, `3`, `TL2`, etc.) | "TL2" means nothing to a first-time reader. Trust progress lives on the profile page + hover card. |
| Tags / topic tags | Tags belong below the post body, not on the author byline. |
| Reaction icons on the byline | Conflict with the action-bar reactions cluster. One source of truth. |

### Tag display moves

Today's `home.php` has tags inline near the author. Target: tags render below the post body in their own row, styled as text-links (not chips), so they read as topic descriptors and not as buttons.

```html
<div class="jt-post-body">{post content}</div>
<div class="jt-post-tags">
    <a href="…/tag/wordpress/" class="jt-tag-link">#wordpress</a>
    <a href="…/tag/plugin/" class="jt-tag-link">#plugin</a>
</div>
```

CSS: `.jt-tag-link { color: var(--jt-text-tertiary); text-decoration: none; font-size: 0.875rem; } .jt-tag-link:hover { color: var(--jt-accent); text-decoration: underline; }`. No chip background.

## C4 — Spacing scale + touch-target compliance

### Spacing tokens (added to `:root, .jt-app` if missing)

```css
--jt-space-xs: 4px;
--jt-space-sm: 8px;
--jt-space-md: 12px;
--jt-space-lg: 16px;
--jt-space-xl: 20px;
--jt-space-2xl: 32px;
```

Use these via `var(--jt-space-*)`; never hardcode pixel values.

### Card pattern

```css
.jt-card {
    padding-block: var(--jt-space-xl);   /* 20 */
    padding-inline: var(--jt-space-xl);  /* 20 */
    border-radius: var(--jt-radius-md);
    border: 1px solid var(--jt-border);
    background: var(--jt-bg);
}
@media (max-width: 640px) {
    .jt-card {
        padding-block: var(--jt-space-lg);   /* 16 */
        padding-inline: var(--jt-space-lg);  /* 16 */
    }
}
```

### Action row

```css
.jt-action-row {
    display: flex;
    align-items: center;
    gap: var(--jt-space-lg);   /* 16 desktop */
    flex-wrap: wrap;
}
@media (max-width: 640px) {
    .jt-action-row { gap: var(--jt-space-md); /* 12 */ }
}
```

### Touch-target minimum

Every interactive surface (`<a>`, `<button>`, icon-button, chevron, vote arrow, react pill) targets ≥ 44 × 44px on touch. Where the visible glyph is smaller, padding fills the rest:

```css
.jt-icon-btn {
    inline-size: 44px;
    block-size: 44px;
    display: inline-flex;
    align-items: center;
    justify-content: center;
    /* glyph rendered via jetonomy_icon at 16-20px inside */
}
@media (hover: hover) {
    /* on desktop with mouse, ok to shrink to 36px since precision is finer */
    .jt-icon-btn { inline-size: 36px; block-size: 36px; }
}
```

### Section rhythm

Sections inside a view (post body → reply count header → reply list → composer) separated by `var(--jt-space-2xl)` (32px) margin-block.

## C5 — Empty / error / 403 states

One shared partial: `templates/partials/empty-state.php`. Every empty / error / 403 surface in the inventory routes through it.

### Partial signature

```php
\Jetonomy\Template_Loader::partial( 'empty-state', [
    'icon'       => 'empty-posts',     // Lucide name from assets/icons/
    'message'    => __( '…', 'jetonomy' ),
    'cta_label'  => __( '…', 'jetonomy' ),    // optional
    'cta_url'    => $base . '/new/',          // optional
    'tone'       => 'info',                   // info | warn | forbidden
] );
```

### Markup the partial outputs

```html
<div class="jt-empty jt-empty-tone-{$tone}">
    <div class="jt-empty-icon">{lucide @ 48px}</div>
    <p class="jt-empty-text">{message}</p>
    <a class="jt-btn jt-btn-primary jt-empty-cta" href="{cta_url}">{cta_label}</a>
</div>
```

CTA renders only when both `cta_label` and `cta_url` are non-empty.

### Tones

| Tone | When to use | Visual |
|---|---|---|
| `info` (default) | "No items yet" | Neutral icon, neutral text |
| `warn` | "Nothing matches your filter" | Same icon, secondary text |
| `forbidden` | 403 / not allowed to view | `lock` icon, secondary text, no CTA |

## Sweep commit sequence (Step 4)

Per the audit plan, 5 sweep commits. Each is self-contained, every surface in the inventory touched in one commit per concern.

| # | Commit | Files (rough) | Risk |
|---|---|---|---|
| 1 | `feat(ui): empty-state partial + sweep all empty paths through it` (C5) | new `partials/empty-state.php` + ~15 view edits | low |
| 2 | `chore(ui): canonical byline structure across all card / row surfaces` (C3) | reply-card.php, post-card.php, member-row partial, notification surfaces, leaderboard | medium |
| 3 | `chore(ui): action-density cleanup — reactions cluster + Reply / Report / ⋮` (C2) | reply-card.php (primary), composer hint, tag.php pagination | medium-high |
| 4 | `chore(ui): icon source uniformity — chevron-left/right added, raw glyphs removed, setup-wizard inline-SVG audit` (C1) | tag.php, setup-wizard.php (or note as exempt), 2 new icon files | low |
| 5 | `chore(ui): spacing scale + touch-target compliance` (C4) | jetonomy.css spacing tokens, all card / action-row / icon-btn classes | low |

Order rationale: C5 first (low-risk, builds infrastructure used by C1–C4 surfaces). C3 second (byline change is visible everywhere — get it landed before action-bar work). C2 third (highest visible change, lowest risk after C3 lands). C1 fourth (cleans up final glyph leftovers). C4 last (fine-tuning across everything else).

Each commit lands free + Pro changes when both are in scope. Each commit's verification runs at 390px and 1280px, both light and dark, plus RTL spot-check.
