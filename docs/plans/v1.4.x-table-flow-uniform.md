# Jetonomy 1.4.x — Uniform Table Creation + Upgrade Flow

**Status:** PLANNING — no code changes until this plan is signed off.
**Scope:** Free + Pro + every Pro extension. One canonical pattern across the entire suite.
**Driver question (verified, 2026-05-04):** "does all table we have are right in activation flow with proper upgrade model" → "plan properly and make sure we have a uniform flow as per any standard plugin when table will be created and how it will be upgraded".

---

## 1. Goal

Every plugin table — free or Pro, base or extension — is created on activation, self-heals on upgrade (no missed `ALTER`s, no missed `CREATE`s for tables added in later versions), drops cleanly on uninstall, and follows **one identical pattern** so a reviewer reading any extension can predict the next without reading source.

A user upgrading from any prior version, by any path (manual replace, WP auto-update, deactivate-reactivate, fresh install, multisite network-activate), ends up with the exact current schema. No orphan tables, no missing columns, no silent half-installs.

### Diagnosis summary

The fundamental anti-pattern in both plugins:

> **The upgrade trigger does not re-run the schema installer.**

dbDelta is idempotent and self-healing — re-running it on every plugin upgrade is exactly what mature WP plugins (WooCommerce, EDD, Yoast, Gravity Forms) do, and it eliminates the need for per-version "create table X" migrations entirely. We're paying the cost of writing migrations for schema additions that dbDelta would handle for free.

## 3. Standard plugin pattern (reference)

Cross-checked against **WooCommerce** (`includes/class-wc-install.php`), **Easy Digital Downloads** (`includes/install.php`), **Yoast SEO** (`src/installer/installation.php`), and **The Events Calendar**:

```
┌─────────────────────────────────────────────────────────┐
│  PLUGIN_DB_VERSION  ←  single constant, bumped any      │
│                        time SCHEMA changes              │
└─────────────────────────────────────────────────────────┘
              │
              ▼
┌─────────────────────────────────────────────────────────┐
│  Install::install()                                     │
│  ─ idempotent                                           │
│  ─ wraps dbDelta() for every current table              │
│  ─ seeds default options if empty                       │
│  ─ safe to call any time, any number of times           │
└─────────────────────────────────────────────────────────┘
       ▲                                          ▲
       │                                          │
┌──────┴──────────────┐              ┌────────────┴──────────────┐
│ activation hook     │              │ plugins_loaded check      │
│ (fresh activate)    │              │ if version_compare(       │
│   1. Install()      │              │   stored, current, '<' ): │
│   2. set version    │              │   1. Install()  ← yes,    │
│   3. set caps       │              │      again, idempotent    │
│   4. schedule cron  │              │   2. Run data migrations  │
│   5. flush rewrites │              │      from registry        │
│                     │              │   3. set version          │
└─────────────────────┘              └───────────────────────────┘
```

**Why it works:**

1. `dbDelta()` is idempotent: re-running it adds missing tables, adds missing columns, changes column types. **It is the only schema-change tool you need for additive changes.**
2. The migration registry is reserved for DATA transforms — moving values from option A to option B, splitting a column, normalising stored data. Not for schema.
3. The auto-update path (which doesn't fire `register_activation_hook`) is covered by the `plugins_loaded` upgrade trigger.
4. Uninstall reads from the same canonical Schema source — never duplicated.

## 4. Target architecture

### Free plugin

```
includes/db/class-schema.php           ← single source of truth
                                          - get_table_names()       [unchanged]
                                          - get_table_definitions() [unchanged]
                                          - install()               [renamed from create_tables() for clarity]
                                          - drop_tables()           [unchanged]

includes/db/class-migrator.php         ← DATA-ONLY migrations from now on
                                          - get_migrations() registry contains ONLY data
                                            transforms (not table creates / column adds)
                                          - schema additions handled by Schema::install()
                                            re-running on every upgrade

includes/class-jetonomy.php
  ├── activate()                       ← unchanged: Schema::install() + caps + cron + rewrites + version set
  └── maybe_upgrade()  [new]           ← runs on plugins_loaded:5 (before everything else)
                                          if version_compare( stored, JETONOMY_DB_VERSION, '<' ):
                                            1. Schema::install()       ← self-heal schema
                                            2. Migrator::run( stored )  ← run any data migrations
                                            3. update_option( db_version )

uninstall.php                          ← reads canonical drop list
                                          (allowed to require/include Schema since it has no
                                           load-time side effects — read-only static helper)

JETONOMY_DB_VERSION                    ← bumped whenever Schema::get_table_definitions changes
                                          OR a new data migration is added
```

### Pro plugin

```
JETONOMY_PRO_DB_VERSION                ← Pro-wide schema version (separate from free)

includes/class-pro-install.php  [new]  ← Pro-wide install orchestrator
  └── install()                          - foreach enabled extension: call $ext->install_schema()
                                          (renamed from create_tables for uniformity)

includes/class-extension.php           ← abstract base
  └── install_schema(): void  [new]    ← every extension implements; idempotent dbDelta wrapper

Each extension:
  includes/extensions/{slug}/class-extension.php
    public function install_schema(): void {
        // dbDelta() for tables this extension owns
    }
    public function activate(): void {
        $this->install_schema();   // delegate
        // ... extension-specific activation work (default seeds, VAPID keys, etc.)
    }

includes/class-jetonomy-pro.php
  ├── activate()                       ← Pro::install() + sets jetonomy_pro_db_version
  └── maybe_upgrade()  [new]           ← runs on plugins_loaded:25 (after free's upgrade)
                                          if version_compare( stored, JETONOMY_PRO_DB_VERSION, '<' ):
                                            1. Pro_Install::install()  ← walks all enabled extensions
                                            2. Pro_Migrator::run()      ← Pro-wide data migrations
                                            3. update_option

uninstall.php (Pro)                    ← drops every Pro table (canonical list pulled from
                                          each extension's install_schema metadata, OR from a
                                          static Pro_Schema::get_table_names() registry)
```

### Cross-cutting

| Concern | How it's handled |
|---|---|
| Multisite | Both Free `activate()` and `maybe_upgrade()` check `is_multisite()`; on network-activate iterate `get_sites()` and run install per blog. New-blog hook (`wp_initialize_site`) also runs install. |
| Order of upgrade | Free upgrades on `plugins_loaded:5` (before Pro). Pro upgrades on `plugins_loaded:25` (after Pro is loaded at 20). Free schema is always at-or-newer than Pro's expectations of free. |
| Failure semantics | `update_option( db_version )` only runs after install + migrations succeed. If migrations fail mid-way, the stored version stays at the previous successful value, so next page load retries. |
| Logging | On any install/migration failure, log to `error_log` with `phpcs:ignore` annotation. No silent failures. |

## 5. Free side — concrete migration steps

Each step is a separate commit on the `1.4.1` branch.

### Step F1 — Rename `Schema::create_tables()` → `Schema::install()` (+ deprecation alias)

Pure rename for clarity. Keep `create_tables()` as a thin wrapper calling `install()` with `_deprecated_function()` so nothing external breaks if someone calls it.

**Files:** `includes/db/class-schema.php`, `includes/class-jetonomy.php` (activate caller).

### Step F2 — Add `Jetonomy::maybe_upgrade()` and hook on `plugins_loaded:5`

```php
public function maybe_upgrade(): void {
    $stored = get_option( 'jetonomy_db_version', '0' );
    if ( ! version_compare( $stored, JETONOMY_DB_VERSION, '<' ) ) {
        return;
    }
    DB\Schema::install();              // self-heal
    DB\Migrator::run( $stored );       // data migrations only
    update_option( 'jetonomy_db_version', JETONOMY_DB_VERSION );
}
```

Move the existing `if ( version_compare … )` block from wherever it lives now (`class-jetonomy.php:319`) into this dedicated method. Hook it from the constructor on `plugins_loaded:5`.

### Step F3 — Convert existing migrations to data-only

Audit each `class-migration_*.php`:
- **1_1_0** (CREATE jt_bookmarks): retire — Schema already creates bookmarks; dbDelta covers upgraders.
- **1_2_0** (ALTER posts ADD published_at): retire — Schema already has the column; dbDelta covers it.
- **1_2_1** (ALTER posts ADD is_private): retire — dbDelta covers it. **Verify** the ADD KEY index is also in the current Schema's `CREATE` (dbDelta updates indexes too).
- **1_2_2** (ALTER posts ADD prefix): retire — dbDelta.
- **1_2_3** (ALTER notifications object_type ENUM): **keep** — column type changes via ENUM modification. dbDelta sometimes handles ENUM widening, sometimes not. Safer to keep as a one-shot.
- **1_2_4** (settings update_option): **keep** — pure data, not schema.
- **1_2_5, 1_2_6**: read each file; classify case-by-case.
- **1_4_1** (ALTER user_profiles ADD verification_reminder_sent_at): retire — dbDelta covers it once Schema's `CREATE TABLE jt_user_profiles` includes the column.

After retire: delete the file AND remove the entry from `Migrator::get_migrations()`. Leave `1_0_0` if its `up()` does anything besides schema; otherwise remove file + don't register.

**Outcome:** registry contains only true data transforms (likely just 1_2_3 and 1_2_4). Schema becomes the only place to look for "what tables/columns exist".

### Step F4 — Verify Schema is current

For every column / table added in 1.3.x and 1.4.0 commits (per `git log includes/db/`), confirm it's in `Schema::get_table_definitions()`. If anything is missing, add it. Bump `JETONOMY_DB_VERSION` if needed.

### Step F5 — Fix `uninstall.php` to use canonical Schema drop list

`uninstall.php` is intentionally standalone (doesn't `require` plugin files for safety after deletion). But `Schema` has zero load-time side effects (no hooks, no DB calls in the class body — only static methods). Safe to require it once for the drop list:

```php
require_once __DIR__ . '/includes/db/class-schema.php';
\Jetonomy\DB\Schema::drop_tables();
```

This guarantees the drop list never goes stale. Eliminates the duplicate `$tables[]` array. Leak of `wp_jt_bookmarks` is fixed permanently.

### Step F6 — Multisite hardening

Wrap `activate()` and `maybe_upgrade()` schema work in:

```php
if ( is_multisite() && function_exists( 'get_sites' ) ) {
    foreach ( get_sites( [ 'fields' => 'ids' ] ) as $blog_id ) {
        switch_to_blog( $blog_id );
        DB\Schema::install();
        restore_current_blog();
    }
} else {
    DB\Schema::install();
}
```

Plus hook `wp_initialize_site` so a new sub-site gets the schema:

```php
add_action( 'wp_initialize_site', function ( $new_site ) {
    switch_to_blog( $new_site->blog_id );
    DB\Schema::install();
    restore_current_blog();
} );
```

## 6. Pro side — concrete migration steps

### Step P1 — Add `Extension::install_schema()` to the abstract base

In `includes/class-extension.php`:

```php
abstract class Extension {
    /** Idempotent dbDelta wrapper. Default is no-op (extensions without
     *  tables don't need to override). */
    public function install_schema(): void {}

    public function activate(): void {
        $this->install_schema();
        // subclass adds extra activation work after super-call
    }
}
```

### Step P2 — Move every extension's `create_tables()` → `install_schema()`

For each: advanced-moderation, private-messaging, reactions, polls, web-push, custom-badges, ai, custom-fields, webhooks.

Rename `create_tables()` to `install_schema()`; rename `create_table()` (singular) to `install_schema()` too. Update each extension's `activate()` to call `parent::activate()` (which invokes `install_schema()`) plus any extension-specific work (badge seeding, VAPID key generation).

### Step P3 — Add `Pro_Install::install()` and `Jetonomy_Pro::maybe_upgrade()`

```php
class Pro_Install {
    public static function install(): void {
        $pro = Jetonomy_Pro::instance();
        foreach ( $pro->get_extensions() as $ext ) {
            $ext->install_schema();   // idempotent for every extension
        }
    }
}

// In Jetonomy_Pro:
public function maybe_upgrade(): void {
    if ( ! defined( 'JETONOMY_PRO_DB_VERSION' ) ) { return; }
    $stored = get_option( 'jetonomy_pro_db_version', '0' );
    if ( ! version_compare( $stored, JETONOMY_PRO_DB_VERSION, '<' ) ) {
        return;
    }
    Pro_Install::install();
    Pro_Migrator::run( $stored );      // future data migrations
    update_option( 'jetonomy_pro_db_version', JETONOMY_PRO_DB_VERSION );
}
```

Hook `maybe_upgrade()` on `plugins_loaded:25`.

### Step P4 — Define `JETONOMY_PRO_DB_VERSION`

Distinct from `JETONOMY_PRO_VERSION`. Bumped whenever ANY extension's schema changes. Currently `1.4.1` (matches `JETONOMY_PRO_VERSION` for now; will diverge as extensions add columns without a version bump on the user-facing release number).

### Step P5 — Pro uninstall: canonical drop

Add `Pro_Schema::get_table_names()` aggregating every extension's table names (each extension declares its tables via a `get_table_names()` method or via the existing `meta()` map). Pro `uninstall.php` requires `Pro_Schema` and drops via canonical list.

### Step P6 — Multisite parity

Same `is_multisite()` + `get_sites()` + `wp_initialize_site` pattern as F6.

## 7. Acceptance criteria

The plan is "done" when ALL of these are true:

- [ ] `JETONOMY_DB_VERSION` and `JETONOMY_PRO_DB_VERSION` are documented as "bump whenever schema changes" in `CLAUDE.md` for both repos.
- [ ] Free `Schema::install()` + Pro `Pro_Install::install()` both wrap `dbDelta()` and are safe to call any number of times.
- [ ] Both plugins have a `maybe_upgrade()` method hooked on `plugins_loaded:5` / `plugins_loaded:25` that calls `install()` then `Migrator::run()`.
- [ ] Migration registries contain ONLY data transforms — never table creates or column adds.
- [ ] Every extension's `activate()` delegates to `install_schema()` via the abstract base.
- [ ] `uninstall.php` (both plugins) reads the canonical drop list from Schema, never a duplicated array.
- [ ] Multisite: activating on a network creates tables on every existing blog, AND `wp_initialize_site` hook handles new blogs.
- [ ] On a fresh install at 1.4.x: every table from `Schema` exists in the database, no orphan migrations run.
- [ ] On an upgrade from 1.0.0 to current: every table including `jt_bookmarks` exists, every column exists, no errors in `error_log`.
- [ ] On uninstall at 1.4.x: zero `wp_jt_*` tables remain, zero `wp_jt_pro_*` tables remain, zero plugin options remain.

Verification harness (manual or scripted):

```
1. Fresh activate free → SHOW TABLES LIKE 'wp_jt_%'  → expect 23 rows
2. Fresh activate free + pro → SHOW TABLES LIKE 'wp_jt_pro_%' → expect N rows (count from Pro registry)
3. Simulate 1.0 install (manual delete of post-1.0 tables) → bump plugin → reload → tables back
4. Manually delete one column added in 1.4.1 → reload → column back
5. Uninstall via wp-admin → assert zero matching rows
```

## 8. Out of scope (deliberately)

- **Renaming any existing table.** dbDelta cannot rename tables; that requires a one-shot data migration. Not needed today.
- **Foreign-key constraints.** WordPress core schema has no FK constraints; adding them would conflict with `dbDelta`'s expectations. Skip.
- **InnoDB engine enforcement on already-existing MyISAM installs.** Out of scope for this plan; if some installs are still on MyISAM, that's a separate one-shot migration.
- **Pro extension data migration framework.** Will be added later if ANY extension needs a data transform. Not needed for this plan since dbDelta covers schema deltas.
- **Network-deactivate cleanup.** Currently no cleanup on per-blog deactivate; matches WC/EDD. Out of scope.

## 9. Risks & mitigations

| Risk | Mitigation |
|---|---|
| `dbDelta` mis-detects a column type difference and tries to ALTER repeatedly on every load | dbDelta caches its result for the request. Bigger risk: bumping `JETONOMY_DB_VERSION` on every plugin update would re-run install constantly. Mitigation: bump version ONLY when schema actually changes; check at code-review time. |
| Multisite `switch_to_blog` loop is slow on large networks | Only runs on activate or on version bump. Acceptable for one-time work. Add a hard cap (e.g. only auto-iterate first 100 blogs; rest get installed lazily on `wp_initialize_site` or first request). |
| Removing migration files breaks anyone whose stored DB version is exactly that migration's version | Keep all existing migration files in place. Only stop registering them. Their stored version transitions naturally past them via the new bump. |
| `Schema::install()` runs on every page load when version is stale, hammering the DB | `dbDelta` runs SHOW CREATE TABLE per table and compares — fast. Once version is bumped, it doesn't run again. Worst case: one extra reload after upgrade. Acceptable. |
| Pro extension activated AFTER Pro upgrade trigger has fired | Each extension's own `activate()` calls `install_schema()`, so enabling later still creates tables. Belt-and-suspenders. |
| `uninstall.php` requiring Schema breaks if Schema autoloader needs Composer | Use direct `require_once` of `class-schema.php` rather than relying on Composer autoload (Composer vendor/ may already be deleted by the time uninstall runs). |
| Network-uninstall vs single-blog uninstall | WP fires `uninstall.php` once per network. Add `is_multisite()` branch in uninstall to iterate `get_sites()` and drop per blog. |
