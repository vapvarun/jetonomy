## Journey alignment check (do these cards serve the two journeys?)

Re-examined the four cards through the site-owner and end-user journeys instead of
code-delta. Verdict: **all four sit on real journey friction points, and the journey lens
corrects three of the requirements.** Journey starts (owner first-run wizard, member
empty-states/join) are already in good shape, so these are legitimate mid-journey refinements.

### Site-owner / moderator journey — "run and keep a community healthy"
- **Mod parity (C2)** is the spine of the daily-moderation journey: accept the best answer when
  the asker didn't, fix a wrong accept, lock a toxic thread. CONFIRMED on-journey.
  - Journey correction: a closed topic must still let **moderators** reply (staff follow-up),
    only members are blocked. Current code hides the composer for everyone — must branch on mod.
- **Flag-icon (C4)** serves "spot reported content while browsing, not just in the queue."
  - Journey correction: the indicator is a dead-end if it only shows a number. The mod's next
    step is to act — the indicator should **link to the moderation queue / open the resolve
    action**, not just display a count.

### End-user / member journey — "participate and find things"
- **Advanced search (C3)** serves "find a past discussion / find an expert's posts" in a large
  community.
  - Journey correction (important): a raw `author_id` field is useless to a member who doesn't
    know user IDs. The journey requires a **name autocomplete** (reuse the existing post/space
    picker pattern), not a numeric input. Also consider filter-bar discoverability (currently
    collapsed behind `<details>`).
- **i18n sweep (C1)** serves "I run/use a non-English community and English leaks in
  toasts/confirms."
  - Journey priority: do the **high-traffic member strings first** (vote / report / delete
    confirms members hit constantly), then rarer admin strings. Not done until `.pot` ships so a
    translator can actually act.

### Honest limitation
These are four individual journey-moment fixes, not a full journey audit. They cluster sensibly
(moderation = owner, search/i18n = member) and each is on-journey, but they were chosen because
they were open cards. A deeper first-run -> daily-life -> find/participate audit could surface
larger gaps. Recommendation: ship these four (they're real and mostly small), and separately
schedule a proper journey audit as its own pass — don't conflate the two.

## Card 2 — Mod tools parity (9894927373)  [REQUIRED] — Effort: S-M

Server is ahead of the card. Three sub-items, two are tiny:

1. **Accept-answer mod gate — 1-line frontend.** `reply-card.php:171-178` shows Accept only when
   `get_current_user_id() === $post->author_id`. The REST `accept_reply()` **already allows mods**
   (`class-replies-controller.php:37-38`: author OR `check_permission('close_posts')`). So just
   widen the template condition to `( author || Permission_Engine::can($uid,'close_posts',$space->id) )`.

2. **Lock/Close topic — only the trigger is missing.** `/posts/{id}/close` route + `close_post()`
   exist; `single-post.php` already renders the closed badge (365), the closed notice (757-759),
   and hides the composer when `is_closed` (757 if/else 762). Missing: an overflow-menu item
   "Close / Reopen topic" (mods only) + `actions.toggleClose` JS calling the existing route.

3. **Un-accept — the only genuinely new build.** No route/action/button exists. Add
   `POST /replies/{id}/unaccept` (mirror accept gate), a JS action, and a button shown when
   `$is_accepted` + viewer is author/mod.

**Done:** mod sees Accept on others' Q&A replies; un-accept clears it; close/reopen toggles
from the menu (banner + composer-hide already work). REST-first, RTL, 390px, per space type.

## Card 4 — Flag-icon moderation indicator (9910490603)  [MODERATION] — Effort: S-M

**Verified state:** flags live in `jt_flags` (status `pending|approved|dismissed`), counted by
`status='pending'` (`class-flag.php`). There is **no `flag_count` on the post/reply payload**, and
the flag icon (`reply-card.php:136` `actions.flagReply`) is only the member *report* affordance —
no moderator "this has open flags" indicator anywhere except the moderation queue.

**Proposed build (NEEDS BEHAVIOR CONFIRMATION):** expose an open-flag count per post/reply for
viewers who can moderate, and render an indicator on the flag icon (e.g. a small count badge),
gated by `Permission_Engine::can($uid,'moderate',$space_id)`; clears when flags resolve.

**OPEN QUESTION — confirm before build:**
- Indicator style: count badge (`3`) vs filled/colored state?
- Surfaces: reply-card + single-post only, or also post-card + space listing?
- Moderators only (yes, assumed) — confirm.

**Done (once confirmed):** mod sees the indicator on flagged content; non-mods don't; clears on resolve.
