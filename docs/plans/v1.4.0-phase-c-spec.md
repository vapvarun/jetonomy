## C.1 (R4) - Inline Mod Tools Cap-vs-Role Regression

### Bug summary

`templates/partials/reply-card.php:93` and `templates/views/single-post.php:23` both gate inline moderation tools on `current_user_can('jetonomy_moderate')`. That cap is assigned exclusively to the editor WP role and above (`includes/permissions/class-capabilities.php:28-30`). A subscriber-role user who is a space moderator in `jt_space_members` (i.e., `Permission_Engine::is_space_privileged()` returns true) sees no inline edit, delete, or split-to-topic controls, even though the moderation queue and flag resolution DO correctly use `Permission_Engine::is_space_privileged()` via `Moderation_Permissions::can_act_on_flag()`. The permission model is split across two systems in the same feature area.

### Root cause

The templates were written using the simpler `current_user_can()` check before the space-role system fully matured. `Permission_Engine::is_space_privileged()` already has the correct three-layer logic (line 289-300 of `includes/permissions/class-permission-engine.php`): it short-circuits on `manage_options`, then falls back to `SpaceMember::get_role()`. The templates never received a corresponding update.

The same root cause applies to the `$is_mod` variable in `single-post.php:23`, which is then used at line 25 to decide whether to show a non-published post. Note: this second usage (showing pending posts to mods) is a correctness gap, not just a display gap - a space mod cannot review their own space's pending posts.

### Files to MODIFY

- `templates/partials/reply-card.php:93,105`
  - Line 93: replace `current_user_can( 'jetonomy_moderate' )` with `\Jetonomy\Permissions\Permission_Engine::is_space_privileged( get_current_user_id(), (int) ( $post->space_id ?? 0 ) )`. The `$post` variable is already in scope (passed by the single-post view).
  - Line 105 (split button guard): same replacement. The `$post->space_id` context is required and is already available in scope.

- `templates/views/single-post.php:23`
  - Replace the assignment `$is_mod = current_user_can( 'jetonomy_moderate' ) || current_user_can( 'manage_options' );` with `$is_mod = \Jetonomy\Permissions\Permission_Engine::is_space_privileged( $viewer_id, (int) $post->space_id );`. The `manage_options` check is already subsumed inside `is_space_privileged` (line 294 of the engine). The `$viewer_id` variable is set two lines earlier at line 22.

### Files to CREATE

- `tests/unit/templates/ModToolsVisibilityTest.php` - PHPUnit test asserting that a user with only subscriber WP role but `moderator` space-member row gets `is_space_privileged() === true` for that space, and `false` for a space they are not a member of. Does not test the template directly (no template unit test infrastructure), but validates the engine method that the templates now call.

### REST surface change

None. This is a pure template-layer and permission-helper fix.

### Tests to add

- `tests/unit/templates/ModToolsVisibilityTest.php`:
  - Assert: subscriber WP role + space `moderator` row -> `is_space_privileged()` returns true for that space.
  - Assert: subscriber WP role + no space row -> `is_space_privileged()` returns false.
  - Assert: `manage_options` user -> `is_space_privileged()` returns true regardless of space membership (regression guard for the WP admin bypass).
  - Regression guard: the exact scenario from the audit - user has `jetonomy_moderate` cap NOT set, space role = moderator, should still get `is_space_privileged() = true`.

### Customer-visible verification

### Cross-cutting impact

- `Moderation_Permissions::can_view_space_queue()` already uses `Permission_Engine::is_space_privileged()` (line 82 of `class-moderation-permissions.php`). No change needed there.

## C.2 (R5) - Join Request Notification URL

### Bug summary

`includes/notifications/class-notifier.php:763` builds the action link for join-request notifications as `admin_url( 'admin.php?page=jetonomy-spaces&action=edit&space_id=...' )`. Space admins and moderators with subscriber or author WP role receive this email, click the link, and hit a wp-admin 403. The 1.3.8 release shipped per-space mod queues at `/community/s/:slug/mod/` and the cross-space dashboard at `/community/mod/`. A community-side URL for join request review already exists - the notifier just never uses it.

### Root cause

The notifier was written before the front-end moderation routes existed. The hardcoded `admin_url()` call at line 763 assumes every recipient has wp-admin access. The recipient list built at lines 769-792 explicitly includes subscriber-role space moderators, making the broken link guaranteed to reach non-admin recipients.

### Files to MODIFY

- `includes/notifications/class-notifier.php:762-764`
  - Replace the `admin_url(...)` URL with a conditional that picks the correct URL per recipient. Extract a private method `build_join_request_url( object $space, int $recipient_id ): string` that:
    1. If `user_can( $recipient_id, 'manage_options' )`: return `admin_url( 'admin.php?page=jetonomy-spaces&action=edit&space_id=' . $space->id . '&tab=join_requests' )`.
    2. Otherwise: return `\Jetonomy\base_url() . '/s/' . $space->slug . '/mod/'`.
  - Move the URL generation inside the `foreach ( array_keys( $recipient_ids ) as $mod_id )` loop at line 801, calling `build_join_request_url( $space, $mod_id )` per recipient.
  - Pass the per-recipient URL into `create_and_maybe_email()` at line 813 instead of the shared `$space_url`.

Current code structure in the method:

```
$space_url = $space ? admin_url( '...' ) : '';   // line 762-764 (single shared URL)
...
foreach ( $recipient_ids as $mod_id ) {
    $this->create_and_maybe_email( $mod_id, ..., $space_url );  // line 805-813
}
```

After fix, `$space_url` is computed per recipient inside the loop.

## C.3 (R6) - Banned-User Login Filter and Frontend Ban Affordance

Two sub-fixes. Both must ship together to close the gap.

### Sub-fix 1: wp_authenticate filter to reject banned users at login

#### Bug summary

No `wp_authenticate`, `wp_login`, or `authenticate` filter exists in any Jetonomy file. A globally banned user in `wp_jt_restrictions` (type='global_ban') can log in to WordPress successfully. After login, every Jetonomy REST call returns 403 silently (via `Permission_Engine::resolve()` layer 0 at line 100). The user sees no explanation - features just fail. The audit confirmed no hook in any jetonomy PHP file intercepts the login flow.

#### Root cause

The `Restriction::is_banned()` model method already queries correctly (line 65-81 of `includes/models/class-restriction.php`), but it is only called inside `Permission_Engine::resolve()` on REST requests - never at login time.

#### Files to MODIFY

- `includes/class-jetonomy.php` (the main singleton bootstrap): add a hook registration in the constructor or `boot()` method:
  ```php
  add_filter( 'authenticate', [ $this, 'reject_banned_user_login' ], 99, 2 );
  ```
  The priority 99 runs after WP's own credential check so `$user` is already a `WP_User` object when our filter fires.

  Add the callback method `reject_banned_user_login( mixed $user, string $username ): mixed`:
  1. If `$user` is a `WP_Error`, return it unchanged (let WP's own error propagate).
  2. If `$user` is a `WP_User` and `Restriction::is_banned( $user->ID )` is true, return a new `WP_Error( 'jetonomy_banned', __( 'Your account has been suspended. Contact site admin for assistance.', 'jetonomy' ) )`.
  3. Otherwise return `$user` unchanged.

  The `authenticate` filter is the correct WordPress hook for blocking login at the credential layer. `wp_login` fires AFTER login is complete; `authenticate` fires during it and accepts a `WP_Error` return to abort.

#### Files to CREATE

None.

#### REST surface change

None.

## C.4 (R7) - Tags Fixes

Two sub-bugs. Both in the same commit.

### Sub-fix 1: Non-clickable tags in post-card.php

#### Bug summary

`templates/partials/post-card.php:64` renders tags as `<span class="jt-tag">`. The tag name has no href. The A2 audit confirmed `tag: "SPAN", href: null` in the DOM. Meanwhile, `templates/views/single-post.php:254` correctly renders tags as `<a>` elements. The inconsistency is isolated to `post-card.php`.

The tag URL pattern used elsewhere is `{base}/tag/{slug}/` - confirmed at `templates/views/tag.php:48`: `$tag_url = $base . '/tag/' . $tag->slug . '/'`.

#### Root cause

The post-card partial renders tags inline without access to the base URL builder (or rather, `$base` is already computed at line 14 of `post-card.php` as `\Jetonomy\base_url()`). No URL helper was added for the tag loop.

#### Files to MODIFY

- `templates/partials/post-card.php:63-65`
  - Change the tag rendering loop from:
    ```php
    foreach ( $tags as $post_tag ) :
        <span class="jt-tag"><?php echo esc_html( $post_tag->name ); ?></span>
    endforeach;
    ```
    to:
    ```php
    foreach ( $tags as $post_tag ) :
        <a class="jt-tag" href="<?php echo esc_url( $base . '/tag/' . $post_tag->slug . '/' ); ?>"><?php echo esc_html( $post_tag->name ); ?></a>
    endforeach;
    ```
  - `$base` is already set at line 14. `$post_tag->slug` is available from `Tag::list_for_post()` which returns full tag rows from `Tag::$table`.
  - The parent `<a>` wrapping the entire post-card row (line 35 of `post-card.php`) means tag clicks would normally bubble to the row link. Add `onclick="event.stopPropagation()"` or, better, restructure: `post-card.php` wraps the whole card in an `<a>`. Nesting `<a>` inside `<a>` is invalid HTML. The correct fix is to move the card wrapper to a `<div>` with a JS click handler, matching how the tag page's own `<a>` links work outside the card context. Check if tags are rendered inside the `<a class="jt-row">` wrapper (line 35): yes, they are in `.jt-row-sub`. The link nesting creates invalid HTML.
  - Correct approach: the `.jt-row-sub` area tag links must be `<button>` elements with JS `location.href` assignment, or the outer `<a>` wrapper must become a `<div data-href="...">` with a delegated click handler that ignores clicks on `.jt-tag` children. The simpler pragmatic fix used by many forum UIs: keep the outer `<a>`, make inner tags use `<a>` but add `onclick="event.stopPropagation()"` to prevent row navigation. This is technically invalid HTML but works in all modern browsers. For full correctness, replace the outer `<a class="jt-row">` with a `<div class="jt-row" tabindex="0" role="link">` with keyboard handler.

#### Files to MODIFY (continued)

- `assets/css/jetonomy.css`: ensure `.jt-tag` when inside `.jt-row-sub` as an `<a>` element has no underline and matches the existing pill style. The existing `.jt-tag` token-based styles should apply; verify no `color: inherit` override suppresses the `:hover` state.

Note: the tag model has `Tag::list_by_tag( $slug, $limit )` with a hardcoded join query (lines 171-197 of `class-tag.php`). The view currently runs its own raw query rather than calling this helper. Either extend `Tag::list_by_tag()` to accept `$offset` and `$limit` parameters, OR keep the raw query in the view and add pagination variables. Adding offset/limit parameters to the model method is the cleaner path per the plugin's "all data access via model classes" coding pattern.

- `includes/models/class-tag.php:171-197`

## C.5 (R8) - Read Status Wire-Up

## C.7 (R10) - Mention Autocomplete

### Bug summary

`assets/js/composer.js` (667 lines) has zero autocomplete code. No `GET /jetonomy/v1/users/suggest` endpoint exists in `includes/api/class-users-controller.php`. ### Root cause

The server-side mention parser and notification delivery work correctly. The frontend discovery layer (autocomplete dropdown, user suggest endpoint) was never built.

## C.8 (R11) - GDPR Display Name

### Bug summary

`includes/class-privacy.php:70` reads `$profile->display_name` as part of the GDPR profile export. The `jt_user_profiles` table schema does not have a `display_name` column (the `bio`, `trust_level`, `reputation`, `post_count`, `reply_count` columns exist, but not `display_name`). `display_name` lives on `wp_users`. The result is that every GDPR data export returns `"Display Name": ""` for all users. Confirmed by the A3 audit via direct `wp eval` against the export function.

### Root cause

The export function queries `jt_user_profiles` via raw SQL (`SELECT *`, line 54-59 of `class-privacy.php`) and then accesses `$profile->display_name`. Since the column does not exist, PHP returns `null`, and `?? ''` coalesces to an empty string at line 70. The `WP_User` object for this user is available via `$user` (set at line 45: `$user = get_user_by( 'email', $email )`).

### Files to MODIFY

- `includes/class-privacy.php:70`
  - Change `'value' => $profile->display_name ?? '',` to `'value' => $user->display_name ?? '',`.
  - `$user` is already in scope (set at line 45). No additional query needed.

### Additional sweep: all `$profile->` accesses in `class-privacy.php`

The audit finding notes a risk of similar column-name bugs for other fields. Verify each:

- `$profile->bio` - exists on `jt_user_profiles` (check schema). If confirmed present, no change.
- `$profile->trust_level` - exists (integer). No change.
- `$profile->reputation` - exists (integer). No change.
- `$profile->post_count` - exists (denormalized counter). No change.
- `$profile->reply_count` - exists (denormalized counter). No change.

None of the other five fields have the same column-vs-table mismatch. The fix is isolated to `display_name`.

### Files to CREATE

None.

### REST surface change

None. This is a GDPR data export callback registered via the WordPress `wp_privacy_personal_data_exporters` filter.

### Tests to add

- `tests/unit/privacy/PrivacyExportTest.php` (extend or create):
  - Assert: `export_profile( $user_email )` returns a data array where the item with `name = 'Display Name'` has a non-empty `value` equal to `$wp_user->display_name`.
  - Regression guard: assert `value` is not an empty string for a user whose `jt_user_profiles` row exists.
