# v1.4.0 Plan: Space Self-Service (front-end governance)

**Status**: planning
**Branch**: `1.4.0` (both `jetonomy` and `jetonomy-pro`, lockstep per release rule)
**Depends on**: 1.3.8 shipped (moderation queue + member role dropdown)

## Goal

Make a space fully self-serviceable from the front end. A space owner should never need to open wp-admin to understand or run their space. A visitor should never wonder who runs the space they just landed on.

This version closes 18 items across four phases — 7 customer-contract G-items (front-end governance), 3 Phase A REST migrations (app-readiness), 8 Phase C audit-driven fixes (🔴 → 0), plus opportunistic yellow pull-ins (Y.11 shipped) and 2 customer-reported bug-card fixes shipped early on the branch.

## Customer contract

| # | Contract | Today | After 1.4.0 |
|---|---|---|---|
| G1 | A visitor can see who runs this space without clicking around | No info anywhere on the space page | Sidebar "Managed by" card lists admins + moderators |
| G2 | A space owner has an obvious affordance to edit their space | No button anywhere on front end | "Edit space" link in sidebar + owner badge on space header |
| G3 | On any post or reply I can tell staff from regular members at a glance | Only trust level is shown | Role pill ("Admin" / "Mod") next to author in-space |
| G4 | Role changes cannot accidentally orphan a space or demote the last admin | REST accepts any role change blindly | Server + client guards, plus a confirm modal |
| G5 | A space owner can edit title / description / visibility / join policy from the front end | Only wp-admin | Full front-end edit at `/community/s/:slug/edit/` |
| G6 | A qualified user can create a space from the front end (when the site owner allows it) | Only wp-admin, even though the cap plumbing exists | Front-end `/community/new-space/`, gated by a new admin setting |
| G7 | A moderator of multiple spaces has a single landing view | Scattered | `/community/my-spaces/` |

## Scope rules

- Every item touches BOTH backend and frontend. No UI-only or data-only halves.
- Every item verified at 390px before marking done (per verify-per-item rule).
- **REST-first for every functionality.** Every read AND every mutation in this release must be reachable through a REST endpoint, not just an AJAX action or a server-rendered form post. The front-end may use REST directly; admin-only actions still go through REST so a third-party tool, the WP CLI, or a future mobile client can do the same thing without scraping. Reuse an existing controller if the contract fits; only add a new controller when no existing endpoint can carry the operation. Document the route, method, payload shape, response shape, and permission callback for each item. CRUD coverage is mandatory: a space owner can read, write, update, and delete every governance object via REST. Treat any "AJAX-only" or "form-post-only" path as a 1.4.0 bug.
- **RTL ready out of the gate.** Every new template / partial / block in this release ships with RTL parity. Use logical CSS properties (margin-inline-start, padding-inline-end, inset-inline-end, text-align: start) instead of left/right, so the browser handles flipping for free. Where a value genuinely cannot be expressed logically (icon directionality, mirrored arrows, scroll snap), provide a hand-tuned `[dir="rtl"]` override in the same SCSS / CSS source. The grunt `rtlcss` task auto-generates `*-rtl.css`, but auto-flipping is not enough on its own — verify each new feature visually under `<html dir="rtl">` before marking done. CRITICAL: feature is not done if the RTL viewport breaks layout or reverses any directional affordance customers will read wrong.
- Pro parity: Pro plugin gets a version bump with "Compatibility: Aligned with Jetonomy 1.4.0" readme entry. No Pro code changes planned.

## Phase A: frontend AJAX -> REST migration (added 2026-04-26)

The plugin still has three frontend AJAX handlers that must move to REST before the app team starts consuming the API. Backend / wp-admin AJAX is out of scope (see `feedback_frontend_rest_only_backend_ajax_ok.md`). Pro has zero frontend AJAX.

| Today | Migration |
|---|---|
| `wp_ajax_jetonomy_upload_image` (composer image paste / drag-drop / toolbar pick, called from composer.js, posts multipart to admin-ajax.php) | `POST /jetonomy/v1/media` (multipart). Permission: logged-in member. Response: `{ id, url, alt, mime, width, height }`. composer.js fetches the new endpoint with `X-WP-Nonce: wp_rest`. |
| `wp_ajax_nopriv_jetonomy_quick_login` (Login block submit) | `POST /jetonomy/v1/auth/login`. Permission: `__return_true` (public). Body: `{ user_login, user_password, remember? }`. `wp_signon()` sets the auth cookie internally. login-block.js fetches with `X-WP-Nonce: wp_rest`. |
| `wp_ajax_nopriv_jetonomy_quick_register` (Login block submit) | `POST /jetonomy/v1/auth/register`. Same shape as login. Honors existing rate-limit + duplicate-email checks already in `Blocks::ajax_quick_register`. |
| Forgot-password link drops user on `/wp-login.php?action=lostpassword` (unstyled WP core surface) | `POST /jetonomy/v1/auth/lost-password`. Body: `{ user_login }` (username or email). Calls `retrieve_password()` and returns a generic success regardless of existence. Login block grows a third panel — Forgot password — so the request stays inside Jetonomy's surface. The actual reset (token + new password) still uses WP core's flow because that's one-time-token plumbing WP owns. |

Each migration lands in three commits:
1. Add the REST endpoint (controller + permission_callback + tests).
2. Switch the JS callsite from admin-ajax to the REST endpoint, verify with browser smoke + a 390px pass + RTL pass.
3. Remove the AJAX handler + the `wp_ajax_*` action and any nonce/script localization tied to it.

Phase A unblocks the app and cleans the legacy. Phase B (the G-items) builds new front-end governance on top.

## Decisions (locked, 2026-04-26)

1. **Cover image picker — REST upload, not `wp_enqueue_media()`.** `wp_enqueue_media()` needs `upload_files`, which authors do not have by default, so a real customer hitting G5 from the front-end would get a silent permission wall. We use the REST media endpoint (existing if there is one; otherwise add a thin endpoint that gates on `is_space_admin($space_id)`) so any user who can edit the space can upload a cover, with no wp-admin dependency. Same uploader works on mobile.
2. **G6 gating — trust level OR WP cap (whichever fires first).** The `jetonomy_create_spaces` cap (default on Author+ role) covers staff/integrators, while `min_trust_level_to_create_space` covers earned community access. Either path qualifies a user. The trust default is **TL 2 (Regular)** to match `Trust_Levels::LEVELS[2]` which already grants the `create_spaces` ability — front-end gate stays consistent with the trust-system design. The master toggle `allow_frontend_space_creation` defaults to OFF; when an admin opts in, TL 2 means real engaged members start qualifying within days of normal activity, not weeks.
3. **"Managed by" card — role pill only, no trust level.** Trust level on every name on every space page is visual noise that drowns out the "who runs this place" signal. Trust level info already lives on the user profile and the leaderboard, where it is the primary content. Role pill alone here.
4. **My-spaces — both, with clear sections.** End users want one landing that answers "show me everything I am part of", not a forked UX. View renders two stacked sections: "Spaces I run" (admin / moderator, with quick links to Edit / Mod queue / Members) above "Spaces I'm in" (member-only, link to space). Empty sections are hidden. New helper `Models\SpaceMember::spaces_for_user(int $user_id, ?string $role_filter = null)` covers both queries.

## Decisions (locked, 2026-04-28)

5. **Phase order — strict A → B → C.** Phase A.1 ships first so G5's cover-image upload has zero blockers when its turn comes. Phase B G-items follow the suggested sequence (G1 → G4 → G3 → G2 → G5 → G6 → G7). Phase C closes with the smalls-first order (C.8 → C.6 → C.1 → C.2 → C.4 → C.3 → C.5 → C.7). One context switch per phase keeps PR diffs clean and the audit re-run's 🔴 → 0 drop reads as a single coherent gate.
6. **No migration shims; no schema changes; no backward-compat carry-over.** Plugin is young enough that breaking changes carry less debt than carry-over code. A.1 removes `window.jetonomyUpload.ajaxUrl` outright (no deprecation alias). C.5 read-status uses existing `jt_posts.last_reply_at` (validated at `includes/db/class-schema.php:163`) compared against `jt_read_status.updated_at` (validated at `includes/db/class-schema.php:262` — the columns are `user_id`, `post_id`, `last_read_reply_id`, `updated_at`; no `last_read_at`). No new columns, no `dbDelta()` schema change, no upgrade step. Across all of Phase A / B / C, zero schema mutation. When a behaviour change breaks an old surface, the readme.txt 1.4.0 changelog calls it out in customer language; that is the entire migration story for installs upgrading from 1.3.x.
7. **C.5 ships at full scope.** All four read-status surfaces land in the same release: (a) server-side `mark_read` from `templates/views/single-post.php` so visiting a post auto-marks it read, (b) REST endpoints `GET /posts/:id/read-status` and `POST /posts/:id/mark-read`, (c) post-card "new replies" pill on space view, (d) space-card unread count + bell badge on the home/community page. Computed via `LEFT JOIN jt_read_status rs ON rs.post_id = p.id AND rs.user_id = ?` and `p.last_reply_at > COALESCE(rs.updated_at, '1970-01-01')` — both columns already exist, both indexed. (Validated against `class-schema.php` 2026-04-28: `jt_read_status` columns are `user_id`, `post_id`, `last_read_reply_id`, `updated_at` — no `last_read_at`.) Admin setting `show_unread_indicators` (default ON) lets site owners turn the visual off without deactivating the feature. Y.3 (unread badges on space cards) is fully absorbed by C.5(d) and removed from the yellow list.

8. **Uniform long-term flow — admin self-control + sensible auto-mode.** Every new feature in 1.4.0 follows the same shape so an admin who learned one item already understands the next:
    - **Auto-mode default** — feature works out of the box on a sensible default. New installs see the behaviour without configuring anything.
    - **Admin self-control** — every behaviour change is exposed as a Settings → Jetonomy option (toggle, radio, or numeric clamp). No "edit a constant", no "filter hook in a child theme", no "wait for a follow-up release."
    - **Theme-default escape hatch** — opinionated visuals (Layout panel, badges, role pills) include a "Theme Default" or "Off" option that emits zero rules / renders nothing, so admins who want their theme in charge stay in charge.
    - **No surprise migrations on upgrade** — see decision 6.

    This rule applies retroactively too: where an existing G/C item only had a hardcoded default, add an admin-facing setting before tagging. Any feature that fails the admin-self-control check is a 1.4.0 bug. Self-control + auto-mode is what lets a young plugin scale across the next 4–5 years of installs without depending on developer hand-holding.

9. **Developer-friendly approach — every item is contract-grade, never half-cooked.** A site builder reading the plan or a third-party plugin author reading the code must always be able to pick up where we left off without asking. Every 1.4.0 item meets all four gates before its box is checked:
    - **Public extension surface.** Every behaviour change exposes one or more named extension points: a REST route, a PHP filter / action with documented arguments and return contract, or a JS event. No "internal only" code paths that only Jetonomy can call. Filters are namespaced `jetonomy_*` and listed in `docs/developer/HOOKS.md` (added by the first item that introduces a new hook).
    - **Code-level spec.** Phase A has `v1.4.0-phase-a-spec.md` and Phase C has `v1.4.0-phase-c-spec.md` — both file-by-file with REST contracts, signatures, test paths, and gotchas. **Phase B has no spec yet — that gap closes before G1 begins, as `v1.4.0-phase-b-spec.md`** (added to the verification gate below).
    - **Tests required.** Each item ships with PHPUnit (controller / model / integration) AND a `wp jetonomy qa-actions` smoke entry. Visual / template work also gets browser-verified at 390px AND `<html dir="rtl">`. Box stays unchecked if any of these is missing.
    - **Logical-flow validation.** Before starting any item, the developer (or the agent picking it up) traces the named code paths against the current source — if a referenced method, column, or file in the plan does not exist as written, the plan is wrong, not the code. Update the plan first, then implement. Decision 6 + 7 already validated for the C.5 column references; remaining items must be validated when their turn arrives.

    No backward-compat clause: per Decision 6 the plugin is too young for it to be a 1.4.0 concern. Removed surfaces are removed cleanly; the readme.txt changelog is the only carry-over. Why this matters for a young plugin: the way a site builder feels about Jetonomy in year 1 sets the ceiling for adoption in year 5. Every plan item is a public artifact — write it as a contract, not as a TODO.

10. **Browser verification gates every commit — no exceptions.** Every commit on the 1.4.0 branch is preceded by an end-to-end browser exercise of the surface it changes. Static checks (lint / WPCS / PHPStan) confirm the code compiles; only the browser confirms it works. The rule applies even to commits that have no direct front-end impact — a new REST endpoint without a frontend caller still gets a manual `fetch()` from the admin context to confirm the contract response shape and the permission gate (e.g. A.1 commit 1 verified `POST /jetonomy/v1/media` returns the documented `{id, url, alt, mime, width, height}` with status 200 for a logged-in admin and 401 for anonymous). For commits with a front-end surface, the verification covers desktop AND 390px AND `<html dir="rtl">`. If a commit genuinely cannot be browser-verified (purely static doc / changelog work) the commit body states that and lists the static checks that ran instead. **No browser verification = no commit.** This rule supersedes the implicit "verify-per-item" guidance in CLAUDE.md and makes it commit-level rather than feature-level.

**Gotchas surfaced by the code-level spec — read before starting Phase A:**

1. **Captcha is a silent gap in current `ajax_quick_register`.** The infrastructure (`Captcha_Manager::verify_or_skip()`) exists but is never called. A.3 closes this gap, but it forces converting `submitForm` in `login-block.js` to `async` because reCAPTCHA v3 returns a Promise. Plan an extra ~30 lines of JS rework on A.3.
2. **`jetonomyUpload.ajaxUrl` is not just a URL swap.** Renaming it to a REST URL is a breaking change for any third-party JS reading the global. No in-tree caller exists besides `composer.js`. Note in release docs.
3. **`check_rate_limit` is shared between login + register.** Do NOT remove it in A.2's commit-3 (the AJAX-handler-removal commit) — A.3 still uses it. It can only go after A.3's commit-3.
4. **`wp_signon()` already sets the auth cookie internally.** The original plan said "set cookie via `wp_set_auth_cookie`" — incorrect; `wp_signon` does it.
5. **Controller tests live at `tests/integration/api/`, not `tests/unit/api/`.** Original plan was wrong on this path.

### Phase D — SEO hardening (added 2026-04-28)

Audit on 2026-04-28 (Varun directive: "we have to make sure we are following SEO best practices, these are our ux and content overall") found that the post route emits a full SEO baseline (title / desc / canonical / og:* / twitter:* / DiscussionForumPosting JSON-LD) but **every other route on the community surface emits zero meta** when the Pro `seo-pro` extension is enabled. Root cause: `Template_Loader::set_seo_meta()` short-circuits the entire emit when `seo-pro` is in `jetonomy_pro_extensions`, but `seo-pro::output_meta_tags()` only handles `space` + `post` — leaving `home`, `profile`, `leaderboard`, `search`, `mod`, `tag`, `new-post`, `edit-profile`, `notifications`, `messages`, `invite` silent.

Phase D fixes the contract so the **free plugin alone is SEO-complete**, and `seo-pro` enhances rather than replaces. Customer impact: a community on free plugin is currently invisible to crawlers on every page that isn't a thread; a community on Pro is invisible on every page that isn't a thread or space.

- [ ] **D.1 Free plugin emits baseline meta on every public route.** Drop the `! $seo_pro_active` gate in `Template_Loader::set_seo_meta()`; replace with a "do-not-duplicate" allow-list flag (`jetonomy_seo_emit_seen`) so seo-pro can opt out of any specific tag it wants to override. Free always emits: meta description, canonical, og:title/description/url/type/site_name, twitter:card/title/description, plus image fallback chain (post inline → space cover → space icon → site logo → site icon → null) with og:image:alt.
- [ ] **D.2 Expand free route coverage.** `set_seo_meta()` switch grows from `home / space / post / profile` to **all 15 routes in `$template_map`**: home, category, space, space-members, space-roadmap, space-moderation, post, profile, notifications, search, leaderboard, moderation, tag, new-post, edit-profile, invite. Each route gets a sensible default `$title / $desc / $url`.
- [ ] **D.3 noindex on thin / private / auth surfaces.** `<meta name="robots" content="noindex, follow">` emitted on:
  - moderation, space-moderation (admin tools, not for crawlers)
  - new-post, edit-profile (input forms)
  - notifications, messages, conversation (logged-in personal views)
  - invite (one-shot landing)
  - search (paginated tail / duplicate content)
  - private spaces (any route under a space whose visibility !== 'public')
  - paginated tail beyond `page/3` (thread + space lists) — admin-toggleable.
  All toggles surface as Settings → SEO checkboxes (default ON for all of the above).
- [ ] **D.4 JSON-LD coverage matches route coverage.**
  - `home` → `WebSite` with `potentialAction: SearchAction` pointing at `/community/search/?q={search_term_string}`.
  - `space` → `CollectionPage` with `mainEntity: ItemList` of recent threads (top 10).
  - `post` → keep current `DiscussionForumPosting` / `QAPage` logic, plus `comment` array of accepted reply / top-voted replies (5 max).
  - `profile` → `Person` with `name`, `url`, `image`, `description` (bio), `sameAs` (social_links).
  - `tag` → `CollectionPage` with `mainEntity: ItemList` of tagged threads.
  - `leaderboard` → `CollectionPage` with `ItemList` of top members.
  - `BreadcrumbList` extended to cover all routes (currently only home / space / post).
- [ ] **D.5 og:image fallback + alt.** New helper `Jetonomy\SEO\og_image_for_route()` returns a deterministic image with alt text:
  ```
  post:    inline first <img> > space.cover_image > space.icon (rendered as PNG) > site logo > site icon > null
  space:   space.cover_image > space.icon (rendered as PNG) > site logo > site icon > null
  profile: avatar > site logo > site icon > null
  default: site logo > site icon > null
  ```
  Always emit `og:image:alt` (page title for posts, space title for spaces, display name for profiles, site name otherwise).
- [ ] **D.6 Admin Settings → SEO tab additions.** New checkboxes for D.3 noindex toggles (mod, edit, notifications, messages, search-paginated-tail), single-line text fields for `twitter:site` handle and `default og:image fallback URL`, plus a banner that explains the free baseline auto-emits everything and Pro adds per-space overrides.
- [ ] **D.7 Sitemap completeness check.** Existing `Posts_Sitemap_Provider` + `Spaces_Sitemap_Provider` register with WP core's sitemap, but verify the routes show up at `/wp-sitemap.xml` and that private spaces / drafts / banned-author posts are excluded. Add an admin "Verify SEO" button on the SEO tab that fetches `/wp-sitemap.xml` and reports counts.
- [ ] **D.8 Image alt enforcement on user content.** Avatars rendered via `Jetonomy\get_user_link()` already pass `alt`. Audit user-content images: `wp_kses_post` allows `<img alt="">` — so user-uploaded images via the composer must default-fill `alt` from the file name when the user omits it. Backfill is a nice-to-have, not blocking.
- [ ] **D.9 Doc pass.** `docs/website/admin-settings/05-seo.md` updated with the new noindex toggles, image fallback chain, and the "Pro enhances, free is complete" contract. `docs/website/pro-features/14-seo-pro.md` updated to reflect the new co-existence model.

**Acceptance criteria (Phase D):**
- `curl http://forums.local/community/` emits meta description + canonical + og:* + twitter:* + JSON-LD (`WebSite` + `BreadcrumbList`).
- `curl http://forums.local/community/u/admin/` emits meta description + canonical + og:* + twitter:* + JSON-LD (`Person` + `BreadcrumbList`).
- Same on `/leaderboard/`, `/tag/:slug/`, every space, every post, every category.
- `curl http://forums.local/community/mod/` (and other thin routes) emit `<meta name="robots" content="noindex, follow">` and still emit baseline meta (so paid scrapers / debug tools see context).
- All Phase D items browser-verified at desktop, 390px, and `dir="rtl"`.
- Free plugin alone (no seo-pro) emits the same baseline. seo-pro layered on top adds per-space overrides without duplicate tags.
