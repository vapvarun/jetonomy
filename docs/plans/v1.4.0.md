# v1.4.0 Plan: Space Self-Service (front-end governance)

**Status**: planning
**Branch**: `1.4.0` (both `jetonomy` and `jetonomy-pro`, lockstep per release rule)
**Depends on**: 1.3.8 shipped (moderation queue + member role dropdown)

## Goal

Make a space fully self-serviceable from the front end. A space owner should never need to open wp-admin to understand or run their space. A visitor should never wonder who runs the space they just landed on.

This version closes six customer-visible UX gaps uncovered during 1.3.8 QA.

## Customer contract

| # | Contract | Today | After 1.4.0 |
|---|---|---|---|
| G1 | A visitor can see who runs this space without clicking around | No info anywhere on the space page | Sidebar "Managed by" card lists admins + moderators |
| G2 | A space owner has an obvious affordance to edit their space | No button anywhere on front end | "Edit space" link in sidebar + owner badge on space header |
| G3 | On any post or reply I can tell staff from regular members at a glance | Only trust level is shown | Role pill ("Admin" / "Mod") next to author in-space |
| G4 | Role changes cannot accidentally orphan a space or demote the last admin | REST accepts any role change blindly | Server + client guards, plus a confirm modal |
| G5 | A space owner can edit title / description / visibility / join policy from the front end | Only wp-admin | Full front-end edit at `/community/s/:slug/edit/` |
| G6 | A qualified user can create a space from the front end (when the site owner allows it) | Only wp-admin, even though the cap plumbing exists | Front-end `/community/new-space/`, gated by a new admin setting |
| G7 | A moderator of multiple spaces has a single landing view | Scattered | `/community/my-spaces/` |

## Scope rules

- Every item touches BOTH backend and frontend. No UI-only or data-only halves.
- Every item verified at 390px before marking done (per verify-per-item rule).
- All mutations go through existing REST endpoints unless a gap forces a new one. No controllers added unless truly needed.
- Pro parity: Pro plugin gets a version bump with "Compatibility: Aligned with Jetonomy 1.4.0" readme entry. No Pro code changes planned.

## G1. Show who runs the space (visibility card)

**Backend**
- `Models\SpaceMember::list_privileged(int $space_id, int $limit = 20): array` — returns rows where role IN ('admin','moderator'), ORDER BY role DESC (admin first), joined_at ASC. Reuses existing composite index on `(space_id, role)` if present; otherwise Schema bump adds one.
- No REST surface change — sidebar reads the model directly.
- Unit test: `tests/unit/models/SpaceMemberTest.php` — coverage for mixed roles, empty, order.

**Frontend**
- `templates/partials/managed-by-card.php` — new partial, avatars + display name + role pill. Fall back to "No moderators yet" when empty (admins are always at least one after G6's fix).
- `templates/partials/sidebar.php` — include new partial immediately after the About card.
- CSS: `.jt-managed-by`, `.jt-managed-by-list`, `.jt-managed-by-row`, `.jt-managed-by-role`.
- Token rules apply — no hardcoded colors or spacing.

**Regression guard**
- "Managed by" card is visible on every space page including private / hidden spaces (same visibility as the About card).

## G2. Space-owner affordance

**Backend**
- No new backend. Reuses `Permission_Engine::is_space_admin`.

**Frontend**
- `templates/views/space.php` header: when viewer is space admin, render a compact pill "You admin this space" next to the space title.
- `templates/partials/sidebar.php`: "Edit space" link visible only to space admins — routes to `/community/s/:slug/edit/` when G5 ships, else to the wp-admin edit URL as interim fallback during the 1.4.0 build.

**Regression guard**
- WP admin (manage_options) always sees these affordances even without a space_members row. Enforced in Permission_Engine already.

## G3. Role badges on posts and replies

**Backend**
- Helper `Jetonomy\get_space_role_label(int $user_id, int $space_id): ?string` returning 'admin' | 'moderator' | null.
- Per-request memoization map keyed by `{space_id}:{user_id}` so rendering a reply list of 400 entries does not issue 400 queries. Populated on first call via a bulk `SpaceMember::roles_for_users(int $space_id, array $user_ids)` that fetches all roles in one query.

**Frontend**
- `templates/partials/post-card.php` — render pill next to author when role is set.
- `templates/partials/reply-card.php` — same.
- CSS: `.jt-role-pill`, `.jt-role-pill--admin`, `.jt-role-pill--mod`. Uses `--jt-accent-muted` and `--jt-danger-light` tokens.

**Regression guard**
- Pill only renders for the space the content belongs to. An admin of space A posting in space B gets no pill on the space-B post.

## G4. Role dropdown integrity

**Backend**
- `Models\SpaceMember::count_admins(int $space_id): int` helper.
- `Api\Space_Members_Controller::update_item` (or the existing spaces controller member-update route) rejects:
  - self-demotion — target_user_id === viewer_id and new_role !== 'admin' → `jetonomy_cannot_self_demote`.
  - last-admin demotion — current role is 'admin' and count_admins === 1 and new_role !== 'admin' → `jetonomy_last_admin_required`.
- Error messages localized in existing i18n payload.

**Frontend**
- `assets/js/space-members.js` — before PATCH, show `confirm()`-style modal with role summary; on 400 error from server, display error in existing `.jt-member-role-error` slot.
- Self-row already hidden in template — confirmed in 1.3.8, no change.

**Regression guard**
- Unit tests covering self-demotion and last-admin-demotion REST rejections land in `tests/unit/api/SpaceMembersControllerTest.php`.

## G5. Front-end edit space

**Backend**
- REST PATCH `/spaces/:id` reused as-is. Permission check already `is_space_admin`.
- Route added: `add_rewrite_rule( "^{$base}/s/([^/]+)/edit/?$", 'index.php?jetonomy_route=space-edit&jetonomy_slug=$matches[1]', 'top' )` in `class-router.php`.
- Template map in `class-template-loader.php`: `'space-edit' => 'views/space-edit.php'`.
- Permission gate inside the view: return 404 when `! is_space_admin`. Consistent with other gated views.
- Cover image upload: reuse existing REST upload endpoint if one exists; otherwise fall back to WP media picker via `wp_enqueue_media()`.

**Frontend**
- `templates/views/space-edit.php` — new view. Field parity with admin form: Title, Slug, Description, Category, Type, Visibility, Join Policy, Status, Icon, Cover, + settings (Who Can Post, Who Can Reply, Posts Per Page).
- `assets/js/space-edit.js` — form serialization, fetch PATCH, optimistic UI with rollback on error.
- CSS: `.jt-space-edit-form`, `.jt-space-edit-section`, reusing tokens. Full responsive at 390px (admin form is desktop-only today).

**Regression guard**
- Slug change triggers REST validation that is already in place (`Spaces_Controller::sanitize_slug`). Uniqueness enforced server-side; client shows error inline.

## G6. Front-end create space

**Backend — bug fix first**
- `Models\Space::create` currently does not seed the creator as a space admin. Only `Abilities::execute_create_space` does. Fix: `Space::create` now always inserts a `SpaceMember` row with role 'admin' for the current user when creator context is available. Abilities path becomes a pure wrapper. This is a behavior change that needs release notes.
- New admin setting `allow_frontend_space_creation` (bool, default **false**) under Jetonomy → Settings → General.
- REST POST `/spaces` permission check stays on `jetonomy_create_spaces` cap. The setting gates the front-end UI, not the REST endpoint (so wp-admin + Abilities + CLI keep working regardless).
- Route: `add_rewrite_rule( "^{$base}/new-space/?$", 'index.php?jetonomy_route=new-space', 'top' )`.
- Template map: `'new-space' => 'views/new-space.php'`.

**Frontend**
- Header CTA "+ Create Space" in `templates/partials/header.php` visible only when user has cap AND setting enabled.
- `templates/views/new-space.php` — simplified form (Title, Description, Type, Icon). Defaults: visibility=public, join_policy=open, category=none. Advanced fields deferred to post-create edit.
- `assets/js/new-space.js` — POST /spaces, on success redirect to `/community/s/:new-slug/`.

**Regression guard**
- `wp jetonomy qa-actions` extended to cover "author role creates space via REST → creator is space admin" — catches the Space::create seeding regression if it ever breaks again.

## G7. My spaces landing

**Backend**
- `Models\SpaceMember::privileged_spaces_for_user(int $user_id): array` — returns Space rows where the user has role 'admin' or 'moderator', ordered by role DESC then space title.

**Frontend**
- Route `/community/my-spaces/` and template `templates/views/my-spaces.php`.
- Cards list: space icon + title + role pill + quick links to Edit / Moderation queue / Members.
- Link from header avatar dropdown (if present) and from user profile page.

## Build sequence (suggested)

1. G1 (visibility card) — smallest, highest impact, no REST changes.
2. G4 (integrity guard) — data integrity priority. Does not block UI.
3. G3 (role pills on cards) — quick win once G1's helpers exist.
4. G2 (owner affordance) — depends on G5's route existing OR on interim wp-admin link.
5. G5 (front-end edit) — largest UI work.
6. G6 (front-end create) — requires Space::create fix first.
7. G7 (my-spaces landing) — can run parallel with G5/G6.
