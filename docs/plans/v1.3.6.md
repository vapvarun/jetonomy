# Jetonomy 1.3.6 — Plan

**Branch:** `release/1.3.6` (in both `jetonomy` and `jetonomy-pro`)
**Cut target:** 2026-04-25 (one week after 1.3.5)
**Freeze target:** 2026-04-23 EOD (48h burn-in on community.wbcomdesigns.com staging before tagging)

---

## Theme

**Make what we already shipped work correctly in every supported configuration, before we add anything new.**

1.3.0 through 1.3.5 shipped a lot of surface in 13 days: oEmbed providers, inbound embeds, theme bridges, an AI adapter layer, the Abilities API, base-slug redirects, demo seeding, 24 tables, two Gutenberg blocks, Akismet staff bypass, auto-join space membership, preserved import dates, counter rebuild, and more. Most of it works. Some of it has edge cases we have not hit yet because our own production install (community.wbcomdesigns.com) is about to open publicly and see real traffic patterns for the first time.

1.3.6 is a stabilisation release. No new marquee features. Every item below is either:

- **Correctness** — something we shipped that does not yet work in every supported configuration, or
- **Release hygiene** — something that prevented an incident (the 1.3.5 broken zip, the 1.3.1 hover regression, the wbcom-docs em-dash parser failure) from being caught by CI.

Anything that adds a new capability has been moved to P2 and should be cut from this release unless everything above it is complete.

**Explicitly deferred to 1.4.0 or later:** markdown editor, draft autosave, mention picker v2, topic templates, full accessibility sweep. Mobile app platform (see `docs/plans/app/`).

## P0 — release blockers

### P0.1 Disabled-extension ghost UI audit *(customer-reported — paired with Pro P0.1)*

**Problem.** Disabling a Pro extension (e.g. Messages) leaves its UI triggers visible in the free plugin's templates. Clicking lands on an error page because the extension's routes are unregistered. Same class of bug suspected for Reactions, Polls, Custom Fields, Badges.

**Contract (free-side).** Every free-plugin template location that surfaces a Pro feature must render through a single filterable hook named after the extension:

| Hook | Rendered in | Pro listener |
|---|---|---|
| `jetonomy_profile_message_button` | `templates/views/user-profile.php`, `templates/partials/user-hover-card.php` | `private-messaging` |
| `jetonomy_profile_badge_strip` | `templates/views/user-profile.php`, `templates/partials/user-hover-card.php` | `custom-badges` |
| `jetonomy_profile_custom_fields` | `templates/views/user-profile.php`, `templates/views/user-edit.php` | `custom-fields` |
| `jetonomy_reply_card_reactions` | `templates/partials/reply-card.php`, `templates/partials/post-actions.php` | `reactions` |
| `jetonomy_new_post_form_toolbar` | `templates/partials/new-post-form.php` | `polls` (Add Poll) |
| `jetonomy_admin_dashboard_widgets` *(existing)* | `includes/admin/views/dashboard.php` | `analytics` widget — verify guard |
| `jetonomy_admin_settings_tabs` *(existing)* | `includes/admin/views/settings.php` | Pro "Integrations" tab — verify guard |

Free renders *nothing* by default. Pro attaches only when the extension is enabled + licensed.

**Free-side deliverables:**
- Audit pass across `templates/**/*.php` — remove hardcoded Pro URLs/buttons, replace with the hooks above.
- Documentation in `docs/website/developer-guide/02-hooks-reference.md` — new "Pro extension surfaces" section with the contract.
- Fixture test `tests/integration/test-free-only-rendering.php` — boot free-only (Pro absent), render profile / post / space / moderation pages, assert zero "Pro" UI artifacts.

**Acceptance:** fixture test green; manual smoke on community.wbcomdesigns.com staging (all Pro extensions disabled one at a time) shows clean pages.

### P0.2 `bin/build-release.sh` — extend smoke-test coverage

**Problem.** The 1.3.5 bootstrap fatal surfaced because we added a smoke test. Different fatals manifest at different entry points. Current smoke only covers `plugins_loaded` + `init`.

**Deliverable.** Extend `tools/smoke-test.php` to boot the plugin through five phases, each wrapped in a try/catch with a distinct exit-code signature:
- `plugins_loaded` *(existing)*
- `init` *(existing)*
- `admin_init` *(new)*
- `admin_menu` *(new)*
- `rest_api_init` *(new)*

**Acceptance:** introduce a deliberately broken `admin_menu` callback on a throwaway branch; confirm `bin/build-release.sh` catches it and does not produce a zip.

### P0.3 CI guard: readme version-heading format

**Problem.** `parse_readme` in the wbcom-docs MCP returned 0 versions on 2026-04-18 because `readme.txt` used em-dash version headings. We hand-swapped them. Without a guard, this silently regresses.

**Deliverable.**
- `.githooks/pre-commit` — readme lint that rejects em-dashes, en-dashes, and non-ASCII separators in any `^= \d+\.\d+(?:\.\d+)? ` heading.
- CI step in `.github/workflows/*.yml` running the same lint.
- Shared `tools/lint-readme.sh` so free and Pro invoke identical logic.

**Acceptance:** `composer lint:readme` green on both plugins; dogfood test — running wbcom-docs `parse_readme` returns the full parsed version list.

### P0.6 Feature-hardening sweep — everything shipped 1.3.0 → 1.3.5

**Principle.** Every feature we shipped in the last 13 days has to work in every supported configuration. Not just the configuration we tested during release. This sweep is a forcing function to find the edge cases before real traffic does.

For each feature below, produce a one-line result in `docs/plans/v1.3.6-hardening-log.md`:
- **Pass** (verified in every configuration)
- **Fix needed** (issue found, landing in 1.3.6) + link to the fix commit
- **Defer** (issue found, scoped too large — moved to 1.4.0 with a Basecamp card)

## P1 — included scope (correctness on existing features, no new capability)

### P1.1 Akismet staff-bypass extended to topic creation
- 1.3.4 bypassed Akismet for staff *replies*. Topic creation by staff is still scanned.
- Fix in `includes/moderation/class-akismet-adapter.php::should_check_user()` — same role check (`manage_jetonomy`, space admin, space moderator).
- This is closing a bug in the 1.3.4 feature, not new capability.

### P1.2 Per-space counter rebuild — row action on Spaces admin list
- 1.3.3 added the dashboard-wide rebuild. Customers still open tickets asking for per-space.
- "Rebuild counters" as a row action in `includes/admin/views/spaces.php`; AJAX handler in `includes/admin/ajax/class-spaces-handler.php` (new `wp_ajax_jetonomy_rebuild_space_counters`).
- Reuses the existing rebuild logic. No new logic. UX-only addition on existing feature.

### P1.3 `/discuss/` perf profile *(blocker for the registration flip)*
- community.wbcomdesigns.com memory notes a **30-second page load** on `/discuss/`. Unacceptable before opening public registration.
- Profile with Query Monitor + Xdebug. Suspected hot paths: theme-integration `body_class` filter on every request, oEmbed provider registration on every page load, AI adapter localhost health-check timeout.
- If the load is still > 2s after profiling and fixes, **the registration flip is delayed until it passes.** Non-negotiable.

### P1.4 Docs sync automation
- On 2026-04-18 we manually copied 5 `.md` files to the store repo and ran 14 manual `create_changelog` calls. The lint guard (P0.3) fixes the parse failure; this automates the routine sync.
- `bin/sync-store-docs.sh` — dry-run-capable, copies `.md` files, preserves the store's `docs_config.json` format (bumps version field only), runs the link checker before committing to the store repo.
- `bin/sync-store-changelog.sh` — reads `readme.txt` changelog block, creates any missing versions on the store via wbcom-docs MCP.
- Both wired into `bin/build-release.sh` post-tag. Tooling, not new plugin functionality.

### P1.5 PHPStan baseline shrinkage
- Current level is 5 with a `phpstan-baseline.neon` that suppresses existing issues. The baseline has grown across 1.3.0–1.3.5 as CI was being built up.
- Sweep the baseline: for every entry that can be fixed safely in 1.3.6, fix it and remove the line. For entries that need architectural changes, leave them and add a `# TODO(v1.4.0): …` comment.
- Target: 10% reduction in baseline entries. Zero new entries added.

### P1.6 Jetonomy Login block — correctness fixes on 1.3.5 shipped feature
- **`users_can_register` honoured on every render** — currently the Register tab renders if the block saved `showRegister=true`, even if WP registration is disabled. Correctness fix. *(Not a new feature — closes a gap in the 1.3.5 implementation.)*
- **Server-side rendered error states** (bad password, rate limited, nonce mismatch) without a full page reload — the block currently renders a blank panel on error. Correctness fix.
- **`redirectTo` validated against `wp_validate_redirect()`** — current implementation is open-redirect unsafe. Security fix. **Must land in 1.3.6.**

### P1.7 Jetonomy Navigation block — correctness fix on 1.3.5 shipped feature
- **Respect `exclude_private_from_anonymous` site option** — currently the block renders private spaces in the tree for anonymous viewers when the cache is warm. Correctness + permission-leak fix.
- **Lazy-render is explicitly P2** — useful perf improvement but not required for correctness. Cut if P0 sweep runs long.

## P2 — new capability (cut unless P0+P1 complete with time to spare)

- **Jetonomy Navigation block lazy-render** — perf improvement for 1,000+ space sites. Not blocking correctness.
- **Setup wizard accessibility pass** — small. Worth grabbing before the 1.4.0 full sweep, but defer if the P0 hardening log shows any red.
