# Journey Audit: Site-Owner / Moderator Daily-Life

> **STATUS as of 1.4.4 (current).** Current spec: `journey-map-1.4.4.md`.
> - **Close/Reopen topic** (no UI + no reopen + mods locked out) — shipped `[free]` (`a4843c8`).
> - **Accept-answer mod gate** + **un-accept** — shipped `[free]` (`dc7f975`, `ceceb4d`).
> - **Flag indicator while browsing** — shipped `[free]`, denormalised `flag_count`, links to queue (`c11ab51`).
> - Open `[free]` (not blockers): ban/silence has no front-end UI (wp-admin only); front-end mod dashboard shows flags but not pending post/reply counts; no bulk-select in the front-end queue; admin flags table lacks excerpt/link.
> - Open `[pro]`: advanced-moderation auto-rule edits bypass `jetonomy_check_content`.

**Branch:** 1.4.4-dev

**Scope:** Cross-space moderation dashboard, flag queue, approve/spam/remove, ban/unban/silence, trust-level adjustment, accept/un-accept best answer, lock/reopen a topic, pin/move/merge, flag indicator while browsing, activity log, Pro auto-moderation rules.

## Top 3 Gaps Summary

1. **Close/Reopen topic has no frontend UI and mods are locked out of closed threads.** The REST endpoint exists but there is no button wired to it, no reopen path at all (model-level `close()` is one-way), and the `is_closed` composer guard does not carve out an exception for moderators. A mod cannot lock a thread, and if a thread is already closed (e.g. by import) the mod cannot reply to it from the frontend. This is a blocker for basic moderation workflow.

2. **Accept/un-accept answer is author-only in the UI even though the REST route allows mods, and un-accept does not exist at all.** A moderator who needs to accept the best answer on an orphaned Q&A thread (author deleted account, author inactive) cannot do so from the frontend. There is also no way to un-accept an incorrectly accepted answer without a direct REST call — the contract gap means accepted answers are permanent from the UI perspective.

3. **No passive flag indicator on the browsing surface.** The post/reply REST payload contains no `flag_count` field, and neither post-card.php nor single-post.php renders any signal for mods that a piece of content has open reports. A moderator reading through a space has no way to know which posts have been flagged without opening the mod queue separately — the two surfaces are completely disconnected.
