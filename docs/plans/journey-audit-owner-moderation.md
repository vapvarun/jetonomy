# Journey Audit: Site-Owner / Moderator Daily-Life

**Branch:** 1.4.4-dev
**Date:** 2026-05-23
**Method:** Three-lens (Expectation / Current Reality / Gap) code-grounded audit, with browser verification per `1.4.4-verification.md`.
**Scope:** Cross-space moderation dashboard, flag queue, approve/spam/remove, ban/unban/silence, trust-level adjustment, accept/un-accept best answer, lock/reopen a topic, pin/move/merge, flag indicator while browsing, activity log, Pro auto-moderation rules, Pro badge tooling, recurring moderation jobs on quiet sites.

---

## What got easier on 1.4.4-dev for moderators

- **The "report bomb" attack is closed.** Reporting a post deducts -10 from the author's reputation, and dismissing the flag restores the full -10. A bad-faith reporter can no longer chip away at someone's standing by reporting and walking away. Self-flags are excluded from the deduction at create time, so there is nothing to restore on dismiss.
- **Reporters see their own state on the post.** The Report button on a post now shows an "already reported" state with a tooltip when the current viewer has an open flag on that post. A second click yields a toast, not the reason form again. After a successful first report the same render flips inline, no page reload needed.
- **Pro auto-badges land in seconds.** Posting, replying, getting a reply accepted, voting, gaining reputation, or moving up a trust level all enqueue a per-user badge re-evaluation immediately. The six-hour safety-net cron still runs, but moderators no longer have to explain "wait for the cron" to members.
- **Award Badge in wp-admin is one click again.** The picker fires a single AJAX call. The duplicated confirm popup that used to leave an orphan modal is gone.
- **wp-admin unban dialogs look destructive.** The Banned Users tab uses the design-system danger token: Cancel is the ghost button, Confirm is the danger button. No more legacy primary-blue on a destructive action.
- **Quiet sites no longer drop moderation jobs.** Every recurring free job (trust re-evaluation, expired-ban cleanup, activity pruning, notification cleanup, scheduled-post publishing, verification reminder) now runs through Action Scheduler. Sites with low traffic that previously skipped WP-Cron ticks still see these jobs fire on schedule.
- **No more "called incorrectly" notices on every page load.** Pro extensions that schedule recurring work (email digest, custom badges, AI summarizer, reply-by-email, web push, webhooks) wait for the Action Scheduler data store before scheduling. Admin moderators no longer see the AS warning banner on every wp-admin visit.

## Top 3 open gaps after 1.4.4

1. **Close / Reopen topic still has no frontend UI and mods are locked out of closed threads.** REST route exists; no button, no reopen path, no mod exemption on the closed-composer guard. Still a blocker for basic moderation workflow when the next version opens.
2. **Accept / un-accept is author-only in the UI even though the REST allows mods, and un-accept does not exist at all.** A mod cannot accept the best answer on an orphaned Q&A thread from the frontend, and an incorrectly accepted answer is permanent from the UI perspective.
3. **No passive flag indicator while browsing.** Post REST payload still lacks `flag_count`, and post-card / single-post / reply-card render no signal for mods. The browsing surface and the mod queue stay disconnected; a mod has to leave the conversation to discover what is flagged.
