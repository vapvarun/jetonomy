# Jetonomy 1.7.1 — Cache Conformance Execution Plan

> **Release line:** 1.7.1 (free + pro, lockstep — same `x.y.z`). Ships alongside the 1.7.1 attachment/import work already on `1.7.1-dev`. Mirrors BuddyNext's `1.0.8-extended-object-cache-uniformity.md` playbook, adapted to Jetonomy's centralised `Cache` wrapper.
>
> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Bring every cached read in Jetonomy 1.7.1 (free + pro) into uniform conformance with the Wbcom Caching Standard — no cached datum served stale or leaked after a write, verified against a real persistent object cache. Scope is the complete cached-surface universe below; every function and template that touches a cached datum is covered.

**Architecture:** Jetonomy centralises object caching through one `Jetonomy\Cache` wrapper (`get/set/delete/remember/remember_object/flush`), unlike BuddyNext's per-service-inline shape. We keep that centralised architecture (it is Jetonomy's house pattern) and make it conformant by: (1) a model-level `Space::bust_cache()` that busts *both* space keys, routed through every writer *after* the DB write; (2) explicit id-scoped busts on the one member-triggered set-based write and `Cache::flush()` on the one-shot admin/CLI recompute paths; (3) moving the one hot recomputable read off a transient onto the object cache; (4) a style sweep for TTL constants, the flush guard, and stale docstrings. Every fix is verified prime→write→re-read on a Redis object cache (per Standard §4c: no persistent cache = the test passes for the wrong reason).

**Tech Stack:** PHP 8.1+, WordPress object cache API (`wp_cache_*`), PHPUnit (`composer test` / `composer test:combo`), Redis object-cache drop-in for verification.

## Manifest audit (pre-implementation gate — PASSED 2026-07-15)

Per the plugin's manifest-first rule, this plan was audited against `audit/manifest.json` before any code. Results:

1. **No new manifest-tracked surface.** The plan adds zero REST routes, AJAX actions, cron events, tables, capabilities, settings, or hooks — only internal methods (`Cache::delete_many`, `Space::bust_cache`, a `Space::delete` override). **No manifest refresh is required** by this work (the manifest inventories those surfaces, not PHP methods or cache internals).
2. **No existing helper to reuse (no duplication).** The manifest exposes no cache-bust/invalidate/flush service beyond `Cache::flush()` (kept) — `jetonomy_flush_rules` is rewrite-rules, unrelated. `delete_many` / `bust_cache` are genuinely new, not parallels of existing code.
3. **`Space::delete()` is the confirmed deletion chokepoint.** Actual deletion funnels through `Space::delete($id)` (caller `admin/ajax/class-spaces-handler.php:299`; also journeys). The T2 override therefore catches every deletion caller. The `jetonomy_delete_space` / `jetonomy_remove_space_member` manifest entries are `wp_ajax_*` **entry points**, not `do_action` bust hooks.
4. **Model-level busting chosen over a hook-listener — deliberately.** `jetonomy_after_update_space` is in the manifest but **not fired anywhere in the free tree** (stale entry, matches the manifest's own `discrepancies_2026_04_29` note), and the lifecycle actions that *do* fire (`jetonomy_after_create_space`, `jetonomy_post_updated`, `jetonomy_after_delete_post`) fire in **REST controllers** — so a listener would miss the AJAX-handler, CLI, import, and abilities callers. Busting inside the model methods catches every caller. Do NOT "refactor" this into a hook listener.
5. **No cache-key/transient contract in the manifest.** `jt_priv_members_*`, `space:slug:*`, `jetonomy_online_*` are undocumented internals, so the J3 transient→object-cache swap (T6) breaks no published contract.
6. **Free↔Pro seam clean.** Pro caches only into `jetonomy-pro` and never modifies a free space row, so no new cross-plugin bust hook is needed (verified in the audit; consistent with the manifest's `consumed_by_pro` surface, which this plan does not touch).

```php
// includes/models/class-space.php
/**
 * Bust every object-cache key that serves a space row: the id key and the
 * slug key. find_by_slug() caches under space:slug:{slug}, so a writer that
 * busts only space:{id} leaves the slug-keyed copy (which carries visibility)
 * stale — a privacy leak, not just staleness (Caching Standard §4b).
 *
 * @param int         $id   Space id.
 * @param string|null $slug Known slug; looked up from the row when null.
 */
public static function bust_cache( int $id, ?string $slug = null ): void {
    $keys = [ "space:{$id}" ];
    if ( null === $slug ) {
        $row  = parent::find( $id );           // uncached parent read — never the cached self::find
        $slug = $row->slug ?? null;
    }
    if ( $slug ) {
        $keys[] = "space:slug:{$slug}";
    }
    Cache::delete_many( $keys );
}
```

- [ ] **Step 4: Route `update()` through it, AFTER the write, busting old + new slug**

```php
public static function update( int $id, array $data ): bool {
    $old_slug = ( parent::find( $id )->slug ?? null );
    $result   = parent::update( $id, $data );
    self::bust_cache( $id, $old_slug );

        Cache::delete( "space:slug:{$data['slug']}" ); // a slug rename invalidates the new key too
    }
    return $result;
}
```

### Task 2 — design refinement logged during execution (2026-07-15): slug→id indirection

**Discovery:** `find_by_slug()` originally cached the *entire mutable space row* under `space:slug:{slug}` — a second copy of the same data already cached under `space:{id}`. That double-caching means every count change (`increment_post_count`/`increment_member_count`, hot on post-create/join) would have to bust the slug key too, which needs the slug and therefore an extra uncached read on a hot path.

**Refinement (implemented instead):** cache only the **stable** `slug→id` mapping under `space:slug:{slug}`; `find_by_slug()` resolves the id then delegates row data to `find($id)` (same `?object` return contract, consumers unchanged). Consequences:
- Row-data writes (`update` non-slug fields, both increments, privacy/recount recompute) bust **only `space:{id}`** — no slug lookup, no extra read on hot paths.
- The slug key is stable and only busted on a **slug rename or space delete** (rare) — `update()` busts old+new slug keys when `slug` changes; `delete()` busts `space:{id}` + `space:slug:{slug}`.
- Removes the double-stored mutable row (half the memory, half the staleness surface) — aligns with Standard §2b (don't cache the same datum twice).

`bust_cache(int $id, ?string $slug = null)` stays the primitive: id-only for row changes, id+slug for rename/delete. This supersedes the "bust id+slug on *every* writer" wording in the matrix — increments and count-recomputes are id-only under the indirection.
