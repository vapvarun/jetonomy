# 1.6.0 Remaining Audit Fixes — Remediation Plan (scale-gated)

Source: `audit/full-audit-2026-07-04.md`. Blockers already shipped (free `c526fdb`
/ pro `6c953ca`): 5 private/hidden-space content leaks + reply-by-email fatal/token.
**Still open: 5 Pro criticals + ~55 majors across 24 features.**

## MANDATORY SCALE GATE — validate BEFORE writing any code

Target deployment: **~10,000 sites, up to ~100,000 users AND 100k+ posts per
site.** No fix is implemented until its plan passes this gate in writing. A fix
that closes a bug but scans 100k rows, fires a query per row, or loads an
unbounded set is NOT accepted — it just moves the outage.

Every plan item must answer:
1. **Pagination** — `LIMIT`/`OFFSET` + a `COUNT(*)`; never an unbounded `SELECT`.
2. **Index** — every `WHERE`/`ORDER BY`/`JOIN` column has a `KEY` in `class-schema.php`
   (cross-check before shipping the query; add the index in the same change).
3. **No N+1** — no query inside a `foreach`. Batch-fetch (`WP_User_Query include`,
   `Model::find_many($ids)`, one JOIN) or a set-based check. **This applies to the
   FIX itself** — e.g. a per-recipient membership check in the digest must be one
   batched query, not one-per-post-per-user.
4. **Count via `COUNT(*)`** — never `count(list_all())`.
5. **Cache + invalidation** — shared/expensive reads get a cache key + a clear
   invalidation path on write.
6. **Concurrency** — handles already-deleted / already-resolved / already-voted.
7. **Cron fan-out** — anything sweeping all users/posts runs batched via Action
   Scheduler, not one unbounded pass per request.

Workflow per item: **plan (with scale answers) → you approve → TDD/code → `php -l`
+ REST-auth audit + boot smoke → browser/seeded-data verify → manifest + QA card →
commit.** No blind fixes.

## Wave A — SCALE-CRITICAL (code-checked plans — breaks at 100k users/posts)

Each item below was validated against the actual code (file:line) to confirm it
reuses existing helpers and adds no duplicate logic. **3 new indexes total**,
bundled into ONE migration (new class in `includes/db/migrations/`, registered in
`class-migrator.php::get_migrations()`), since `class-schema.php` only helps fresh
installs — existing sites need the ALTERs.

### A1 — Space members REST (trivial reuse, no new code)
`get_members()` at `includes/api/class-spaces-controller.php:674-695` calls
`SpaceMember::list_by_space($id)` unbounded and fakes `total = count($items)` (so
`has_more` is always false). The model ALREADY paginates: `list_by_space($id,
$limit, $offset)` (class-space-member.php:163) + `count_by_space()` COUNT(*)
(:183) — the frontend template already uses this pair. Fix = mirror the sibling
space-listing endpoint: `get_pagination()` (base:204) → `count_by_space()` →
`list_by_space($id,$limit,$offset)` → `paginated_response($items,['total'=>$total,
'offset'=>$offset])`. Index `space_role_joined` (schema:285) already covers it.
**No new helper, no model change, NO new index.**

### A2 — Moderation queue (consolidate a duplicate + add index)
`get_queue()` at `includes/api/class-moderation-controller.php:260` runs unbounded
raw `SELECT *` (posts :276, replies :284), merges + `usort`s the whole set in PHP,
hard-codes `has_more=false`. A SECOND, already-paginated copy of the same query
lives in the admin screen (`class-admin.php:1118/1131`). Both are raw `$wpdb` in
non-model code (violates the model-only rule). Fix:
- Add 4 model methods mirroring existing idioms: `Post::list_by_status()` /
  `count_by_status()` (copy `list_by_author` :552 / `count_by_space_visible` :461)
  and the `Reply` equivalents (:288 / :364).
- Rewrite `get_queue()` to use `get_pagination()` + the new methods +
  `paginated_response()`; register `get_collection_params()` on the route (:37).
- Repoint the admin screen onto the same methods → ONE queue implementation.
- **New index `status_created (status, created_at)`** on `jt_posts` AND
  `jt_replies` (currently full scan + filesort even for the admin query).

### A3 — Analytics top-contributors (set-based rewrite, add votes index)
`rest_top_contributors()` at `jetonomy-pro/.../analytics/class-extension.php:710-727`
runs 3 correlated subqueries PER user over the whole `jt_user_profiles` table (N+1
over 100k). The aggregate table `jt_pro_analytics_aggregate` has NO author
dimension (by design, class-aggregator.php:10-13) — **do not** use it here. Fix
(stay on the direct "query" path, mirror `rest_overview()` :570-593 grouped style):
- Rank authors with a UNION ALL of `GROUP BY author_id` over posts + replies in the
  date window (served by existing `author_created` index on both tables) → `LIMIT N`.
- Hydrate the ≤N winners with one `WHERE user_id IN (…)` join to profiles/users.
- Sum votes for the N winners only.
- **New index `object_created (object_type, object_id, created_at)`** on `jt_votes`
  (the one index-poor join; votes currently has only PRIMARY + the user_object unique).
- No new helper; reuse `resolve_date_range()` and keep the response shape.

### A4 — Trust-eval cron (cursor + batched fan-out, unify with CLI)
`evaluate_trust_levels()` at `includes/class-cron.php:115-198` selects `WHERE
trust_level < 4 LIMIT 500` with NO `ORDER BY`/cursor → it re-processes the same
first 500 every run; everyone past the window is never evaluated. Fix reuses the
free AS-direct pattern already in `ensure_scheduled()` (:85-107) — NOT Pro's Queue
(free can't depend on Pro):
- Turn the recurring `jetonomy_trust_evaluation` handler into a dispatcher: reset a
  new `jetonomy_trust_eval_cursor` option to 0, enqueue the first batch.
- Add worker `jetonomy_trust_evaluation_batch` (register beside :38-43): keyset
  query `WHERE trust_level < 4 AND user_id > :cursor ORDER BY user_id LIMIT n`,
  reuse the existing `replies_received` join (:139-150) + `Trust_Evaluator::
  evaluate_level()` (trust-evaluator.php:36) + the existing pre/post hooks; persist
  MAX(user_id); self-continue via `as_enqueue_async_action(..., self::AS_GROUP)`
  until a short slice; then clear the cursor. WP-Cron fallback single-events the hook.
- Extract the batch body into one private method the CLI twin (`class-cli.php:53-88`)
  also calls — no parallel implementation.
- **New index `trust_user (trust_level, user_id)`** on `jt_user_profiles` (existing
  `trust_reputation` can't serve the ordered keyset).

### A5 — Embeds cache (reuse Cache::remember, key by URL)
`Embeds::process()` at `includes/class-embeds.php:169` calls `wp_oembed_get()` per
URL on EVERY render (posts-controller:1142, replies-controller:755, single-post:412,
reply-card:83) with zero caching — WP core's oEmbed cache doesn't apply (Jetonomy
uses custom tables, not WP posts). Fix wraps it in the existing `Cache::remember()`
(class-cache.php:28), key `"oembed:".md5($url.'|680')`, TTL `DAY_IN_SECONDS`.
**Gotcha:** `remember()` treats `false` as a miss, and `wp_oembed_get()` returns
`false` for non-embeddable URLs → cache a sentinel (`''`) for negatives so failing
URLs don't keep storming the provider. Key-by-URL means no content-edit
invalidation needed. Use `Cache::remember` (string value), NOT `remember_object`,
NOT a raw `set_transient`.

### Wave A rollup
- New indexes (one migration): `status_created` on jt_posts + jt_replies (A2),
  `object_created` on jt_votes (A3), `trust_user` on jt_user_profiles (A4).
- New model methods: `Post`/`Reply` `list_by_status` + `count_by_status` (A2).
- Consolidations (kill dups): moderation queue (A2, REST+admin→models), trust
  batch body (A4, cron+CLI→one method).
- Pure reuse, no new code paths: A1, A5.

## Wave B — Privacy sweep #2 (release-blocking; each fix ALSO scale-gated)

Same class as the shipped blockers. Reuse `Permission_Engine::can(read)` /
`Space::content_visibility_sql` / `can_read_post`. **Scale note:** every one of
these must be a set-based / batched check — a per-row membership lookup across a
100k digest run or a big reaction list is itself an N+1 and fails the gate.

- [ ] **pro:email-digest** — digest leaks private/hidden-space content to non-members.
  Fix = add `s.visibility='public' OR member` to the compile query (single JOIN,
  not per-post membership lookups).
- [ ] **pro:polls** — `__return_true` on poll read/vote → private-space polls open to
  all. Gate on parent-post read (one check per request, not per option).
- [ ] **pro:site-announcements** — `fetch_announcements()` ignores `is_private` /
  visibility. Add the visibility predicate to the (already capped) query.
- [ ] **pro:reactions** — no read check on toggle/list. Gate + keep the count query
  aggregate.
- [ ] **pro:ai** `rest_suggest_reply` — add `can_read_post` gate.
- [ ] **free:mentions** — `Mentions::notify` notifies users who can't read the space.
  Filter recipients with ONE batched visibility query, not a check per mention.
- [ ] **pro:web-push** — DM/native push must respect visibility.

## Sequencing rationale (scale-first)

- **Wave A before everything** — these are the ones that actually knock over a
  100k-user site; correctness bugs at least return a wrong answer, these return a
  timeout. Also gives us the pagination/index patterns the later waves reuse.

  doesn't reintroduce a Wave A problem.
- Waves C–E follow, batched per extension so free/pro contract changes land together.
