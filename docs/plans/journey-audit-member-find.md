## Top 3 Gaps (summary)

1. **Search author filter is API-only, invisible in the UI.** The REST controller and `search.php` both accept `author_id` (integer) as a filter, but the advanced-filters form renders no author input at all (`search.php:197-218`). A member cannot filter by author from the UI; a developer could craft a URL but only if they know the numeric WP user ID — unusable by anyone.

2. **Seven `state.i18n` keys consumed by `view.js` are absent from the PHP `wp_localize_script` block.** Keys `voteFailed`, `reportPlaceholder`, `reportReplyPrompt`, `reportUserPrompt`, `reportUserPlaceholder`, `madePrivate`, `madePublic`, and `failedTogglePrivate` fall back to hardcoded English strings (`view.js:529,1217,1245,1272,1498,1504`). Every vote-failure toast, every report-a-reply/report-a-user dialog prompt, and every visibility-toggle toast is English-only for non-English members.

3. **Search-page server-side filtered query leaks private posts.** When advanced filters are active, `search.php:53-90` runs a direct `$wpdb->get_results` query with no `is_private` check. The REST `Search_Controller` correctly guards this (`class-search-controller.php:202-210`), but the template's own filter path bypasses the visibility layer — private posts are returned to any logged-in user searching by date, tag, or sort.
