## G1 — "Managed by" sidebar card

**Customer contract:** A visitor can see who runs this space without clicking around.

### Files to CREATE

**`includes/models/class-space-member.php` (extend, not new)**
- Add `public static function list_privileged( int $space_id, int $limit = 20 ): array`
  - Returns rows where `role IN ('admin','moderator')`, `ORDER BY FIELD(role,'admin','moderator'), joined_at ASC`.
  - Hydrates `display_name` + `avatar` for the template by joining against `wp_users` and using `get_avatar_url()` per row (memoise the per-user avatar URL inside the loop).
  - Cache-aware: store result under transient key `jt_priv_members_{space_id}` for 60s; bust on `SpaceMember::add` / `remove` / role-update.

**`templates/partials/managed-by-card.php`**
- Renders the card. Receives `$members` (array of objects with `user_id`, `display_name`, `role`, `avatar_url`) from the sidebar caller.
- Empty state: "No moderators yet." (admins are always at least one after Phase 0's `Space::create` fix landed).
- Markup: `<aside class="jt-managed-by"><h3>…</h3><ul class="jt-managed-by-list"><li class="jt-managed-by-row">…</li></ul></aside>`.

**`tests/unit/models/SpaceMemberListPrivilegedTest.php`**
- `test_orders_admins_before_moderators()`
- `test_orders_within_role_by_join_time()`
- `test_excludes_members_and_viewers()`
- `test_respects_limit_argument()`
- `test_empty_space_returns_empty_array()`

### Files to MODIFY

**`includes/api/class-spaces-controller.php`**
- Add `GET /spaces/{id}/privileged-members` route in `register_routes()`. Public (`__return_true`) per the REST-first reconciliation in v1.4.0.md. Response: bare array of `{user_id, display_name, avatar_url, role}`.
- New handler `list_privileged_members( WP_REST_Request $request ): WP_REST_Response|WP_Error` — wraps `SpaceMember::list_privileged()` and adds the `display_name` + `avatar_url` fields.

**`templates/partials/sidebar.php`**
- Include the new partial via `Template_Loader::partial( 'managed-by-card', [ 'members' => SpaceMember::list_privileged( $space_id ) ] )` immediately after the About card.

**`assets/css/jetonomy.css` (Token-rule compliant — no hex/px)**
- New rules: `.jt-managed-by`, `.jt-managed-by-list`, `.jt-managed-by-row`, `.jt-managed-by-role` (role pill, reuses `--jt-accent-muted`, `--jt-warn-light`).

### REST contract

```
GET /jetonomy/v1/spaces/{id}/privileged-members
Permission: __return_true (publicly readable)
200 → [ { user_id: int, display_name: string, avatar_url: string, role: 'admin'|'moderator' }, … ]
404 → jetonomy_not_found  (when space_id is invalid)
```

### Tests to add

- `tests/integration/api/PrivilegedMembersApiTest.php` — full HTTP test (200 with expected shape, 404 on bad ID, public anonymous reads succeed).
- `tests/unit/models/SpaceMemberListPrivilegedTest.php` — see above.

### Customer-visible verification

1. Visit any space page → sidebar shows "Managed by" card with admin avatar + role pill.
2. Add a moderator via `/spaces/{id}/members/{user_id}` PATCH → card refreshes within 60s (transient TTL) or on page reload.
3. Card visible on private/hidden spaces (same visibility as About card).
4. Browser-verified at 390px and `<html dir="rtl">`.

### Gotchas

- `display_name` is on `wp_users`, NOT on `jt_user_profiles` — the bulk `list_privileged` query JOINs `wp_users`. Avatar URL via `get_avatar_url($user_id, [ 'size' => 48 ])`.
- The transient key includes `space_id` only. If a site has 10k spaces and they're all hot, that's 10k transients — fine for the wp_options autoload story because each transient lives in its own row, not autoloaded.

## G4 — Role-change integrity

**Customer contract:** Role changes cannot accidentally orphan a space or demote the last admin.

### Files to MODIFY

**`includes/models/class-space-member.php` (extend)**
- Add `public static function count_admins( int $space_id ): int` — `SELECT COUNT(*) FROM jt_space_members WHERE space_id = %d AND role = 'admin'`.

**`includes/api/class-spaces-controller.php` — `update_member_role()` handler (existing PATCH `/spaces/{id}/members/{user_id}` route)**
- Two new server-side guards before the role update:
  1. **Self-demotion** — if `$target_user_id === $viewer_id` and `$new_role !== 'admin'`: return `WP_Error( 'jetonomy_cannot_self_demote', 'You cannot remove your own admin role.', [ 'status' => 400 ] )`.
  2. **Last-admin demotion** — if `SpaceMember::get_role( $space_id, $target_user_id ) === 'admin'` AND `count_admins( $space_id ) === 1` AND `$new_role !== 'admin'`: return `WP_Error( 'jetonomy_last_admin_required', 'A space must keep at least one admin.', [ 'status' => 400 ] )`.

**`assets/js/space-members.js` — existing role-change handler**
- Before PATCH, show a `confirm()` modal with a role-summary string. If user cancels, restore the dropdown to the previous value (track the original via `select.dataset.prevValue`).
- On 400 from server with one of the two new error codes, surface the `error.message` inline in the existing `.jt-member-role-error` slot. Keep the dropdown at the original value.

### REST contract change

```
PATCH /jetonomy/v1/spaces/{id}/members/{user_id}
Body: { role: 'admin' | 'moderator' | 'member' | 'viewer' }
NEW errors:
  400 jetonomy_cannot_self_demote   — viewer === target and new_role !== admin
  400 jetonomy_last_admin_required  — last admin trying to demote
```

### Tests to add

**`tests/integration/api/SpaceMembersUpdateGuardTest.php`** — new file:
- `test_self_demotion_returns_400()`
- `test_last_admin_demotion_returns_400()`
- `test_two_admins_demoting_one_succeeds()` (sanity baseline — guard does NOT fire when count_admins ≥ 2)
- `test_admin_demoting_a_moderator_succeeds()` (negative — only fires for admin→non-admin transitions on the last admin)
- `test_self_demotion_admin_to_admin_is_not_blocked()` (no-op edits don't trigger the guard)

### Customer-visible verification

1. Log in as the only space admin → open Space Members → try to demote yourself. Confirm modal appears. After confirming, server returns 400 and inline error renders. Dropdown reverts.
2. Add a second admin → re-try demoting yourself → succeeds.
3. With two admins, demote the OTHER admin → succeeds.
4. Try to change a moderator to member → succeeds (not gated by either guard).
5. Browser-verified at 390px (modal usable on mobile) and dir="rtl".

### Gotchas

- The two guards are separate by design — the same user could trigger BOTH at once (last admin and target is self). The handler checks self-demotion FIRST so the error message is precise. Do not collapse to a single error.

## G6 — Front-end create space

**Customer contract:** A qualified user can create a space from the front end (when the site owner allows it).

### Files to MODIFY

**`includes/admin/class-admin.php` — settings registration**
- Two new settings under Jetonomy → Settings → General:
  - `allow_frontend_space_creation` (bool, default `false`)
  - `min_trust_level_to_create_space` (int 0–5, default `2` = Regular per Decision 2)
- Add to the `sanitize_settings` whitelist with appropriate clamps.

**`includes/admin/views/settings.php`**
- Render the two new fields in the General tab.

**`includes/api/class-spaces-controller.php` — `create_item()` handler**
- Permission callback was `current_user_can('jetonomy_create_spaces')`. Update to OR-semantics per Decision 2:
  ```
  $has_cap = current_user_can( 'jetonomy_create_spaces' );
  $tl      = (int) ( UserProfile::find_by_user( $uid )->trust_level ?? 0 );
  $min_tl  = (int) get_option( 'jetonomy_settings' )['min_trust_level_to_create_space'] ?? 2;
  $has_tl  = $tl >= $min_tl;
  if ( ! $has_cap && ! $has_tl ) return permission_error();
  ```
- Note: the front-end UI gate (header CTA visibility) is separately gated by `allow_frontend_space_creation`; the REST endpoint itself is NOT gated by that toggle (CLI / Abilities / wp-admin paths must continue to work regardless).

**`templates/partials/header.php`**
- New "+ Create Space" CTA, visible only when:
  - `allow_frontend_space_creation` is true, AND
  - viewer has `jetonomy_create_spaces` cap OR trust_level ≥ setting.

**`includes/class-router.php`**
- New rewrite rule `^{$base}/new-space/?$ => index.php?jetonomy_route=new-space`.

**`includes/class-template-loader.php`**
- Map `'new-space' => 'views/new-space.php'`.

### Files to CREATE

**`templates/views/new-space.php`**
- Simplified form: Title, Description, Type, Icon. Defaults: visibility=public, join_policy=open, category=none. Advanced fields go to G5's edit screen.

**`assets/js/new-space.js`**
- POST `/jetonomy/v1/spaces` → on 201, redirect to `/community/s/{new-slug}/`.

### REST contract change

```

Permission: current_user_can('jetonomy_create_spaces') OR trust_level >= min_trust_level_to_create_space
Body: { title, description?, type?, icon? }
201 → created space object
403 jetonomy_forbidden  — neither cap nor trust threshold
400 jetonomy_validation
```

### Tests to add

  - `test_subscriber_with_tl_2_can_create()`

- Permission widening is on the REST endpoint, not the cap registration. Don't add `jetonomy_create_spaces` to the Subscriber WP role — that would over-grant. The trust-level branch only kicks in on the REST handler.

## Build sequence

Per Decision 5 phase-strict + Phase B suggested order: **G1 → G4 → G3 → G2 → G5 → G6 → G7**.

Rationale:
- G1 first builds the model+REST foundation (`list_privileged`, `roles_for_users`) that G3 reuses.
- G4 lands data-integrity guards before any G-item ships UI that exercises the role-change path.
- G3 piggy-backs on G1's bulk loader so reply-list pages don't N+1.
- G2 needs G5's route to exist OR the wp-admin fallback URL — start with the fallback, swap when G5 lands (one-line change in `get_space_edit_url`).
- G5 is the largest UI piece; lands when its dependency (A.1 media REST) is already shipped from Phase A.
- G6 reuses Phase 0's `Space::create` seeding fix; settings UI is the only new admin work.
- G7 can run in parallel with G6 if needed; both touch separate files.
