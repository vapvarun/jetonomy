## Cross-cutting findings (from this scan)

### Self-introduced regressions this session

The em-dash in `aria-label` for the Hidden badge — committed in `6ecd62c` (Bug 1 fix today) — violates `feedback_no_emdash_in_i18n_strings.md`. Sweep must include a hot-fix to `home.php:42` and `category.php:63` regardless of which contract sweep commit lands first.

```php
// templates/views/home.php:42 — fix needed
- aria-label="<?php esc_attr_e( 'Hidden space — only admins and members can see this listing.', 'jetonomy' ); ?>"
+ aria-label="<?php esc_attr_e( 'Hidden space - only admins and members can see this listing.', 'jetonomy' ); ?>"
```

(Hyphen-minus is the canonical fix per the rule. Same for category.php.)

### Lucide adoption gap

8 frontend surfaces (`avatar.php`, `breadcrumb.php`, `managed-by-card.php`, `pagination.php`, `invite.php`, `edit-profile.php`, `bookmarks.php`, `drafts.php`) have 0 or 1 Lucide call but are user-facing. Either they intentionally have no icons (avatar, breadcrumb, managed-by-card) or they should add them (pagination chevrons, invite-page CTA icons). Decision: per surface in Step 2.

### Admin / Dashicon split decision pending

10 of 14 admin views use Dashicons. The Lucide-only rule's wording is "in plugin UI strings" — admin chrome could legitimately stay on Dashicons since WP itself uses them and they ship for free with admin already loaded. Sweep target needs an explicit yes/no for admin migration. Default proposal: leave Dashicons in place for admin wp-admin chrome, but use Lucide for any frontend-facing surface even if rendered from admin (e.g., setup-wizard preview cards).

### setup-wizard.php inline SVG

The single inline-SVG file. Could be the wizard's logo / hero image (legit pasted brand asset, not an icon-as-icon). Step 2 verifies: if it's branded chrome, exempt; if it's a hand-drawn icon, replace with Lucide.

### tag.php pagination

Already on the patterns docket. Replace literal `←`/`→` with `chevron-left.svg` / `chevron-right.svg` (both shipped). 1-line template change × 2 lines.

## Anomalies / surprises (top 3)

1. **My recent Bug 1 fix introduced an em-dash i18n violation.** Caught by the audit, not by pre-commit. Pre-commit should grep for em-dash in `__()` family calls — would block the next regression. (Add to release CI later.)
2. **Reply-card already has the Quote Lucide icon.** The earlier UX punch-list item #5 was based on a visual misread of the actual SVG. Audit caught the false positive. Lesson: trust the inventory data, not screenshot intuitions.
3. **Pro registers zero shortcodes / blocks of its own.** Every user-facing Pro feature renders by hooking free's existing surfaces (filters in reactions, render injection in private-messaging, etc.). Means the sweep's Pro scope is mostly "audit the inline render hooks in 11 extensions," not "audit Pro shortcodes."
