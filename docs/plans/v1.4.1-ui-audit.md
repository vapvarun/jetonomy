# Jetonomy 1.4.1 — UI Uniformity Audit + Sweep

**Status:** PLANNING — no code changes until this plan is signed off.
**Scope:** Free + Pro plugins, every UI surface, single coordinated sweep.
**Driver rules (memory):** clean / spacious / universal UI · Lucide-only icons · no partial UI fixes · long-term fixes end-to-end.
**Both audits required:** code audit (inventory + findings) AND visual audit (screenshot baseline + diff) before any sweep ships. Live-site safety > sweep speed.

## Prior art — DO NOT redo what exists

These files are authoritative starting points. The audit consumes them rather than rescanning:

- `audit/manifest.json` — already enumerates 8 blocks (render handler, interactivity, purpose), 9 shortcodes (handler, attrs), 14 admin pages (capability, source file). Pull directly via `jq` queries.
- `audit/customer-experience-matrix.md` — 57 surfaces already triaged by 5 parallel Sonnet agents (April 2026): 13 🔴, 28 🟡, 12 🟢. Each row already three-lens analysed. Several of this plan's UX concerns already appear there as yellow findings — extract those into our findings table instead of re-discovering them.
- `audit/journeys.json` — customer journey definitions used by the matrix above.
- `jetonomy-pro/audit/manifest.json` — Pro mirror; has `extension_patterns`, `free_filters_hooked`, `free_services_consumed`. Every Pro extension's surface area is here.

What the manifests **do not** cover (this is the gap-fill scope of Step 1):

| Missing from manifest | What we add |
|---|---|
| `templates/views/*` (21 files) | Frontend page templates need a row each |
| `templates/partials/*` (11+ files + sub-folders) | Reusable cards / byline / composer partials need rows |
| `templates/emails/*` (2 files) | Email surfaces need rows |
| Pro `class-extension.php` inline UI | Extensions that render DOM from inline hooks |
| `includes/notifications/class-notifier.php` inline strings | Notification copy with HTML |
| **Icon-source counts per surface** | No manifest tracks this — new contribution |

---

## Why this exists

Single-template patches make the codebase *less* consistent — the touched template becomes the new outlier. Customers describe the product as a whole; clutter on one screen and breathing-room on another reads as "this thing is half-finished," which is worse than uniform clutter.

Bug 10 (mod queue) and Bug 1 (hidden spaces) shipped surface-specific fixes earlier today. This plan is the opposite: **one coordinated pass** across every UI surface so the design contract is uniform from the home page to the email digest.

---

## Goal

After the sweep, every Jetonomy UI surface satisfies the same five contracts:

| # | Contract | Description |
|---|---|---|
| C1 | **Icon source** | Every icon comes from `assets/icons/` (Lucide library) via `jetonomy_icon()` / `jetonomy_render_space_icon()`. Zero hand-drawn SVG, zero Unicode emoji in plugin-rendered strings, zero raw glyphs (`""`, ▲, ✓, ⚑). Dashicons allowed only in legacy-tag space-icon path (already emoji-defended). |
| C2 | **Action density** | Visible interactive controls per row capped at the right number for the surface (see C2 spec). Redundant controls are *removed*, not hidden behind `⋮`. |
| C3 | **Byline structure** | Single canonical pattern for "user · role · timestamp" across post cards, reply cards, member rows, leaderboard, notifications. No surface-local variants. |
| C4 | **Spacing & touch targets** | One canonical scale: card padding-block 16/20px, action-row gap 12px+, every interactive surface ≥ 44 × 44px on touch. Verified at 390px and 1280px. |
| C5 | **Empty / error / 403 states** | Every empty state has: Lucide icon + 1-line plain-language message + 1 primary CTA (or no CTA when truly nothing applies). No bare "No items." |

## Step-by-step plan

### Step 1 — Inventory completion (NO CODE)

**Output:** `docs/plans/v1.4.1-ui-audit-inventory.md`

For each surface above:
- File path
- Surface type (view / partial / block / shortcode / admin / email)
- Audience (guest / member / mod / admin)
- Key controls rendered (vote, react, reply, flag, edit, delete, etc.)
- Icon usage count: lucide / emoji / dashicon / raw / hand-drawn

Scan via `Glob` + `Grep` + targeted `Read` (header + first 100 lines per surface).

**Exit gate:** every surface listed in this plan has a row in the inventory file.

### Step 2 — Audit against the 5 contracts (NO CODE)

**Output:** `docs/plans/v1.4.1-ui-audit-findings.md`

Build a table:

| Surface | C1 icons | C2 density | C3 byline | C4 spacing | C5 empty | Notes |
|---|---|---|---|---|---|---|
| reply-card.php | ✘ raw `""` quote | ✘ 7 controls | ⚠ trust-level number | ✘ tight | n/a | … |

`✓` = compliant, `⚠` = partial, `✘` = non-compliant.

Findings are descriptive, not prescriptive — we do not propose fixes yet.

**Exit gate:** every cell filled with ✓ / ⚠ / ✘ + at least one note for any non-✓ cell.

### Step 3 — Lock canonical patterns (NO CODE)

**Output:** `docs/plans/v1.4.1-ui-audit-patterns.md`

For each contract, write the **one** target shape. Examples:

#### C2 — Action density spec

| Surface | Visible controls | Goes behind `⋮` (rare admin only) | Removed entirely |
|---|---|---|---|
| Reply card | React, Reply, Report | Edit, Delete, Pin, Permalink | Vote arrows (replaced by React), separate Quote button (now in composer toolbar) |
| Post card | React, Reply count, Report | Edit, Delete, Pin, Permalink | Vote arrows, separate Quote button |
| Member row | Message, Follow | Block, Mute, Report | (n/a) |
| Notification row | Mark read, Open | Mute thread, Mark all read | (n/a) |

#### C3 — Byline spec

```
[Avatar] Username · ROLE_BADGE · OP_BADGE · 3 weeks ago
```

- Trust level number drops from byline (lives on profile + hover card only).
- Tags move to a row below the post body, not in byline.
- ADMIN / MOD / OP badges are short-text chips with role-color background.
- Timestamp is relative + `title=` absolute.

#### C4 — Spacing scale

- Card padding-block: 20px desktop, 16px mobile
- Card padding-inline: 20px desktop, 16px mobile
- Action-row gap: 12px minimum
- Section-to-section margin: 32px vertical
- Touch target minimum: 44×44px (chevrons, vote buttons, icon-only actions all conform)

#### C5 — Empty-state spec

```
[Lucide icon, 48px, neutral grey]
[Plain-language one-liner — no error tone]
[Single primary CTA (or none)]
```

Centralised partial: `templates/partials/empty-state.php` accepting `icon`, `message`, `cta_label`, `cta_url`.

#### C1 — Icon palette + naming

Whitelist the Lucide names we'll use across the plugin. Anything outside this list needs a Basecamp card before adoption. Goal: consistency, not minimalism.

**Exit gate:** every contract has one canonical pattern documented; reviewers (and future agents) can apply it without rediscussion.

### Step 3.5 — Visual baseline (NO CODE — but takes screenshots)

**Output:** `docs/plans/v1.4.1-ui-audit-visual-baseline/` directory with one screenshot per (surface × viewport × theme-mode).

For every surface in the inventory:
- 390px viewport (mobile)
- 1280px viewport (desktop)
- Light mode
- Dark mode (Reign toggle on)

Filename convention: `{surface-id}__{viewport}__{mode}.png` so post-sweep diffs can match by name. Surface ID is the file slug (e.g. `views-home`, `partials-reply-card`, `block-forum-feed`).

For surfaces gated by audience (admin-only, mod-only), capture at the appropriate role using `?autologin=admin` / `?autologin=bob`.

RTL spot-check on 5 representative surfaces (per the open-question answer): `views/home.php`, `templates/partials/reply-card.php`, `views/single-post.php`, `views/space.php`, `views/notifications.php`.

**Why this step exists:** when sweep commits land, we re-screenshot every surface. The diff tells us *exactly* what changed visually — intentional improvements vs accidental regressions. No diff = nothing to defend in code review or customer support.

**Exit gate:** every inventory surface has at least 4 baseline images (2 viewports × 2 modes). Folder committed to the branch (lightweight PNG, captured at scale=css). Visual-baseline manifest written as JSON so the post-sweep diff tooling can iterate.

### Step 4 — Sweep (CODE LANDS HERE)

One commit per contract, every surface touched in that commit.

**Commit sequence:**

1. `chore(ui): centralise empty-state partial + sweep all empty paths through it` (C5)
2. `chore(ui): byline canonical structure across post / reply / notification / member surfaces` (C3)
3. `chore(ui): action-density cleanup — remove redundant controls, ⋮ menu only carries true admin actions` (C2)
4. `chore(ui): icon source uniformity — every icon via jetonomy_icon, no raw glyphs / emoji / hand-drawn SVG` (C1)
5. `chore(ui): spacing scale + touch-target compliance` (C4)

Each commit is self-contained, passes WPCS / PHPStan / PHPUnit. Pre-commit hook enforces.

Free + Pro changes ship together when a Pro surface is in the affected set.

### Step 5 — Verify uniformly

For each surface in the inventory:
- 390px viewport screenshot
- 1280px viewport screenshot
- 4 viewer modes where relevant (guest / member / mod / admin)
- RTL pass (`html[dir=rtl]`)
- Both light + dark mode (mirror via Reign toggle)

**Output:** `docs/plans/v1.4.1-ui-audit-verification.md` with one block per surface and ≥ 2 screenshots.

**Ship gate:** zero ⚠ / ✘ in the post-sweep version of the findings table. Any surface that didn't ship the fix gets the fix in this PR or it doesn't merge.

## Out of scope (deliberately)

- **New features.** No new buttons, no new menus, no new pages. Cleanup only.
- **Theme-side fixes.** Reign's vertical dark-mode pill, Reign's mobile bottom-nav z-index — separate cards on the Reign repo.
- **Admin AJAX → REST migration.** Different rule (REST-first frontend, AJAX OK in backend). Not part of UI uniformity.
- **Block editor (Gutenberg) chrome.** Editor previews must not regress, but redesigning the editor sidebar is out.
- **Email full redesign.** We sweep contracts on existing email templates; we don't reflow brand identity.
- **i18n string churn.** Only change strings when their structure changes (e.g. byline copy). No "quality copy editing" pass mixed in.

---

## Risks + mitigations

| Risk | Mitigation |
|---|---|
| Sweep is large — accidental regressions | One commit per contract, each with its own browser-verification gate. Smaller blast radius per change. |
| Pro lockstep — Pro extensions render UI we may miss | Step 1 explicitly enumerates each of 14 Pro extensions; Step 4 commits include the matching Pro file. |
| RTL drift — logical CSS properties not used everywhere | Step 5 includes RTL viewport in verification; existing `*-rtl.css` is rebuilt by `grunt rtlcss` at release. |
| Trust-level number removal feels like loss of feature | Patterns doc spells out the move (TL # only on profile + hover card). Hover-card affordance covers discoverability. |
| Action-density change breaks muscle memory for power users | The actions still exist (most move into `⋮` for admin/owner cases, which are the only ones that change). Reader-facing actions (React, Reply, Report) stay visible. |
| Reactions extension is Pro — free users have only Vote today | C2 says "vote arrows replaced by React"; for free-only installs, *vote* stays as the primary engagement and *React* simply isn't rendered. Free-side spec uses `vote`; Pro-side spec uses `react` and Pro replaces the partial. Spelled out in Step 3 patterns doc. |

## Open questions for sign-off

1. **Vote vs React on free.** Free has Vote (up/down arrows + score), Pro has Reactions. Do we keep Vote as free-only and let Pro swap-in Reactions transparently? (My read: yes — fewest moving parts.)
2. **Trust-level number visibility.** Are we OK losing the inline TL2 / TL3 chip from bylines, or is that a recognised forum convention we want to keep? (My read: drop from byline, keep on profile / hover-card only.)
3. **Empty-state partial extraction.** Are we OK adding `partials/empty-state.php` as part of this sweep? It's the cleanest path but it's one new file. (My read: yes — saves duplicate copy across 21 views.)
4. **Verification cost.** Step 5 with 4 viewer modes × 4 viewports × 60+ surfaces is ~1 dev-day of screenshots. Can we narrow to 2 viewports × 2 viewer modes per surface? (My read: yes — guest + admin, 390px + 1280px is enough; RTL spot-checked on 5 representative surfaces.)
5. **Pro surface scope.** Step 1 lists 5 view files in Pro. We may discover more inline UI in `class-extension.php` files. Are we scoping all 14 extensions in this audit, or only the 3 with explicit `views/` folders? (My read: all 14 — anything that emits HTML on the frontend gets audited.)
