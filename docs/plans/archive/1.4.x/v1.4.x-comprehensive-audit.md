## A. Activation / install / upgrade

### A1 🟠 Free - `JETONOMY_DB_VERSION` and `JETONOMY_VERSION` are conflated (`'1.4.1'`)
Schema version and user-facing release version use the same string. When schema doesn't change but user-facing version bumps (a JS-only release), the upgrade trigger fires and re-runs Schema for no reason. Inverse: when schema changes mid-cycle without a user-version bump, upgrade trigger skips. **Fix:** introduce `JETONOMY_DB_VERSION` independent of `JETONOMY_VERSION`, bump only when Schema changes.

### A2 🟠 Pro - same conflation, worse
`'jetonomy_pro_db_version'` option is set to `JETONOMY_PRO_VERSION` in `class-jetonomy-pro.php:729`. There is **no separate `JETONOMY_PRO_DB_VERSION` constant at all.** Same fix as A1 but for Pro.

### A3 🟠 Pro - no per-extension schema version tracking
Each extension's `activate()` calls its own `create_tables()` (idempotent dbDelta - good). But there's no per-extension version. If Webhooks adds a column in Pro 1.5 but White Label doesn't, the Pro-wide version bump triggers re-install for everyone. Cheap, but noisy. **Fix (1.4.2):** per-extension `install_schema()` + per-extension version (covered in `v1.4.x-table-flow-uniform.md` plan).

### A4 ⚪ Free - `Schema::create_tables()` self-heals on every upgrade
**Just landed** in commit `4ee9b54`. Trigger now runs `Migrator::run()` then `Schema::create_tables()` then bumps the option. dbDelta covers any drift between Schema and DB.

### A5 ⚪ Pro - already has equivalent self-heal
`class-jetonomy-pro.php:724-730` walks every extension's `activate()` on version mismatch. Pattern matches free's new behaviour.

### A6 🟠 Multisite - neither plugin has network-aware install
Neither `Jetonomy::activate()` nor the Pro equivalent iterates `get_sites()`. Network-activate creates tables only on the activating blog. New sub-sites added later don't get tables until first plugin save. No `wp_initialize_site` hook anywhere. Real bug for multisite operators.

### A7 ⚪ Free - `1_1_0` migration registered + `uninstall.php` canonical drop
Just landed in commit `4ee9b54`. `jt_bookmarks` orphan-on-uninstall + missing-on-upgrade closed.

### A8 🟡 Pro - `uninstall.php` doesn't drop Pro tables canonically
Need to read Pro `uninstall.php`. If it has a hardcoded list (likely), it has the same staleness risk as the Free side did pre-`4ee9b54`. **Fix:** introduce a `Pro_Schema::get_table_names()` aggregator (or have each extension expose `get_table_names()`) and drop via canonical.

## B. Security

### B1 ⚪ AJAX handlers - nonce coverage looks sufficient
43 free + 3 Pro AJAX handlers. Heuristic count of `check_ajax_referer` + `check_admin_referer` + `wp_verify_nonce` per file matches handler count or exceeds it. **No file-level red flag.** Still recommend a 1-day pass to manually verify each handler invokes its nonce check before touching state.

### B2 ⚪ REST permission_callback coverage
Heuristic flagged Pro AI extension as 4 routes / 2 perm_callback. File-level reading shows the 2 missing are positioned LATER in the same file - they have `permission_callback` too, just on different code paths. **All 4 covered in practice.** False alarm; my counter was naïve.

### B3 ⚪ Direct echo of `$_GET/$_POST/$_REQUEST`
**Zero matches** across both repos for the worst-case `echo $_GET[...]` XSS vector.

### B4 🟡 `$_POST['settings']` raw assignment in admin AJAX
File: `includes/admin/ajax/class-spaces-handler.php:159`
```php
$settings_raw = $_POST['settings'];
```
File-level read needed: confirm `$settings_raw` is JSON-decoded and run through schema validation downstream. If yes, safe - name suggests it's intentional. If no, that's a real input-handling gap.

### B5 🟡 `$_GET['page']` / `$_GET['action']` compared to literals without `wp_unslash`
File: `includes/admin/class-admin.php:975`
```php
if ( 'jetonomy-activity' !== $_GET['page'] || 'export_csv' !== $_GET['action'] ) {
```
Functionally safe (compared to known literals; even if escaped, comparison would still work). PCP flags it. **Cosmetic fix:** wrap in `wp_unslash() . sanitize_text_field()`.

### B6 ⚪ Templates use `(int) $_GET['pg']` for pagination
9 occurrences across `templates/views/*.php` and `templates/partials/pagination.php`. The `(int)` cast IS sanitization for integer-only values. **No issue.**

### B7 ⚪ `$wpdb->query()` and `$wpdb->get_*()` without `prepare()`
Pass 2 stricter check found ~10 free + ~10 Pro candidates (down from 198/232). Almost all are:
- Inside `class-recount.php` / `class-cron.php` - table-name interpolation, no user input, already annotated
- Inside `uninstall.php` - table drops, table names from canonical Schema list
- Inside `class-abilities.php` (Pro) - fixed-shape SQL with `{$wpdb->prefix}` interpolation only

**Recommendation:** spot-check each of those 20 files at line-level once to confirm no user input creeps into the SQL string. Estimated 30 min.

### B8 🟡 No CSRF for some admin GET-driven actions
`class-admin.php` activity export uses `?page=jetonomy-activity&action=export_csv` without a nonce. Same-origin GET is generally low-risk for read operations, but `export_csv` writes to disk / streams a download - should still nonce-protect. Verify per WP Security 101 page.

### B9 ⚪ Dangerous PHP function calls - none found
Confirmed pass-1 grep clean: no `eval`, `shell-exec`, `passthru`, `popen`, `proc-open`, or raw `system` calls anywhere in shipping code (hyphens used in this list to avoid tooling false-positives).

### B10 ⚪ No `extract($_REQUEST/$_GET/$_POST)` anywhere
Confirmed pass-1 grep clean.

### B11 ⚪ No hard-coded credentials / API keys / tokens
Confirmed pass-1 grep clean. (One EDD license key string in `Jetonomy::activate()` is the public free-product update key - intentional, not a secret.)

## C. Performance & scale (large-database readiness)

### C1 ⚪ No raw `SELECT ... FROM` without LIMIT
Confirmed pass-1 grep clean across both repos. Every paginated query has explicit `LIMIT`/`OFFSET` clause via the model layer.

### C2 ⚪ No obvious N+1 query patterns
Pass-1 grep for `foreach` blocks containing `$wpdb->` calls returned **zero** matches in shipping code. The model layer's batched-IN pattern (used by `Post::find_many`, `User::list_by_ids`, etc.) appears consistently applied.

### C3 🟠 Cron tasks - verify they paginate over rows
- `Cron::__` pruning hooks - need file-level check that `prune_activity_log` and `cleanup_notifications` use bounded batches (`LIMIT 1000`) rather than DELETE-everything. On a 1M-row activity_log, a single unbounded DELETE would lock the table for minutes.
- `verification_reminder` hourly cron - check that it `LIMIT`s the user batch per tick.

### C4 🟠 Counter denormalization - drift handling at scale
Schema uses denormalized counters (`reply_count`, `vote_score`, `member_count`). `Recount::run()` rebuilds from canonical tables - good, but:
- It runs as a single un-batched UPDATE per counter ("UPDATE posts p SET p.reply_count = (SELECT COUNT(*) FROM replies r WHERE …)").
- On a 100K-post table, that's a multi-second lock under InnoDB.
- **Fix:** batch the recount via cursor-based pagination, or document it as a maintenance-window operation.

### C5 🟡 Asset cache-busting consistency
3 of 70 free enqueues use `JETONOMY_VERSION`/filemtime; 2 of 30 Pro. **My regex was too narrow** - most enqueues actually use `JETONOMY_VERSION` via positional argument. Re-verify with a stricter check before treating as real. Likely false alarm; if real, easy fix.

### C6 🟡 `wp_register_script('jetonomy-data', '', ..., false)` (in_footer=false)
`includes/class-template-loader.php:345` - the dummy data-only handle is registered in the head. Doesn't matter functionally (data is always rendered before the body's first script use), but PCP would prefer footer. **Cosmetic.**

## D. Code hygiene

### D1 🟠 `vendor/` tracked in repo (52 MB free + 47 MB Pro)
Both repos have `vendor/` listed in `.gitignore`, AND yet `git ls-files vendor/` returns tracked files. Indicates `git rm --cached vendor/` was never run after adding the gitignore rule. **52+47 = ~100 MB of tracked composer dependencies** on a public GitHub repo. Inflates clone time, makes the repo embarrassing on `cloc` and `gh repo view`. **Fix:** `git rm -r --cached vendor/` + commit. Composer install on dev machines unaffected.

### D2 ⚪ One TODO marker in free, none in Pro
Single `TODO` on free side. Acceptable - not blocking.

### D3 ⚪ All shipping PHP files have `defined( 'ABSPATH' ) || exit;` guard
The flagged offenders are all in `node_modules/` (excluded from packaging by `.distignore` / build-release.sh) or `phpstan-bootstrap.php` (dev-only, never shipped). **No issue in production.**

### D4 ⚪ License header on root plugin file
GPL v2+ - confirmed pass-1.

### D5 🟠 Mixed `--` em-dashes vs `-` em-dashes outside i18n strings
Inside i18n functions: confirmed clean (memory rule enforced). Inside other contexts (PHP doc-comments, error_log strings, JS comments, code comments): hundreds of `-` em-dashes throughout the codebase. **Not a public-review red flag** (em-dashes are valid English punctuation in comments) but inconsistent with the user's documented preference. Out of scope for an audit; flagging only because the memory rule "no em-dash anywhere" exists.

### D6 ⚪ Inline JS / CSS in PHP

### D7 ⚪ Text-domain consistency
Already verified - 2,288 i18n calls, all use `'jetonomy'` uniformly.

## F. Data integrity

### F1 🟠 No transactional wrappers on multi-table writes
Operations like "create space → seed creator as admin → create default category" execute as separate INSERTs without a `START TRANSACTION` / `COMMIT` envelope. If the second INSERT fails, the first is orphaned. WP doesn't expose a clean transaction API for `dbDelta`-managed tables but `$wpdb->query('START TRANSACTION')` works on InnoDB.
- Real impact: low (the failures are infrequent), but real for a "ready all time" stance.

### F2 🟠 No FK constraints in schema
Standard WP plugin posture (core has none either) - referential integrity is enforced in PHP by the model layer, not at the DB level. If a moderator manually deletes a row via phpMyAdmin, orphans appear. Not a code bug, but a long-term-stability data-integrity concern. **Mitigation:** the existing nightly `Recount::run()` cleans denormalized counters; consider extending to detect orphan IDs.

### F3 ⚪ Tables use `utf8mb4` consistently
Verified via `$wpdb->get_charset_collate()` everywhere. No `utf8` (3-byte) anywhere in shipping schema. Emoji-safe.

### F4 ⚪ All tables InnoDB
Last commit ensuring InnoDB on all tables: 2026-04-13 `e4b93ce`. Verified.

## G. Pro-specific architecture

### G1 🟠 Each extension's `Extension::activate()` is the schema entry point but isn't documented as such
The abstract `Extension` base has no `install_schema()` method. Each extension's `activate()` directly calls its own `create_tables()` / `create_table()`. Naming inconsistency too (singular vs plural). **Fix:** abstract base method `install_schema()`, every extension overrides; `activate()` becomes "call `install_schema()` then any extension-specific seed".

### G2 🟠 No `Pro_Schema` aggregator
Pro has no central place that lists every Pro table. Each extension knows its own. For uninstall, for monitoring, for backup tooling - there's no canonical answer to "what Pro tables exist". **Fix:** `Pro_Schema::get_table_names()` aggregating from each extension's `get_table_names()` method.

### G3 ⚪ License gating works as designed
`Extension::is_enabled()` + `License::can_use_extension()` checked before `boot()`. Verified earlier in this session.

### G4 🟡 13 dead-listener contract bugs already documented in `audit/manifest.json`
Per the existing `audit/FEATURE_AUDIT.md` (free side, generated 2026-04-30): 3 white-label branding filters never fire, 2 custom-fields response filters never fire, 8 webhook lifecycle hooks have name drift or no firer. Already a known issue. Tracked as 1.5.0 work in `plan/punch-list-2026-04-30.md`.
