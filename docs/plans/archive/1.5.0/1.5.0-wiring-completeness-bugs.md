## Orphan bugs (a human cannot use the feature)

### [HIGH] POST /spaces/(id)/invite (generate_invite) + {prefix}jt_invite_links write path (rest)
- **Missing:** Both admin UI and frontend UX for invite-link generation. The whole 'invite-only' join policy is unusable from any human surface because no one can mint a link.
- **Fix:** Add an invite-link generation surface. Minimum: an 'Invite' panel in includes/admin/views/space-edit.php (a generate button + max_uses/expires_at fields + list of active links calling POST /spaces/{id}/invite via REST per the frontend-REST rule), and ideally a space-admin button on the frontend space header / space-members.js. Without it the 'Invite only' join policy and jt_invite_links table are dead weight shipped to customers.
- **Evidence:** Registered class-spaces-controller.php:137-156; handler at :875 calls InviteLink::generate() (:896), the only non-test/non-CLI writer of jt_invite_links (model includes/models/class-invite-link.php:29; table manifest line 1547). NO UI caller: grep for generate_invite / POST '/spaces/.../invite' / InviteLink::generate across includes/admin, templates/, and all non-min frontend JS (space-edit.js, space-members.js, new-space.js, composer.js) returns ZERO. Only other refs are class-demo-seeder.php (seeds demo links) and class-schema.php (table def) - not UI. The 'invite' join_policy option exists in admin (space-edit.php:115) and frontend (space-edit.php:166), and invite CONSUMPTION has a landing page (templates/views/invite.php) + GET /invite/{token}, but a space admin has NO button anywhere to actually GENERATE/copy an invite link.

### [HIGH] {prefix}jt_invite_links table (table)
- **Missing:** Both human write surfaces (admin + frontend) for generating invite links. Same root gap as POST /spaces/{id}/invite.
- **Fix:** Ship an invite-link generation UI (admin Invite tab on space-edit, and/or frontend space-admin button) calling POST /spaces/{id}/invite. Until then the table + invite-only join policy are half-wired - the consumption path works but nothing populates the table through a human surface.
- **Evidence:** Manifest line 1547. Sole production writer is InviteLink::generate() (class-invite-link.php:29), reachable only from POST /spaces/{id}/invite (class-spaces-controller.php:875,896) and CLI. No admin or frontend UI invokes generation (grep across includes/admin, templates, non-min JS = zero). Consumption side (InviteLink::accept) IS wired via invite.php landing + GET /invite/{token}, but rows can only be created by API/CLI/demo-seeder - a human site admin cannot mint an invite link from any screen.

### [MED] GET /users/(?P<id>\d+)/posts (rest)
- **Missing:** BOTH: no frontend caller AND no admin caller. The user-profile page that would show a user's posts renders them via direct SQL, not this route.
- **Fix:** ORPHAN. Either (a) wire user-profile.php's posts tab to hydrate/paginate via GET /users/{id}/posts (replacing the inline SQL, giving one source of truth + matching the REST-first acceptance rule), or (b) if the route exists only for the upcoming app, document it as intentional-api-only in audit/manifest.json. Right now it is half-cooked: a REST read no human surface uses, while the equivalent UI bypasses it with raw SQL.
- **Evidence:** Reg: includes/api/class-users-controller.php:74-86 (get_user_posts; handler at :392-394). Caller search: grep 'users/.*/posts | getUserPosts | user_posts' across assets/js (non-min), jetonomy-pro, templates, includes returns ZERO consumers (only the controller registration itself + WPCS/stub vendor noise). The profile posts list is server-rendered: templates/views/user-profile.php:75 'WHERE p.author_id = %d AND p.status = publish' raw query, NOT this REST route. Pagination JS (pagination-hydrator.js, pagination-frontend.js) does not reference it either.

### [MED] GET /jetonomy/v1/tags (Tags_Controller::list_tags) (rest)
- **Missing:** Both human surfaces. No frontend JS consumer (tag autocomplete in composer, tag-cloud widget, or async tag list would be the natural consumer) and no admin UI consumer. Pure API-only read with no caller.
- **Fix:** Wire GET /tags to a real consumer or document the exception. Best fit: tag autocomplete in the composer (assets/js/composer.js currently has none) and/or an async tag-cloud block. If it is intentionally an external/MCP read endpoint, mark it intentional-api-only in the manifest. As-is it is a half-wired feature: a queryable endpoint no human surface touches.
- **Evidence:** Registration: includes/api/class-tags-controller.php:29 register_rest_route('/tags', list_tags @ :60). NO consumer: grep for `v1/tags`, `restFetch('/tags')`, `apiBase}/tags`, `restTags` across assets/js (non-min) + templates + includes returns ZERO hits. Frontend tag surfaces resolve the data SERVER-SIDE instead: templates/views/tag.php:11 uses Tag::find_by_slug(); sidebar.php tag cloud, search.php:360, single-post.php:409, post-card.php all link to /tag/:slug/ pages rendered server-side. No block/shortcode reads it (class-blocks.php has no tag-cloud block; class-shortcodes.php has no tags shortcode that hits REST).
