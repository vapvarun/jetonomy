# Audit fixes — getting-started (12 findings)

## 1. [CRITICAL] `getting-started/04-in-page-auth.md` — inaccurate
- **Issue:** Doc claims three always-available full-page routes /community/login/, /community/register/, /community/lost-password/ using /auth/* endpoints.
- **Fix:** Remove the 'Dedicated Pages' table (lines 36-46) or rewrite it to describe the Login block (wp:jetonomy/login) and wp-login.php as the actual sign-in surfaces. There are no /community/login/, /community/register/, /community/lost-password/ frontend pages.
- **Evidence:** Router includes/class-router.php add_rewrite_rules() (lines 54-136) registers NO login/register/lost-password rewrite rules; auth-adjacent frontend route is only new-space (line 117). /auth/login, /auth/register, /auth/lost-password exist ONLY as REST routes in includes/api/class-auth-controller.php (rest_base='auth' line 23; routes registered lines 28, 60, 110). Template loader uses wp_login_url() for loginUrl (line 245) and redirects (lines 22,35,42). Doc says routes at 04-in-page-auth.md:38-46.

## 2. [CRITICAL] `getting-started/04-in-page-auth.md` — inaccurate
- **Issue:** Doc says Private mode confines guests to /community/login/ etc. and the setting lives under Jetonomy -> Settings -> Privacy.
- **Fix:** Change 'Jetonomy -> Settings -> Privacy' to 'Jetonomy -> Settings -> General -> Access Control'. State that private-mode guests are redirected to the WordPress login screen (wp_login_url) with a redirect_to back to the community, not to a /community/login/ page.
- **Evidence:** Private mode (guest_read off) redirects anonymous visitors to wp_login_url(home_url(REQUEST_URI)) — includes/class-template-loader.php:35,42 — i.e. the WordPress login page, not /community/login/. There is no Privacy tab: settings tabs are general/permissions/email/appearance/seo/antispam (includes/admin/views/settings.php:47-54). The Public/Private radio (guest_read) is in 'Access Control' under the general tab (settings.php:278,299). Doc claims at 04-in-page-auth.md:92-101.
