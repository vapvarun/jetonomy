# Owner First-Run & Configuration Journey Audit

> **STATUS as of 1.4.4-dev.** Current spec: `journey-map-1.4.4.md`. All moments here are `[free]`.

**Branch**: 1.4.4-dev
**Date**: 2026-05-23
**Method**: Code-grounded, three-lens per journey moment (Expectation / Current reality / Gap + Fix direction), with browser verification of the 1.4.4 fixes from `1.4.4-verification.md`.

---

## What got easier on 1.4.4-dev

After activating Jetonomy 1.4.4, you can:

- Open the License page right after install and click Manage License without seeing a broken-script error. The licensing UI is now bundled with the plugin; no other plugin or composer step is required.
- Walk away from a brand new community and trust that the background jobs still run. Trust level promotions, expired ban cleanup, scheduled-post publishing, activity pruning, notification cleanup, and verification reminders now run through a reliable background queue, not WP-Cron, so they fire on time even on quiet sites with no traffic.
- Browse any of your spaces from the front end and click "Edit this space" in the WordPress admin bar. Owners and space admins no longer have to navigate to wp-admin and find the right row to tweak a space's settings.
- Trust that the admin "Unban user" confirmation looks like the destructive action it is: a ghost Cancel and a red Confirm, matching the rest of the design system.
- Build a release zip from a clean clone and have it work on first activation. The build script now refuses to produce a zip that is missing the licensing SDK or the background-job library.

---

## Top 3 Open Gaps

1. **Wizard JS still hardcodes `'community/'` as the initial URL preview.** If the site was previously configured with a different `base_slug` (for example `discuss`), step 1's progress preview shows the wrong URL on first render and only corrects once the owner types. Severity: major. This is the most visible moment of the entire first-run flow.

2. **Trust level thresholds in Settings still have no "what does this level unlock?" column.** The owner sees numeric inputs for posts, days, reputation, replies, but nothing that explains that, for example, Trust Level 2 is the gate for creating spaces. Severity: major. Owners misconfigure trust levels because the contract is hidden.

3. **No "is my community ready to launch?" checklist anywhere in the admin.** After the wizard completes there is no surface that tells the owner whether email is configured, whether a CAPTCHA is enabled, whether demo data has been removed, or whether at least one non-admin member can read and post. Severity: major. Owners go live with broken email, sample content still visible, and no spam protection because nothing prompts them to check.

## Closed in 1.4.4

- **Licensing UI works on first activation.** EDD SL SDK is committed under `libs/edd-sl-sdk/` and ships in every release zip. The Manage License button no longer fails with a missing-script error after a fresh install.
- **Background jobs run on quiet sites.** Action Scheduler 3.9.3 is bundled under `libs/action-scheduler/`. Trust evaluation, expired-ban cleanup, activity pruning, notification cleanup, scheduled-post publishing, and verification reminders all run through the AS queue. They no longer skip on first-run sites with no traffic.
- **Front-end "Edit this space" admin-bar entry.** Any user who passes `Permission_Engine::is_space_admin` sees a one-click Edit link in the admin bar from any page inside a space they administer. Owners no longer detour through wp-admin to tweak a space they are browsing.
- **Destructive confirms match the design system.** The wp-admin Banned Users unban confirm uses the ghost-cancel + danger-confirm pattern. No code change was needed; the older audit screenshot predated the token swap.
- **Release zip integrity is enforced at build time.** `bin/build-release.sh` now asserts the EDD SL SDK loader, the SDK JS bundle, the SDK CSS, and Action Scheduler are all present in staging before producing the zip. A clean clone can no longer produce a broken zip.
