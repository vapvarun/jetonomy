## Conventions extracted from existing controllers

All findings below were drawn from reading `class-base-controller.php`, `class-spaces-controller.php`, `class-posts-controller.php`, `class-api.php`, and `tests/integration/api/CategoriesApiTest.php`.

| Pattern | Convention |
|---------|-----------|
| Namespace | `Jetonomy\API` |
| File naming | `includes/api/class-{slug}-controller.php` where slug is kebab-case of the `rest_base` or logical name |
| Class registration | Add key/class pair to the `$classes` array in `Api::register_routes()` at `includes/api/class-api.php:23` |
| `permission_callback` for authenticated routes | Method on the controller, typed `bool\|WP_Error`. Project uses both `login_permission_check()` (defined in Base_Controller, returns `WP_Error` with 401) and inline `require_login_check()` (identical, defined locally on Spaces_Controller). The base class method is preferred for new code. |
| `permission_callback` for public routes | String literal `'__return_true'` |
| Response shape - single item | Bare object/array (no envelope wrapper). Example: `new WP_REST_Response( $data, 201 )`. |
| Response shape - collection | `$this->paginated_response( $items, $meta )` which wraps into `{ data: [], meta: { count, has_more, cursor_next } }`. |
| Error codes | `jetonomy_forbidden` (403), `jetonomy_not_found` (404), `jetonomy_validation` (400), `jetonomy_unauthorized` (401). All via `$this->permission_error()`, `$this->not_found()`, `$this->validation_error()`. |
| Args schema style | Inline array with `type`, `required`, `sanitize_callback`, `validate_callback` keys. No JSON Schema `$ref`. |
| Integration test pattern | `WP_UnitTestCase`, real `WP_REST_Server`, `Schema::create_tables()` in `set_up()`, `do_request()` helper that calls `wp_set_current_user()` then `$this->server->dispatch()`. Lives in `tests/integration/api/`. |
| No `tests/unit/api/` directory exists | Unit-level API tests go in `tests/integration/api/` - that is the project convention for controller tests even though they are labeled "unit" in some docs. |

### REST contract

```
Route:   POST /jetonomy/v1/media
Auth:    Cookie + X-WP-Nonce: wp_rest  (wp_verify_nonce enforced by WP core REST middleware)
Content-Type: multipart/form-data
```

**Args schema** (registered in `register_routes()` `args` key):
```php
'args' => [
    'file' => [
        // Note: 'file' is NOT in the args schema for REST - WP REST does not
        // process multipart files through the args schema. The controller
        // reads it directly from $_FILES['file']. Document the expectation
        // in the docblock only; no args entry needed.
    ],
]
```

Because `$_FILES` is outside the REST args schema pipeline, the controller must do its own validation (file present, type allowed).

**Permission callback** (`upload_permission_check`):
```php
public function upload_permission_check(): bool|WP_Error {
    if ( ! is_user_logged_in() ) {
        return new WP_Error( 'jetonomy_unauthorized', __( 'You must be logged in.', 'jetonomy' ), [ 'status' => 401 ] );
    }
    $can_upload = current_user_can( 'upload_files' )
        || current_user_can( 'jetonomy_upload_media' )
        || current_user_can( 'jetonomy_create_posts' )
        || current_user_can( 'jetonomy_create_replies' );
    if ( ! $can_upload ) {
        return $this->permission_error(); // 403
    }
    return true;
}
```
This exactly mirrors the cap logic in the current `Media::handle_upload()` at `includes/class-media.php:31-34`.

**Handler body** (`upload_item`):
```php
// 1. Check $_FILES['file'] is present.
// 2. require_once WP admin includes: image.php, file.php, media.php.
// 3. $attachment_id = media_handle_upload( 'file', 0 );
// 4. If WP_Error, return $this->validation_error( $e->get_error_message() ) with status 422.
// 5. Collect metadata:
//    $url    = wp_get_attachment_url( $attachment_id );
//    $alt    = get_post_meta( $attachment_id, '_wp_attachment_image_alt', true ) ?: '';
//    $meta   = wp_get_attachment_metadata( $attachment_id );
//    $mime   = get_post_mime_type( $attachment_id );
//    $width  = $meta['width']  ?? 0;
//    $height = $meta['height'] ?? 0;
// 6. Return new WP_REST_Response( compact( 'id', 'url', 'alt', 'mime', 'width', 'height' ), 201 );
```

**Response shape (HTTP 201):**
```json
{
  "id": 42,
  "url": "https://example.com/wp-content/uploads/2026/04/photo.jpg",
  "alt": "",
  "mime": "image/jpeg",
  "width": 1200,
  "height": 800
}
```

**Error responses:**
- `401` - not logged in
- `403` - logged in but lacks upload cap
- `400` - no file in request (`$_FILES['file']` missing or `$_FILES['file']['error'] !== UPLOAD_ERR_OK`)
- `422` - `media_handle_upload()` returned `WP_Error` (file type rejected, size limit, disk full)

**No explicit nonce parameter** - WP REST middleware validates `X-WP-Nonce` automatically via `check_ajax_referer` equivalent in the REST authentication layer. The old `check_ajax_referer( 'jetonomy_upload', 'nonce' )` is gone.

### JS migration

**`assets/js/composer.js`, function `uploadImage` (lines 201-247)**

Current code reads `jetonomyUpload.nonce` (an `admin-ajax`-style nonce) and `jetonomyUpload.ajaxUrl`, then appends `action=jetonomy_upload_image` and `nonce=<nonce>` to the `FormData`.

**New code:**
```js
function uploadImage( file, editor ) {
    var placeholder = document.createElement( 'div' );
    placeholder.className = 'jt-upload-placeholder';
    placeholder.textContent = 'Uploading…';
    editor.appendChild( placeholder );

    // After the localization change, jetonomyUpload.uploadUrl is the REST
    // endpoint and jetonomyUpload.nonce is a wp_rest nonce.
    var uploadUrl = ( typeof jetonomyUpload !== 'undefined' && jetonomyUpload.uploadUrl )
        ? jetonomyUpload.uploadUrl
        : '/wp-json/jetonomy/v1/media';
    var nonce = ( typeof jetonomyUpload !== 'undefined' && jetonomyUpload.nonce )
        ? jetonomyUpload.nonce
        : '';

    var formData = new FormData();
    formData.append( 'file', file );
    // No 'action' field - REST does not use wp_ajax action routing.
    // No 'nonce' body field - REST nonce goes in the header.

    fetch( uploadUrl, {
        method: 'POST',
        headers: { 'X-WP-Nonce': nonce },
        body: formData,
        credentials: 'same-origin',
    } )
    .then( function( r ) { return r.json(); } )
    .then( function( res ) {
        placeholder.remove();
        // REST returns bare object on success (201), WP_Error shape on failure.
        if ( res.id && res.url ) {
            var img = document.createElement( 'img' );
            img.src = res.url;
            img.alt = res.alt || file.name;
            img.style.maxWidth = '100%';
            img.style.height = 'auto';
            img.style.borderRadius = '8px';
            img.style.margin = '8px 0';
            editor.appendChild( img );
            editor.appendChild( document.createElement( 'br' ) );
        } else {
            var msg = ( res.message ) || ( res.data ) || 'Upload failed';
            ( window.bnToast ? window.bnToast( msg, 'error' ) : null );
        }
    } )
    .catch( function() {
        placeholder.remove();
        ( window.bnToast ? window.bnToast( 'Upload failed', 'error' ) : null );
    } );
}
```

**Key changes from old code:**
- `fetch( uploadUrl, ... )` not `fetch( ajaxUrl, ... )`
- `headers: { 'X-WP-Nonce': nonce }` added; no `Content-Type` header (browser sets it with boundary for multipart automatically)
- `formData.append( 'action', ... )` removed
- `formData.append( 'nonce', nonce )` removed
- Response parsing changes: old code checked `res.success && res.data.url` (admin-ajax envelope); new code checks `res.id && res.url` (bare REST response)

**Script localization change in `includes/class-template-loader.php` (lines 351-360):**
```php
// Before:
'ajaxUrl' => admin_url( 'admin-ajax.php' ),
'nonce'   => wp_create_nonce( 'jetonomy_upload' ),

// After:
'uploadUrl' => rest_url( 'jetonomy/v1/media' ),
'nonce'     => wp_create_nonce( 'wp_rest' ),
```

### REST contract

```
Route:   POST /jetonomy/v1/auth/login
Auth:    No cookie required (public endpoint)
Content-Type: application/x-www-form-urlencoded  OR  application/json
```

**Args schema:**
```php
'args' => [
    'user_login' => [
        'type'              => 'string',
        'required'          => true,
        'sanitize_callback' => 'sanitize_text_field',
        'validate_callback' => function( $v ) { return '' !== trim( $v ); },
    ],
    'user_password' => [
        'type'     => 'string',
        'required' => true,
        // No sanitize_callback - passwords must not be sanitized (would strip special chars).
        // validate_callback checks non-empty only.
        'validate_callback' => function( $v ) { return '' !== $v; },
    ],
    'remember' => [
        'type'    => 'boolean',
        'default' => false,
    ],
],
```

**Permission callback:** `'__return_true'` (public unauthenticated endpoint).

**Handler body** (`login`):
```php
// 1. check_rate_limit( 'login' ) - return 429 with message if exceeded.
// 2. $login    = $request->get_param( 'user_login' );
//    $password = $request->get_param( 'user_password' );
//    $remember = (bool) $request->get_param( 'remember' );
// 3. $user = wp_signon( [ 'user_login' => $login, 'user_password' => $password, 'remember' => $remember ], is_ssl() );
// 4. If WP_Error: return new WP_Error( 'jetonomy_auth_failed', __( 'Incorrect username or password.', 'jetonomy' ), [ 'status' => 401 ] );
//    Generic message intentional - prevents account enumeration (same as AJAX handler).
// 5. // wp_signon() already set the auth cookie internally.
//    $redirect = $request->get_param( 'redirect_to' );
//    $redirect = $redirect ? esc_url_raw( $redirect ) : home_url( '/' );
// 6. Return new WP_REST_Response( [ 'user_id' => $user->ID, 'redirect' => $redirect ], 200 );
```

**Note on cookie behavior:** `wp_signon()` calls `wp_set_auth_cookie()` internally when `$user` is returned. No separate `wp_set_auth_cookie()` call is needed unless you want to set it a second time. The plan says "sets auth cookie via `wp_set_auth_cookie()`" - this is satisfied by `wp_signon()` automatically.

**Response shape (HTTP 200):**
```json
{
  "user_id": 7,
  "redirect": "https://example.com/"
}
```

**Error responses:**
- `429` - rate limit exceeded. Body: `{ code: "jetonomy_rate_limited", message: "Too many attempts..." }`
- `400` - missing/empty required field (handled by REST schema validation before the handler runs)
- `401` - wrong credentials. Body: `{ code: "jetonomy_auth_failed", message: "Incorrect username or password." }`

**CSRF note:** WP REST authentication for cookie-based requests does require a nonce (`X-WP-Nonce` or `_wpnonce` query param). However, this is a `__return_true` permission endpoint so WP REST does NOT enforce cookie authentication - it only enforces nonce when using `Cookie` authentication. For a truly unauthenticated public endpoint, no nonce is checked by WP core. The plan mentions "nonce body field for CSRF parity" - implement this as an **optional** `nonce` param: if present and non-empty, verify it with `wp_verify_nonce( $nonce, 'jetonomy_quick_login' )` and return 403 on failure. This protects sites that pass the nonce from login-block.js during the coexistence window (commits 1 and 2 in flight), then the nonce param can be dropped in commit 3 cleanup.

**`assets/js/login-block.js`, function `submitForm` (lines 33-77)**

Current code sends `FormData` to `block.dataset.ajaxUrl` with `action=jetonomy_quick_login` and `nonce=<login_nonce>` fields in the body.

**New code:**
```js
function submitForm( block, form, endpoint ) {
    var submitBtn = form.querySelector( '.jt-login-submit' );
    var originalLabel = submitBtn ? submitBtn.textContent : '';
    if ( submitBtn ) {
        submitBtn.disabled = true;
        submitBtn.textContent = '...';
    }
    setMessage( form, '', false );

    // Collect form fields into a plain object for JSON.
    var fields = {};
    ( new FormData( form ) ).forEach( function( value, key ) { fields[ key ] = value; } );

    var nonce = block.dataset.restNonce || '';

    fetch( endpoint, {
        method: 'POST',
        credentials: 'same-origin',
        headers: {
            'Content-Type': 'application/json',
            'X-WP-Nonce': nonce,
        },
        body: JSON.stringify( fields ),
    } )
    .then( function( res ) {
        return res.json().then( function( json ) { return { ok: res.ok, status: res.status, json: json }; } );
    } )
    .then( function( payload ) {
        if ( ! payload.ok ) {
            var msg = ( payload.json && payload.json.message )
                || 'Something went wrong. Please try again.';
            setMessage( form, msg, false );
            if ( submitBtn ) { submitBtn.disabled = false; submitBtn.textContent = originalLabel; }
            return;
        }
        setMessage( form, 'Success', true );
        window.location.reload();
    } )
    .catch( function() {
        setMessage( form, 'Network error. Please try again.', false );
        if ( submitBtn ) { submitBtn.disabled = false; submitBtn.textContent = originalLabel; }
    } );
}
```

### Related considerations

- **Cookie-setting under REST:** `wp_signon()` calls `wp_set_auth_cookie()` internally. However, REST requests go through the normal WP bootstrap so cookies set during the REST request are sent in the response `Set-Cookie` header. This works correctly. No extra `wp_set_auth_cookie()` call is needed.
- **`redirect_to` parameter:** The current AJAX handler does not return a redirect URL - it just sends `"Signed in. Reloading..."` and the JS does `window.location.reload()`. The new REST response adds `redirect` so future callers (mobile app) can follow the URL. The JS in commit 2 still uses `reload()` for simplicity - switching to `location.href = payload.json.redirect` is a separate improvement.
- **No captcha on login:** The `Captcha_Manager` at `includes/captcha/class-captcha-manager.php` does not hook into the login flow. The existing AJAX handler has no captcha check. The REST endpoint should match: no captcha on login, only on register (see A.3).

### REST contract

```
Route:   POST /jetonomy/v1/auth/register
Auth:    No cookie required (public endpoint)
Content-Type: application/x-www-form-urlencoded  OR  application/json
```

**Args schema:**
```php
'args' => [
    'username' => [
        'type'              => 'string',
        'required'          => true,
        'sanitize_callback' => function( $v ) { return sanitize_user( wp_unslash( $v ), true ); },
        'validate_callback' => function( $v ) { return strlen( trim( $v ) ) >= 3; },
    ],
    'email' => [
        'type'              => 'string',
        'required'          => true,
        'sanitize_callback' => 'sanitize_email',
        'validate_callback' => 'is_email',
    ],
    'password' => [
        'type'     => 'string',
        'required' => true,
        // No sanitize_callback on passwords.
        'validate_callback' => function( $v ) { return strlen( $v ) >= 8; },
    ],
    'captcha_token' => [
        'type'     => 'string',
        'required' => false,
        'default'  => '',
    ],
],
```

**Permission callback:** `'__return_true'` (public unauthenticated endpoint).

**Handler body** (`register`):
```php
// 1. Check users_can_register option. If false, return 403 with code
//    'jetonomy_registration_disabled', message __( 'Registration is disabled on this site.', 'jetonomy' ).
// 2. check_rate_limit( 'register' ) - return 429 if exceeded.
// 3. Validate captcha if Captcha_Manager::is_enabled():
//    $token  = $request->get_param( 'captcha_token' ) ?: '';
//    $ip     = sanitize_text_field( wp_unslash( $_SERVER['REMOTE_ADDR'] ?? '' ) );
//    $result = Captcha_Manager::verify_or_skip( 0, $token, $ip );
//    If false (failed), return 400 with code 'jetonomy_captcha_failed',
//    message __( 'Security check failed. Please try again.', 'jetonomy' ).
// 4. $username = $request->get_param( 'username' );
//    $email    = $request->get_param( 'email' );
//    $password = $request->get_param( 'password' );
// 5. Duplicate checks:
//    if ( ! validate_username( $username ) || username_exists( $username ) )
//        return $this->validation_error( __( 'That username is unavailable.', 'jetonomy' ) );
//    if ( email_exists( $email ) )
//        return $this->validation_error( __( 'That email is unavailable.', 'jetonomy' ) );
// 6. $user_id = wp_create_user( $username, $password, $email );
//    If WP_Error, return $this->validation_error( $user_id->get_error_message() );
// 7. do_action( 'jetonomy_user_registered', (int) $user_id );
//    // Intentionally NOT calling wp_send_new_user_notifications() - same rationale
//    // as the AJAX handler: Jetonomy Notifier owns the welcome email.
// 8. wp_set_current_user( (int) $user_id );
//    wp_set_auth_cookie( (int) $user_id, false, is_ssl() );
// 9. $redirect = $request->get_param( 'redirect_to' );
//    $redirect = $redirect ? esc_url_raw( $redirect ) : home_url( '/' );
// 10. Return new WP_REST_Response( [ 'user_id' => $user_id, 'redirect' => $redirect ], 201 );
```

**Response shape (HTTP 201):**
```json
{
  "user_id": 12,
  "redirect": "https://example.com/"
}
```

**Error responses:**
- `403` - registration disabled sitewide
- `429` - rate limit exceeded
- `400` - captcha failed (code `jetonomy_captcha_failed`)
- `400` - username taken / invalid (code `jetonomy_validation`)
- `400` - email taken / invalid (code `jetonomy_validation`)
- `400` - `wp_create_user()` returned `WP_Error`

**CSRF note:** Same as A.2 login - `__return_true` permission callback means WP does not enforce a cookie-auth nonce. The `captcha_token` field provides the bot-protection layer for public registration. For browser clients, the login-block.js passes `data-rest-nonce` in the `X-WP-Nonce` header as an additional CSRF guard during the coexistence window.

### JS migration

The `submitForm` function rewrite in A.2 covers the register form as well. The register form `submit` listener at `login-block.js:94-100` changes from:
```js
// Before:
```

The form field names in the register form are `username`, `email`, `password` - these match the REST args schema exactly. No field renaming needed.

**If `Captcha_Manager::is_enabled()` is true**, the login block's register form needs a captcha token field populated before submit. The existing `Captcha_Manager::enqueue_scripts()` injects `window.jetonomyCaptcha` with `provider` and `siteKey`. The `submitForm` function must be extended to populate `captcha_token` before the `fetch()` call:
```js
// Before JSON.stringify( fields ):
if ( window.jetonomyCaptcha && window.jetonomyCaptcha.provider && typeof grecaptcha !== 'undefined' ) {
    // For reCAPTCHA v3, execute first, wait for token, then fetch.
    // This requires converting submitForm to async or promise-chaining.
    // Flag for A.3 implementer: this is a non-trivial async refactor.
}
```
**This is the single most complex part of A.3.** See the related considerations below.

### Related considerations

**CAPTCHA token injection (gotcha - non-trivial async work):**
The current AJAX handler at `includes/class-blocks.php:808-809` calls `check_ajax_referer()` first, then immediately reads `$_POST` fields. There is no captcha check in the existing `ajax_quick_register()` method. However, `Captcha_Manager` exists, is initialized at bootstrap, and enqueues scripts globally. **The captcha token is currently never verified on the register path** - this is a pre-existing gap, not a regression introduced by A.3.

The REST endpoint should close this gap by checking `Captcha_Manager::verify_or_skip()`. The JS must supply the token. For reCAPTCHA v3, obtaining a token is asynchronous (`grecaptcha.execute()` returns a Promise). The `submitForm` function in `login-block.js` is currently synchronous. The implementer must:
1. Detect `window.jetonomyCaptcha.provider` before submitting.
2. If reCAPTCHA v3: call `grecaptcha.execute( siteKey, { action: 'register' } )` and await the promise before calling `fetch()`. Refactor `submitForm` to return a Promise or convert the submit listener to `async`.
3. If Turnstile: read the rendered widget's response token from the hidden input that Turnstile injects.
4. If no captcha: `captcha_token` is omitted or empty; the REST handler skips verification.

The implementer should add a `data-jt-panel` check so captcha execution only fires for the register panel, not the login panel.

**User metadata set on registration:**
The `do_action( 'jetonomy_user_registered', $user_id )` hook is the only post-creation hook in the existing AJAX handler. Verify which listeners are registered on this hook:
```bash
grep -rn "jetonomy_user_registered" includes/ --include="*.php"
```
The Notifier likely hooks here for welcome emails. No other user-profile seeding or trust-level initialization was found in the AJAX handler. The REST handler must fire the same hook.

**`wp_send_new_user_notifications()` suppression:**
The existing AJAX handler deliberately does NOT call `wp_send_new_user_notifications()` - the code comment at `includes/class-blocks.php:844-847` explains why: Jetonomy's Notifier owns branded emails. The REST handler must preserve this suppression.

**`users_can_register` option check:**
This is a sitewide WP option, not a Jetonomy setting. The AJAX handler checks it at line 810-813. The REST endpoint must check it too. The check belongs at the top of the handler, before rate-limit and captcha, so unauthenticated callers get a clean 403 without burning a rate-limit slot.

**No "auto-login after register" hook or trust-level initialization** was found in the AJAX handler. The `wp_set_current_user()` + `wp_set_auth_cookie()` calls at lines 850-851 are the only post-create side effects (besides the hook). Trust level is initialized lazily on first login/activity via the Trust evaluator, not at registration time.

**Hidden caller check for `check_rate_limit`:** The static method is private to `Blocks`. No external caller exists. Confirmed safe to remove in A.3 commit 3.

## Shared patterns and cross-cutting notes

**Auth controller file placement:** Both A.2 and A.3 endpoints live in one file: `includes/api/class-auth-controller.php`. The `Api::register_routes()` file-discovery logic at `includes/api/class-api.php:46` derives the filename from the key: key `'auth'` maps to `class-auth-controller.php`. This is consistent.

**`check_rate_limit` helper duplication:** The `Blocks::check_rate_limit()` private static method is copied verbatim into `Auth_Controller::check_rate_limit()` as a protected instance method. The logic is four lines. The alternative (making `Blocks::check_rate_limit()` public/static and calling it from the controller) would create a dependency between an API controller and the Blocks class, which is the wrong layering. Copy is correct here.

**Response envelope - deviation from `data` wrapper:** Existing collection endpoints use `{ data: [], meta: {} }`. The new endpoints return bare objects (A.1 returns `{ id, url, ... }`, A.2/A.3 return `{ user_id, redirect }`). This matches the existing single-item endpoint pattern in `Spaces_Controller::get_item()` and `Posts_Controller::get_item()` which also return bare objects. No deviation from convention.

**`login_permission_check()` vs `require_login_check()` vs `'__return_true'`:** For A.2 and A.3, use `'__return_true'` directly - do not use `login_permission_check()` from Base_Controller for public endpoints. For A.1, use the named method `upload_permission_check()` defined on `Media_Controller` rather than inheriting `login_permission_check()`, because the cap check is more complex than just "is logged in."

**Test directory:** There is no `tests/unit/api/` directory. All controller tests go in `tests/integration/api/` per the existing `CategoriesApiTest.php` example. New files: `MediaControllerTest.php` and `AuthControllerTest.php`.

**`grunt build` reminder:** After modifying `composer.js` or `login-block.js`, run `grunt build` to regenerate the `.min.js` counterparts before commit. The `bin/build-release.sh` Step 5b assertion will fail the release build if source and min are out of sync.
